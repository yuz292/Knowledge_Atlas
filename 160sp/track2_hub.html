What Track 2 owns

Owned by Track 2

  • reference import and PDF intake
  • metadata enrichment and abstract recovery
  • taxonomy scoring, question relevance, and triage
  • duplicate detection, ranking, and review queues
  • building AE job bundles that conform to the handoff schema

Not owned by Track 2

  • claim extraction from PDFs
  • Mathpix rebuild logic
  • belief generation, BN edges, or downstream AE internals
  • any claim that a paper is "fully extracted" once AF has only staged or bundled it

Boundary to keep in mind

AF's contract with AE is the job bundle and its metadata, not the extraction result. If your bundle is invalid, duplicate-ridden, or poorly justified, the failure belongs to Track 2 even if AE later crashes somewhere else.

What success looks like in this track

You can show the smallest trustworthy AF path.

One topic or question can be discovered, triaged, reviewed, and bundled with visible evidence at each step.

You can distinguish accept, review, reject, and duplicate cases.

The system is not "good" if it simply pulls many papers. It must separate useful candidates from false positives and show why.

You can explain the handoff.

Students should know what goes into paper.json, why schema validity matters, and what AE should be able to trust from AF.

You can name the current failure modes.

A mature demo includes known weaknesses, not only happy-path screenshots.

How to use AI in Track 2

Good uses

  • ask for the smallest plausible implementation path
  • ask for likely failure modes and adversarial tests
  • ask for benchmark design, not just code generation
  • ask for comparisons between two triage strategies or two dedup thresholds
  • ask AI to restate a spec in executable terms before it writes code

Bad uses

  • letting AI invent repo commands that do not exist
  • accepting AI judgments about paper relevance without checking title, abstract, and metadata
  • shipping code without a test or benchmark because the output "looks right" once
  • using AI prose as a substitute for schema validation, count checks, or duplicate probes
  • hiding uncertainty instead of logging it as a near miss, topic expansion signal, or new-topic candidate

Baseline prompt pattern for this track

When you ask AI for help, give it the exact repo file or command, the success condition, the tests you want, and one or two false positives you are worried about. If the AI response does not return to those criteria, tighten the prompt and ask again.

Core verification habits

  • 1
    Name the success condition before you build.
    "Improve intake" is not a success condition. "The second run of the same query makes zero external API calls and returns no duplicate DOI rows" is.
  • 2
    Test observable behavior, not intentions.
    Look at counts, schema validity, duplicate probes, logs, and bundle contents.
  • 3
    Record false positives explicitly.
    If a paper looks right but is wrong, that is a gift. Keep the example and use it to harden the pipeline.
  • 4
    Keep the repair loop visible.
    When a test fails, state what failed, what changed, and what new evidence shows the repair worked.
