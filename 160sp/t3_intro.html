TRACK 3 — VR STUDIO

Track 3: VR Studio

Build a pipeline that turns natural-language instructions into modified 3D rooms in real time — collect VR-ready interior models, prove every environmental factor can be manipulated, and ship an AI front-end that converts “a cozy library with 4-metre ceilings and exposed brick on the north wall” into an instant scene change.

3 tasks 10 weeks 75 + 75 + 75 points 120+ fps target

What you’ll build

The Knowledge Atlas catalogues the empirical evidence that interior environments alter human cognition and affect — ceiling height changes thinking, daylight shifts mood, materials change the feel of a room. A neuroarchitecture researcher who wants to test these claims has to vary one feature while holding everything else constant, which means she needs 3D rooms she can modify parametrically. No existing product gives her that. Tools like Veras and Decory produce 2D images, not editable 3D models; Meshy and Luma generate new models, not parametric edits of existing ones.

You will close that gap. In Task 1 you assemble a library of at least 20 VR-ready 3D interior models, each annotated so a program knows which mesh is the ceiling, which is the north wall, which is a window. In Task 2 you load those models in a browser-based VR viewer and prove every factor can be modified in real time — sliders for ceiling height, light intensity, material roughness; per-wall material swatches; path-traced lighting that converges in two seconds. In Task 3 you wire up a language model that reads the user’s instruction, returns a constrained JSON spec, validates every field against your viability matrix, and applies the modifications atomically.

The result is the first tool that lets a researcher say “give me the Meyers-Levy high-ceiling condition on this office” and get a modified 3D stimulus in seconds instead of weeks.

The three tasks

TASK 1
Collect & Catalog VR-Ready 3D Interior Models
Source at least 20 openly-licensed glTF/GLB interior models (Sketchfab, BlenderKit, Poly Haven, plus at least three generated through World Labs Marble), evaluate each model against eight manipulation classes, segment the meshes so every wall is individually addressable, and produce a mesh_roles.json per model.
4 phases · 75 points
TASK 2
VR Conversion & Manual Factor Testing
Build a browser-based A-Frame viewer with real-time slider controls for all eight manipulation classes, including per-wall material selection and at least HDRI-based realistic lighting. Test every factor on at least ten models and produce a factor viability matrix supported by before/after screenshots.
3 phases · 75 points
TASK 3
AI Front-End for Parametric Scene Modification
Add a natural-language input that runs through an LLM with a constrained JSON schema, a validation gate that checks every modification against your factor viability matrix, and a set of research-derived presets (Meyers-Levy, prospect-refuge, biophilic restoration). Every modification must apply atomically and degrade gracefully when a model does not support it.
5 phases · 75 points

The eight manipulation classes

Every model and every viewer has to support the same eight classes of environmental manipulation. These are derived from the Tagging Contractor registry; together they cover 330 of the 424 tags. Your mesh annotations in Task 1 must allow every class; your viewer in Task 2 must demonstrate every class; your AI front-end in Task 3 must accept natural-language instructions targeting every class.

Class What researchers manipulate Example slider
Geometry Ceiling height, room volume, enclosure ratio, isovist area Ceiling height: 2.0 m – 6.0 m
Lighting Illuminance, colour temperature (CCT), daylight vs artificial CCT: 2700 K – 6500 K
Materials Wood vs concrete vs glass; surface roughness; natural-material ratio Wall roughness: 0.0 – 1.0
Color Hue (warm vs cool), saturation, palette diversity Wall colour swatches
Biophilic elements Plant count, water features, view content (greenery vs built-up) Window-view backdrop selector
Furniture layout Seating count, sociopetal vs sociofugal arrangement, clutter density Furniture visibility toggle
Acoustic properties Acoustic privacy, natural sound sources, ambient sound level Ambient audio loop selector
Visual complexity Fractal dimension, symmetry, ornament density Partition depth: 0.0 m – 2.5 m

Tools and accounts you’ll need

Track 3 has more setup than the other tracks because it spans three software ecosystems — the browser, Blender, and the LLM API. Get these in place before the first sprint.

Blender 4.x  Required
Free and open-source. You’ll use it to import glTF models, segment fused meshes into named architectural elements, and run headless Python scripts for batch processing. Install via brew install --cask blender (Mac) or the official installer.
BlenderMCP  Recommended
Model Context Protocol bridge that lets Claude or Cursor control Blender via natural language with viewport awareness. The most powerful AI-assisted segmentation option for Task 1. brew install uv first, then follow the README.
World Labs Marble  At least 3 worlds
Photo → 3D world generator. The free plan gives you four worlds, the Standard plan gives twelve. You must generate at least three of your twenty models with Marble to demonstrate the photo → 3D → parametric pipeline.
A-Frame  Browser-based
WebXR framework that wraps Three.js. Loads glTF natively, runs in any browser, no install. Your Task 2 viewer is a single HTML file that drops in A-Frame via a CDN tag.
three-gpu-pathtracer  Stretch goal
Optional path-traced lighting library. Converges in ~2 seconds on any modern laptop with WebGL 2. Worth the upgrade for the realism win on Task 2; not required.
Gemini API key  Required for Task 3
Free at aistudio.google.com. Use the gemini-2.0-flash model with responseMimeType: "application/json" for constrained-JSON parameter extraction. Other LLM APIs work equivalently if you prefer.

Repos you’ll need

Track 3 lives mostly in the Knowledge_Atlas repo, but the AI front-end pulls from two sibling repos for vocabulary and presets.

Knowledge_Atlas
The Atlas itself — this is where your model catalog, VR viewer, and AI front-end ship. All three Task deliverables go in this repo on your track3/your-name branch.
Tagging_Contractor
The 424-tag registry: 330 of those tags are continuous or ordinal and represent manipulable environmental features. Reference for which factors map to which manipulation classes.
Article_Eater
The 166 PNU templates that describe what neuroarchitecture researchers actually manipulate in the published literature — the source of evidence for your Task 3 presets.

The contracts you write in each phase are graded as carefully as the working code — vague contracts are flagged at the contract gate and keep your work out of the integrated pipeline. Read the “Contract objective” box at the start of each phase and write your contract before you ask any AI to build anything.

How submission works for Track 3

Track 3 uses a single git branch per student — track3/your-name — across all three tasks. Each completed task is submitted as its own Pull Request to the instructor. The full mechanics are on each task page; here’s the lifecycle in one paragraph:

  1. Once at the start of the track: fork Knowledge_Atlas, clone, set the upstream remote, and create track3/your-name off upstream/main.
  2. Continuously while working: commit small, push at the end of each session. Models, mesh annotations, and HTML files all live on this single branch.
  3. At the end of each task: open a Pull Request from your branch to dkirsh:main, title it Track 3 · Task N: ... — your name, paste your file manifest, tag @dkirsh, and do not merge.
  4. Across tasks: keep using the same branch; additional task PRs are opened against the previous task’s merge commit.

The detailed commands and the per-task PR title format are on each task page under “Submit Task N on GitHub.” Track 3 submissions also include screenshots and short screen recordings as deliverables; pointers go in your PR body and the actual files go in a track3/<your-name>/media/ directory on your branch.
