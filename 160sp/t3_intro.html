TRACK 3 · VR STUDIO

Track 3: VR Studio — Parametric Rooms for Environmental Cognition Research

Every room in this track is generated parametrically by Infinigen Indoors (Raistrick et al., 2024) — from the floorplan down to the doorknob — and exposed to students through a thin Python wrapper, never through Blender’s UI. The AI front-end you ship in Task 3 translates a natural-language instruction into a validated parameter dict, calls the wrapper, and returns a glTF the browser displays in seconds. The cohort’s flagship deliverable is a single shared parametric module: a curved-wall worship room grounded in Vartanian et al. (2015) on the cathedral effect.

3 tasks 10 weeks 75 + 75 + 75 points Flagship: curved-wall worship room

What you’ll build

Environmental psychology rests on causal claims: ceiling height changes thought, daylight changes mood, fractal patterning changes affect. Testing those claims demands stimuli a researcher can sweep one factor at a time while holding the rest of the room constant. Veras and Decory output 2D images, not 3D scenes. Meshy and Luma generate fresh models, not parametric edits of existing ones. BuildingNet, ProcTHOR, and HSSD ship beautiful artist-authored scenes, but the parameters are baked into the geometry. None of these tools lets a researcher request the Meyers-Levy high-ceiling condition on a specified office.

Infinigen Indoors closes that gap. Princeton’s Vision and Learning Lab released it in 2024 under a BSD licence (Raistrick et al., 2024; Khanna et al., 2024); every surface, fixture, and prop is procedurally generated from a parameter graph rather than retrieved from a library. Out of the box it covers seven room types: kitchen, living room, bedroom, bathroom, dining room, hallway, and office. You drive it from Python with a JSON parameter dict and never touch Blender’s UI. We adopt it as the parametric backbone for this track.

You ship three deliverables on top of that backbone. In Task 1 you install Infinigen, render the seven built-in rooms, and write a JSON-Schema parameter manifest for two assigned room types — five to seven parameters apiece, each citing the environmental-psychology literature that motivates it. Rooms outside the seven covered classes get a parameter extension proposal instead. In Task 2 you wrap Infinigen in a Python function that takes the manifest’s dict and exports glTF, then load that glTF in the browser with Google’s <model-viewer> web component; the deliverable includes a parameter-sweep gallery showing each parameter at three settings, plus a divergence note comparing your room to its HSSD artist-authored counterpart. In Task 3 you add the AI front-end, ship literature-grounded presets, and contribute one component to the cohort’s shared curved-wall worship room.

The cohort output is a research artefact, not a homework set: the first open-source environmentally-instrumented parametric room library, partially closing the gap that BuildingNet, ProcTHOR, and HSSD do not fill. We will publish it with all contributors named.

Realism via splat augmentation

Parametric models are perfectly editable but visually flatter than real-world scans. Sprint 4 added a complementary path: students capture one real exemplar of each room type with Polycam (or Luma AI) producing a 3D Gaussian splat (Kerbl et al., 2023), and the cohort pipeline extracts the splat’s lighting environment as an HDRI plus a small library of PBR material packs. Subsequent parametric renders inherit real-world lighting and real-world surface appearance while keeping every parameter editable. The split is clean: parametric for what students manipulate (ceiling height, daylight, material class, fixture density); splat- derived for what students capture (the look of real oak, the feel of late-afternoon clerestory light). See scripts/track3/splat_to_hdri.py and scripts/track3/splat_to_materials.py for the extraction wrappers, and Task 1 Phase 2.5 for the per-student capture protocol.

The flagship — curved-wall worship room

One room of the fifteen breaks Infinigen’s axis-aligned-box assumption: the worship room. Sacred interiors are defined by curvature, by ceilings that arch and rise, by fractal ornament that draws the eye upward. Reproducing those features inside a parametric framework designed for orthogonal residential geometry is genuine engineering work. The cohort takes it on together.

We pick this room because the published evidence on its environmental effect is the strongest in the fifteen. Vartanian et al. (2015) demonstrated the cathedral effect — high ceilings raise approach motivation and shift cognitive style toward abstraction. Joye (2007) tied biophilic awe to the same vertical and fractal cues that religious architecture exploits. Bermudez et al. (2017) reported neural correlates of contemplative experience inside sacred space. A clean parametric implementation of these features is therefore citable in its own right, not merely a course exercise.

The implementation decomposes into sub-components: Bezier wall paths, ceiling-height curve, fractal patterning generator, light-source positioning, congregation-seating arrangement. Each student in Track 3 owns one sub-component during Task 3 and contributes it to the single shared module Knowledge_Atlas/3d_rooms/worship_curved/. At term end the cohort merges its contributions into one published parametric artefact with all contributors as co-authors.

Tell the instructor in Week 4 which sub-component you want to own.

Tools and accounts you’ll need

Track 3 spans three layers: the Python that drives Infinigen, the browser that renders the glTF, and the LLM that translates language to parameters. Get these in place before Sprint 1.

Blender 4.x  Required
Free and open-source. Installed but never opened by hand: Infinigen drives Blender headlessly through Python. Install via brew install --cask blender (Mac) or the official installer.
Infinigen Indoors (Python 3.10+)  Required
The parametric backbone for the whole track (Raistrick et al., 2024; Khanna et al., 2024; BSD-licensed). Course ships an infinigen_wrapper.py that converts a JSON parameter dict into an Infinigen scene call and exports glTF; the upstream repo lives at infinigen.org.
<model-viewer> web component  Browser-based
Required for Task 2. Free, no install, CDN-hosted by Google. One <model-viewer> tag in an HTML file loads the glTF, supports orbit and AR, and renders identically across browsers.
LLM API key (Gemini, Claude, or OpenAI)  Required for Task 3
Any of the three works. The wrapper requests JSON-mode output via responseMimeType: "application/json" (Gemini) or the equivalent constrained-decoding option on Claude or OpenAI. Free Gemini keys at aistudio.google.com.
Honeybee Rooms (Ladybug Tools)  Optional
AGPL-licensed substrate for coupling a parametric room to daylight, energy, or acoustic instrumentation. Optional for any student who wants to extend their room with physically-grounded environmental measurements.

How submission works for Track 3

Track 3 uses a single git branch per student — track3/your-name — across all three tasks. Each completed task is submitted as its own Pull Request to the instructor. The full mechanics are on each task page; here’s the lifecycle in one paragraph:

  1. Once at the start of the track: fork Knowledge_Atlas, clone, set the upstream remote, and create track3/your-name off upstream/main.
  2. Continuously while working: commit small, push at the end of each session. Parameter manifests, Python wrappers, viewer HTML, and preset JSON all live on this single branch.
  3. At the end of each task: open a Pull Request from your branch to dkirsh:main, title it Track 3 · Task N: ... — your name, paste your file manifest, tag @dkirsh, and do not merge.
  4. Across tasks: keep using the same branch; additional task PRs are opened against the previous task’s merge commit.

Track 3 submissions also include screenshots and short screen recordings of the parameter-sweep gallery and the AI front-end demos; pointers go in your PR body and the actual files go in a track3/<your-name>/media/ directory on your branch. The cohort flagship worship-room module is collaboratively merged at term end via a single shared PR that lists every contributing student as co-author.
