Region 1
Environment setup

VR work has a non-trivial stack. Getting it right in week 1 saves weeks of downstream confusion. The instructor-recommended baseline is Unity 2022 LTS with OpenXR; Unreal 5 is acceptable but unsupported.

Headset

  • Primary target: Meta Quest 3 (class has 4 units on loan via CSE loan cage)
  • Development: Quest Link over USB-C or Air Link over 5 GHz Wi-Fi
  • Desktop fallback for headset-less prototyping: Mock HMD runtime

Software

  • Unity 2022.3 LTS + Meta XR SDK + OpenXR
  • Git LFS (scenes and assets are binary; never commit *.unity without LFS)
  • Python 3.11 for post-hoc log analysis and AI-agent scripting
  • Optional: OpenAI / Anthropic API keys for LLM-driven NPCs

First-run checklist

  1. Install Unity Hub, add Unity 2022.3 LTS with Android Build Support.
  2. Clone ucsd-cogs160/vr_scenes, ensure Git LFS pulled all assets.
  3. Open the BaselineRoom scene and press Play in Editor.
  4. Build to Quest; verify tracking, hand controllers, and frame rate ≥ 72 fps.
  5. Run scripts/verify_vr_scene.py BaselineRoom.

Region 3
Ways of working

Ten students is a lot for VR; expect pair or triple sub-teams per scene rather than solo builds.

  • Sub-teams of 2–3; each owns one scene or one instrumentation module.
  • Weekly Monday sub-team stand-up + Friday all-hands scene demo.
  • No scene merges to main without another student running it on their own headset (not just the Editor).
  • Binary assets: commit to LFS, never to the main object database.
Region 4
Task context & assignment manifests

VR tasks are grouped into three streams: (a) scene construction, (b) instrumentation, (c) AI-agent behaviour. Each manifest specifies the measurable outcome the scene or module is supposed to enable.

VR-001 · Ceiling-height variant scene Well-specified
Goal
Build a single room with three swappable ceiling heights (2.4 m / 3.0 m / 3.6 m), identical in every other respect.
Outputs
Unity scene + a config file listing variant IDs and their exact geometry deltas.
Acceptance
Runtime ceiling swap via one button. Variant reports correct height to the logging system.
Verification
Automated: headless Unity test switches variants and asserts reported geometry matches config file.
VR-002 · Gaze logger with 60 Hz sampling Well-specified
Goal
Log participant head/eye direction at ≥ 60 Hz to a CSV, with timestamp, position, forward vector, and currently-fixated object ID.
Outputs
GazeLogger.cs + CSV output schema.
Acceptance
30-second recording yields ≥ 1800 rows. Object-ID field non-null ≥ 90 % of frames.
Verification
Automated: fixture scene + test script loads the CSV and asserts row count, column completeness, and timestamp monotonicity.
VR-003 · LLM-driven wayfinding NPC Partially specified
Goal
An NPC that answers "how do I get to X?" using an LLM conditioned on a map prompt.
Open question
How to functionally verify that the NPC's directions are correct without a ground-truth router. Current plan: hand- written scenarios with known answers.
Acceptance
Passes ≥ 8 of 10 fixture scenarios.
VR-004 · Daylight / circadian condition shader Speculative
Goal
Simulate melanopic-weighted lux from a configurable window/sun system.
Why speculative
Headset photometry is imperfect; we cannot calibrate melanopic lux accurately. Deliverable may be a relative index rather than absolute lux.
Region 5
Functional verification

VR deliverables are graded by does it run and instrument correctly, not by visual polish. Two verification layers:

  1. Headless Unity test runner — loads the scene in batch mode, exercises the module's API, asserts expected outputs.
  2. On-headset smoke test — another student dons the Quest, runs the build, and confirms the scene is performant (≥ 72 fps) and interaction works.

A scene or module that fails layer 1 does not merge regardless of how it looks.

scripts/verify_vr_scene.py BaselineRoom
