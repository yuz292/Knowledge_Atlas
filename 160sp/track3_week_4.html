Week 4

Question for the week: How do you expand from one proof of concept to a small library of reliable transformations?

Use AI well

Before you ask AI for help, make sure you can state what you are trying to learn this week and what evidence would count as a useful answer.

Class meetings

Meeting A — Build the first operator library
Start here: State which room-edit operators the class most needs to make reliable now.

In class

  • Work on three class-priority operators: ceiling/volume, wall geometry/curvature, and window or enclosure changes.
  • Use the validation checklist as you go.
  • Keep a visible list of what each operator changes, what it never changes, and where manual cleanup is still required.
  • Ask AI for alternative implementation strategies when one path is blocked, but compare them against your asset and tool reality.

Home

  • Push each operator onto at least two rooms from the corpus.
  • Log which rooms support the operator well and which do not.
  • Write a short note on portability: how much of the operator depends on one specific asset structure?
Meeting B — Compare operators across rooms
Start here: Bring evidence from at least two rooms so the comparison is about portability, not one lucky case.

In class

  • Present operator results side by side across multiple rooms.
  • Discuss where operator definitions need to become more conservative.
  • Revise the benchmark-room notes so they reflect what you now know from testing rather than what you guessed in Week 1.
  • Agree on which operator behaviors are required for the final demo and which are optional.

Home

  • Produce cleaner documentation for the operator library.
  • Add screenshots that make the differences visible at a glance.
  • Mark every operator as robust, partial, or experimental.

Week deliverables

D7 Operator library v1

Rationale
By this point the studio should have more than one isolated demo. The operator library is the first reusable technical core.

What to submit
A compact library description covering the implemented operators, their status, parameters, supported room conditions, and known limitations.

What good work looks like
The library is honest about what is reliable. It is documented at a level another team could test. It includes evidence from more than one room.

Example
A table listing each operator, required room features, pass/fail results on benchmark rooms, and notes such as 'works on simple office shells; fails when trim is merged into wall mesh'.
