Run one real topic through a first working AF path

This week shifts from setup to operation. The point is to prove that one real topic or question can pass through discovery, triage, and first-pass review without dissolving into hand-waving.

This week's question

Can we run one concrete topic through Article Finder and show which candidates are accepted, which are borderline, which are duplicates, and why?

Why this week matters

By the end of this week you should have one genuinely operational example, not a speculative architecture. That example may still be small. Small is acceptable. Unproved is not.

Choose one topic and stay with it long enough to learn from it

Changing topics every day hides failure. Pick one topic, one question bundle, or one research front and keep enough pressure on it to expose duplicates, weak metadata, and borderline cases.

What you need to understand before building

Corpus growth and triage quality are different

Adding papers is easy. Showing that the added papers are relevant, non-duplicate, and defensibly prioritized is the real work.

Borderline papers are informative

If a paper sits between reject and send-to-eater, that is where the question filter, constitutions, and review logic become valuable.

Review is part of the system

A human-reviewed sample is not an admission of failure. It is how you learn whether automated ranking and triage are credible.

Real topics drift

The first query will often retrieve papers that are adjacent, not central. That drift should be measured and documented.

How to use AI this week

Ask AI for

  • query variants or synonym expansions for one topic
  • a candidate benchmark set of obvious accept / obvious reject / borderline papers
  • expected failure modes in metadata enrichment or relevance scoring
  • a review rubric for checking the top ten returned papers

Do not ask AI to

  • declare relevance without you reading the title and abstract
  • collapse borderline cases into accept just to make the counts look stronger
  • generate fake "representative" outputs instead of using your real run logs

In-class activities

  1. 20 minutes: choose a topic, research front, or starter question constitution.
  2. 25 minutes: run one AF path such as discover, citations --fetch, or a staged intake batch.
  3. 20 minutes: inspect the top returned candidates and manually label 5 likely accepts and 5 borderline or reject cases.
  4. 15 minutes: compare notes with a partner and identify the first false-positive pattern.

At-home work

  • rerun your chosen topic path cleanly and record what changed and what did not
  • improve one weak point: query wording, dedup threshold, metadata recovery, or borderline review logic
  • prepare a small evidence bundle that shows the topic can be handled repeatably, not just once

Deliverables

Deliverable A real topic run

Seed-corpus run with evidence

Rationale
You need one real slice of the literature, not only a repo diagram.
What to submit
A short run report naming the topic, the commands you used, the number of candidate papers returned, and a sample of the resulting records or IDs.
What good looks like
The report shows a small but defensible seed set, not an inflated count with no inspection.
Required verification
Include raw counts, date/time of the run, and one repeated run or explanation of drift if API sources changed.
Common false positives
Counting everything returned by a broad query as useful; claiming the run is reproducible without recording the exact command and thresholds.
Example
python cli/main.py discover --email you@ucsd.edu --iterations 1 --expansion-limit 20 --pdf-limit 5
Deliverable B review sample

Manual review packet: five accepts and five borderline cases

Rationale
The system is only useful if you can tell strong candidates from plausible-looking noise.
What to submit
Ten reviewed records: five you would currently send onward, and five you would mark review or reject, each with one or two sentences explaining why.
What good looks like
Your explanations refer to observable evidence: topic fit, article type, outcomes, or metadata quality. They do not rely on vague intuitions.
Required verification
At least one borderline paper must be included because it exposed a weakness in your current filter or query.
Common false positives
Only showing easy accepts; using AI labels as if they were the review; excluding all edge cases because they make the report look messy.
Example
A nature-and-attention run that includes one restoration-only paper marked review because it lacks a direct attention measure.
Deliverable C repair loop

One focused repair or clarified spec

Rationale
Week 4 should not end with "we noticed problems." It should end with one concrete repair, tightened threshold, or clarified review rule.
What to submit
A branch, patch, or tightly written spec note that changes one thing in response to observed evidence.
What good looks like
The repair links directly to a failure from your run or review packet.
Required verification
Show the before/after behavior or explicitly state why the change is documentation-only and what future run will verify it.
Common false positives
Changing three variables at once; improving prose without reducing uncertainty; saying "fixed" with no rerun.
Example
Lowering an overly broad query variant, then rerunning the same topic and showing the borderline count dropped while core papers stayed present.

Verification checklist

  • ✓
    One real topic: you can name it and reproduce the exact command used to fetch or stage papers.
  • ✓
    One reviewed sample: you have five strong candidates and five problematic or borderline ones, with reasons.
  • ✓
    One false positive: you can point to a paper that initially looked relevant but should not automatically pass.
  • ✓
    One repair: you changed either the code, the query, or the spec in response to evidence.
  • ✓
    Pass/fail basis: your evidence bundle makes it clear what succeeded and what remains uncertain.

Exit condition for the week

You can move on when you have one topic path that another student could rerun, inspect, and challenge. The exact counts may still improve later. The important thing is that the path is now explicit, inspectable, and no longer imaginary.
