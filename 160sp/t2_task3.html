TRACK 2 · TASK 3

Search Execution & Abstract-First Triage

Wire the four-scraper harvest layer (SerpAPI, scholarly, paper-scraper, and PDF-only-last-resort scidownl), write every candidate into the article_references table, walk each row through the three-stage triage funnel (metadata-only, then abstract-and-classifier, then PDF retrieval), and acquire PDFs only for candidates that have cleared abstract triage — never before. The proof of the pipeline is a PRISMA funnel reconstructed directly from article_references.

Track: Article Finder 75 points PRISMA deliverable

Track: Article Finder  ·  Prerequisite: Task 2 (you need your ranked gap list and query pairs).

What you build: Run your search queries through SerpAPI (which scrapes Google Scholar), collect each paper's abstract through a fallback chain of free academic APIs, decide at the abstract level whether each paper is worth keeping, and report the whole funnel on a PRISMA-style dashboard.

Core lesson: Never download a PDF to decide whether it is relevant. Triage at the cheapest level — abstracts — using free APIs. Then prove the pipeline works with real PRISMA funnel numbers.

What PRISMA Is and Why You Are Building One

PRISMA (Preferred Reporting Items for Systematic Reviews and Meta-Analyses) is the gold-standard reporting checklist for systematic literature searches. It forces you to document your funnel transparently — every paper that came in, every paper that fell out, and the reason for each removal.

Records identified through database searching        (n = ?)
    ↓
Duplicates removed                                    (n = ?)
    ↓
Records screened by title + abstract                  (n = ?)  →  Excluded (n = ?)
    ↓
Full-text articles assessed for eligibility           (n = ?)  →  Excluded with reasons (n = ?)
    ↓
Studies included in final synthesis                   (n = ?)

Your dashboard must show these numbers. They are the proof that your pipeline works.

Phase 1: Understand the Search & Triage Architecture

Four pieces have to fit together: SerpAPI to find papers, an abstract fallback chain to enrich them, the triage logic to classify them, and a deduplication check to skip what we already have. This phase walks you through each piece before you start building.

1A. Understand SerpAPI

SerpAPI scrapes Google Scholar and returns the results as structured JSON. Ask your AI:

AI prompt

"Read the SerpAPI Google Scholar documentation. What fields does it return per result? Does it return full abstracts? What about DOIs?"

What you should learn: Each SerpAPI result gives you:

  • title, link, snippet (a 2–3-sentence fragment, NOT the full abstract)
  • publication_info (authors, venue, year)
  • inline_links.cited_by.total (citation count)
  • Sometimes a resource.link (a direct PDF link)
  • It does NOT reliably return DOIs or full abstracts

So SerpAPI gets you only halfway. You then need a second step that looks each paper up by title or DOI to fetch the full abstract.

1B. Understand the abstract fallback chain

The Article Eater ships with working API clients for the major free abstract sources. When SerpAPI hands you a title but no abstract, you try each source in order:

SerpAPI result (title + snippet + maybe DOI)
  ↓ extract DOI from link if possible
  1. Semantic Scholar (fetch_by_doi or search by title) → abstract?
  2. CrossRef (fetch by DOI) → abstract?
  3. PubMed (search by title) → abstract?
  4. OpenAlex (api.openalex.org/works/doi:XXX) → abstract?
  5. If ALL fail → tag: MISSING_ABSTRACT

Ask your AI:

AI prompt

"Read Article_Eater/src/services/paper_fetcher.py. Find SemanticScholarClient, CrossRefClient, and PubMedClient. Each has a search() method and a fetch() or fetch_by_doi() method. Show me how I would: (1) take a title from SerpAPI, (2) search Semantic Scholar for it, (3) get the abstract from the result."

Once you have abstracts, you classify each paper into one of five buckets:

Phase 2: Wire the search runner — four scrapers, one harvest layer

Four scrapers, not one. The SerpAPI integration below is the primary harvest channel and the one you must wire end-to-end. Three additional scrapers extend the layer: scholarly (free Google Scholar fallback when SerpAPI's 250-credit/month quota is exhausted), paper-scraper (free preprint-server channel: arXiv, bioRxiv, medRxiv, chemRxiv), and scidownl (PDF retrieval only, last resort, see Phase 5 for the policy gate). All three optional scrapers write into the same article_references table as SerpAPI; only discovered_via differs. Wire SerpAPI first; add the others as your contract grows.

The search runner is the first executable stage of the Task 3 pipeline. It takes the query pairs from Task 2 and turns them into raw SerpAPI results.

Phase 3: The article_references table — every candidate, one row

Before any candidate is triaged, abstract-collected, or downloaded, it must live as a row in the article_references table in pipeline_lifecycle_full.db. This table is the candidate buffer between harvesting and acquisition: it holds papers the system knows about but does not yet have a PDF for. Without it, every harvester re-deduplicates on its own, the PRISMA dashboard cannot count, and your pipeline is unverifiable.

3A. The schema you write into

The full DDL is in scripts/coordination/lifecycle/schema.sql and documented in the AF pipeline recon doc §3. The columns your search runner and harvesters must populate on insert:

  • Identity: reference_id (generated, format REF-YYYY-MM-DD-NNNNNN), doi (normalised, lowercased, no URL prefix), title_raw, title_normalized, first_author_surname, publication_year, venue.
  • Provenance: discovered_via (one of review_pdf_extract, serpapi_scholar, scholarly_search, paperscraper_search, openalex_expansion, crossref_search, student_upload); discovered_from_paper_id (FK to papers, when harvested out of an existing PDF); discovered_query; discovery_run_id.
  • Triage state (initial, on insert): triage_stage = 'metadata_only'; discovered_at = ISO 8601 timestamp.
  • Raw evidence: raw_citation (the messy reference-list line as captured), snippet (search snippet or abstract fragment).

3B. Deduplication on insert (do this before every INSERT)

  1. Normalise the candidate's DOI with the existing normalize_doi() helper from build_neuro_review_acquisition_queue.py.
  2. If a row exists with the same doi, do not insert. Instead UPDATE … SET discovered_via = discovered_via || ', ' || NEW.discovered_via so you preserve the multi-channel provenance.
  3. If no DOI but a fuzzy-matched title_normalized already exists in pdf_identity_inventory above the existing-corpus threshold, insert with triage_stage = 'duplicate' immediately.

3C. Two harvesters, one table

Both the four-scraper search runner (Phase 2 above) and the review-PDF reference harvester write into this same table. The review-PDF harvester is largely already written for you in the AE coordination scripts:

  • scripts/coordination/extract_neuro_key_review_references.py — extracts reference lists from review-kind PDFs using pdfplumber. Use its logic; do not write a parallel parser.
  • scripts/coordination/build_neuro_review_acquisition_queue.py — canonical DOI normaliser and corpus-dedupe filter. Reuse its functions in your insert path.
  • Real harvested data already exists at /Users/davidusa/REPOS/_Collecting Articles/Neuro key articles/_atlas_inventory/latest_neuro_review_reference_harvest.json — 46 review PDFs, dictionary of {doi: cite_count}. Use this as your prototyping corpus before you wire the live harvester.
Free-floating outputs do not count. A reference list extracted into a JSON file that never lands in article_references is invisible to the rest of the pipeline. Every harvested candidate must be inserted (or deduplicated against an existing row) before any later phase touches it. Your grader checks for this.

Phase 4: Three-stage triage — metadata, abstract, PDF

The pipeline you build has one cardinal rule: never download a PDF to decide whether you want the paper. The three-stage triage funnel exists to enforce that rule cheaply and defensibly. Sub-phase 4A is the metadata-only screen (Stage 1). Sub-phase 4B is the abstract collection that follows (Stage 2). Sub-phase 4C is the triage decision on the abstract. Stage 3 (PDF retrieval) is the subject of Phase 5; it runs only after a row reaches triage_decision = 'ACCEPT' here.

4A. Stage 1 — metadata-only triage

Runs immediately at ingest, before any abstract is fetched. Pass the candidate's title and venue through the atlas_shared classifier; if classifier confidence is below threshold (suggested: 0.20), set triage_stage = 'rejected_at_metadata' and never enrich further. This is where 30 to 50 per cent of search noise should die without spending a Semantic Scholar request, an OpenAlex polite-pool slot, or a SerpAPI credit.

4B. Stage 2A — collect abstracts (only for survivors of 4A)

SerpAPI hands you titles and snippets. To triage, you need full abstracts. This sub-phase walks each Stage-1 survivor through the fallback chain (Semantic Scholar → CrossRef → PubMed → OpenAlex) until a real abstract appears — or until every source comes up empty and you tag the paper MISSING_ABSTRACT.

4C. Write YOUR OWN abstract-collector contract

Contract objective: "I want a program that takes SerpAPI results (which have snippets, not abstracts) and finds the full abstract for each paper from free academic APIs."

Contract is with: The SemanticScholarClient, CrossRefClient, PubMedClient in Article_Eater/src/services/paper_fetcher.py, and the OpenAlex API.

Prompt hint: "I need a contract for an abstract collector. It takes search results with DOIs and tries to find full abstracts from Semantic Scholar, CrossRef, PubMed, and OpenAlex in fallback order. Papers with no abstract from any source get tagged MISSING_ABSTRACT. Help me write the contract."

Minimum bar your contract must cover:

  • Takes SerpAPI results as input (with DOIs where available)
  • Tries multiple abstract sources in fallback order (S2 → CrossRef → PubMed → OpenAlex)

  • What counts as a "found" abstract versus a snippet?
  • How do you handle ambiguous title matches?

Phase 4D: Stage 2B — the triage decision

With abstracts in hand, the pipeline can finally make decisions. Triage is the choke point: it converts a long list of candidate papers into a short list of papers worth downloading.

4A. Write YOUR OWN triage contract

Contract objective: "I want a program that reads each paper's abstract and decides: is this paper worth downloading?"

Contract is with: The atlas_shared classifier (from Task 1) and score_voi() from Article_Eater/src/cmr/voi_scoring.py.

Prompt hint: "I need a contract for an abstract triage module. It runs each abstract through the atlas_shared topic classifier, then scores it with score_voi(). Output is a 4-way classification: ACCEPT, EDGE_CASE, REJECT, or MISSING_ABSTRACT, each with a human-readable reason. Help me write the contract."

Minimum bar your contract must cover:

  • Runs each abstract through the atlas_shared classifier (topic matching)
  • Scores each abstract with score_voi() from cmr/voi_scoring.py
  • Produces a four-way classification: ACCEPT / EDGE_CASE / REJECT / MISSING_ABSTRACT
  • Attaches a human-readable triage_reason to every decision
  • Stores ACCEPT papers in the lifecycle DB; stores EDGE_CASE separately

Success conditions you must define:

  • What is the minimum number of papers triaged?
  • What classifier confidence threshold separates on-topic from off-topic?
  • What VOI threshold separates ACCEPT from EDGE_CASE?

4B. Write your tests BEFORE you build

  • Every triaged paper carries a triage_decision field
  • Every triaged paper carries a triage_reason (never empty)
  • ACCEPT papers appear in the database
  • EDGE_CASE papers are stored but flagged
  • REJECT papers are logged, not silently dropped
  • MISSING_ABSTRACT papers skip triage rather than getting scored as REJECT

Phase 5: PDF acquisition cascade and the scidownl policy gate

Stage 3 of the funnel is PDF retrieval. It runs only for rows where Phase 4 set triage_decision = 'ACCEPT', and it walks a strict source cascade in order. Every attempt is recorded in pdf_acquisition_attempts (count) and pdf_acquisition_last_source (which source was tried last). On success, the new PDF is added to papers and the originating article_references row's acquired_paper_id is set to the new paper_id.

5A. The source cascade (in order)

  1. Unpaywall — the existing ingest/pdf_downloader.py handles this. Always tried first; legal, free, license-clean.
  2. OpenAlex OA URL — tried second. The open_access field on the OpenAlex record carries a direct PDF URL when the work is OA. Free, license-clean.
  3. scidownl — tried only if the previous two fail and the policy gate (5B) is passed. Sci-Hub-mirror-backed; legality varies by jurisdiction; UCSD library policy unresolved.

5B. The scidownl policy gate (read this in full before Phase 5 begins)

scidownl is not a default scraper and not a dependency you wire in unconditionally. It is a last-resort PDF source with an unresolved institutional policy question. Your pipeline must satisfy all four of the following before a single scidownl call is made:

  1. The config flag enable_paid_or_grey_sources is set to true in your YAML config (defaults to false).
  2. The file policy_clearance.json exists in your project root and is countersigned by your instructor (a single line of text plus a date is sufficient; the file's existence is what the gate checks).
  3. Both Unpaywall and OpenAlex OA have failed for this specific reference_id in the current run; you log pdf_acquisition_last_source reflecting that.
  4. The row's triage_decision is exactly 'ACCEPT'; EDGE_CASE rows do not get scidownl attempts.

Every scidownl call writes pdf_acquisition_last_source = 'scidownl' and a row to lifecycle_transitions with the reference_id, the timestamp, and the success/failure outcome. There is no silent fallback. Talk to your instructor before the flag is turned on; do not make this decision unilaterally.

5C. The acquisition queue (where the worker reads from)

Use the convenience view v_acquisition_queue, which surfaces every ACCEPT row whose acquired_paper_id is null, sorted by voi_score descending. Your PDF-retrieval worker reads from this view and walks the source cascade for each row. Rows whose acquisition fails through every cascade source remain in the queue with their pdf_acquisition_attempts incremented; the dashboard surfaces them as the wanted-but-unobtainable bucket.

Two named failure modes to avoid. (1) Downloading PDFs before triaging — the funnel state machine in triage/funnel_state.py is what stops this. Do not write a parallel acquisition path that bypasses it. (2) scidownl as default rather than last-resort — the four-condition policy gate above is what stops this. The flag defaults to false, the clearance file does not exist by default, and the cascade only reaches Step 3 after Steps 1 and 2 both fail.

A Note About Reuse

The PRISMA funnel you just built is not a one-off deliverable. Every future task that adds papers to the corpus will require the same funnel, updated with new numbers. Design your dashboard with that in mind — when the next task runs new searches, the same page should refresh with new counts rather than be rebuilt from scratch. Treat the PRISMA dashboard as durable infrastructure for the rest of the course, not a deliverable you ship and forget.

  2. On GitHub, open a Pull Request from <YOUR-USERNAME>:track2/<your-name> to dkirsh:main
  3. Title the PR: Track 2 · Task 3: Search Execution & Triage — <your name>
  4. In the PR body, paste your file manifest (run git diff --name-only upstream/main) and a one-paragraph summary of what you built and what you tested. Include your PRISMA funnel numbers.
  5. Add the label track2-task3-review and tag the instructor (@dkirsh).
  6. Leave the PR open. Do not merge. The instructor will review and either merge, request changes, or comment for follow-up.

Task 3 is the last task in Track 2. After review, the instructor will signal closure on your track. If you want to keep iterating on the dashboard or extend the pipeline, leave your branch in place and open follow-up PRs against the merged Task 3 commit.

If you get stuck

  • Merge conflict: git fetch upstream && git rebase upstream/main, resolve conflicts, then git push --force-with-lease.
  • Forgot to commit before switching machines: git stash on the old machine, push, then git stash pop after pulling on the new machine.
  • Unsure if the instructor has reviewed: the PR page on GitHub shows review state. Don't email; let the PR system carry the conversation.
