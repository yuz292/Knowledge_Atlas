Sprint 1 Planning: Contracts and Specifications

Git setup — required before you begin track work

Every student must have a working git setup by the end of today. This means: a GitHub account (you should have created one in Week 1), git installed on your laptop, the Knowledge Atlas repo cloned locally, and a personal branch created. If you are new to git, pair with a teammate who has used it before — this is a 15-minute setup, not a full git tutorial.

Steps: (1) Install git if needed — brew install git on Mac, or download from git-scm.com. (2) Clone the repo: git clone https://github.com/dkirsh/Knowledge_Atlas.git. (3) Create your branch: git checkout -b track/N-staging/your-username where N is your track number. (4) Push it: git push -u origin track/N-staging/your-username. (5) Confirm your branch appears on GitHub.

Read Your Track Assignment

2

Write Your Sprint 1 Contract

A contract is a one-page document that specifies what you will build, what inputs it requires, what outputs it produces, how you will test it, and what "done" means. This is step 3 (Spec) of the seven-step workflow, applied to your first sprint. A clear contract prevents misunderstandings mid-sprint and gives you a concrete way to know when you are finished.

What a contract looks like

Title: [Sprint 1 deliverable name]. Input: [what data or files the component receives]. Output: [what it produces]. Acceptance test: [3–5 concrete conditions that must be true for the deliverable to pass]. Dependencies: [what other components or data this relies on]. Estimated time: [hours].

3

Peer Review of Contracts

Swap contracts with another team. Check: Are the acceptance criteria testable and concrete? Are the inputs and outputs clearly specified? Could you verify this deliverable without reading the code? Use peer feedback to strengthen your contract before submitting it for approval.
