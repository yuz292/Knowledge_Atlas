Step-by-Step Walkthrough

Follow these steps exactly. No programming experience required. The only part that requires your own thinking is composing the prompts you send to the AI.

What This Assignment Actually Is

You are going to build a working software component — a real piece of the Knowledge Atlas system — without writing a single line of code yourself. An AI will write all the code. Your job is something harder and more important: you will direct the AI by thinking carefully about what the component should do before asking the AI to build it.

This matters because AI writes code fast but thinks about requirements poorly. If you say "build me a search filter," it will — and the result will look professional, seem to work, and quietly fail on half the cases that matter. The component will search the wrong fields, display misleading numbers, or crash on edge cases that any thoughtful person would have anticipated. The AI's code is only as good as the instructions you give it.

The assignment teaches you a method for giving those instructions. It has seven steps. Steps 1 through 3 happen before any code is written — they are about thinking, not programming. Steps 4 through 7 are about building and testing. The entire process takes 60–80 minutes. You will submit a document showing every step, including your mistakes.

Here is the sequence:

The seven steps, in plain language: 1. Ask the AI to pretend to be a panel of experts who argue about what the component should do. 2. Write down exactly what the component takes as input, produces as output, and how to know if it worked. 3. Write specific test cases — before any code exists. 4. Tell the AI to build it according to your specification. 5. Run the tests. See what fails. 6. Figure out why it failed — was your specification wrong, or did the AI build it wrong? 7. Fix the problem. Test again. Repeat until it works.

That is the whole assignment. What follows is exactly how to do each step.

Before You Begin: Three Things You Need

You need an AI chat (Claude at claude.ai, ChatGPT, or similar), a web browser, and your context file. That's it. No software to install. No code editor. No terminal. No GitHub account.

The context file is a document that describes the data your component will work with — the field names, the data types, what the numbers mean, and where the data lives. Your AI cannot see the Knowledge Atlas database. It does not know what warrant_discount means or that credence ranges from 0 to 1. The context file gives it that knowledge. Without it, the AI will guess — and guess wrong.

The Steps

0 Give the AI Your Context File

Open a new chat with your AI. This should be a fresh conversation — don't continue an old one.

Do this: Drag the context file you downloaded into the chat window, or click the attachment/paperclip button to upload it. Most AI tools (Claude, ChatGPT) accept file uploads directly.

If file upload doesn't work, open the downloaded .md file in any text editor, select everything (Ctrl+A or Cmd+A), copy it (Ctrl+C or Cmd+C), and paste it into the chat.

Then type this message and send it:

Copy & Paste This I've attached the context file for my exercise. Before we do anything else, please explain: 1. What is this task about, in your own words? 2. What are the main data fields I'll be working with? 3. What do you think will be the hardest part? 4. What could go wrong if we're not careful?

Copy the text above exactly. This is not the creative part — this is setup.

Read the AI's response carefully. Does its description match what the context file says? If it mentions field names that aren't in the context file, or misunderstands what a field means, correct it now. Every step after this depends on the AI understanding the data correctly.

If you skip this step, everything downstream breaks. The panel will discuss the wrong requirements. The spec will define the wrong inputs. The code will use the wrong field names. It will all look fine until you test it — and then nothing will work, and you won't know why.
1 Convene the Expert Panel

Now you ask the AI to simulate a panel of experts who argue about what your component should do. This is not decorative — the panel's job is to surface requirements that you would not have thought of on your own. The experts should disagree with each other. If they all say the same thing, the panel failed.

Do this: Go to the exercises page, find your assigned exercise (A, B, C, or D), and look for the section labeled "Step 1 — Convene Panel." You will see a gray box containing a prompt. Copy that entire prompt and paste it into your AI chat.

The prompt names specific experts (real researchers and designers) and tells the AI what role each one plays. The AI will generate a simulated discussion where the experts argue about what matters most.

What you're looking for: At least one requirement that surprises you — something you would not have included if you were just writing a feature list. If the panel produced three experts who all politely agree, tell the AI: "The experts are being too agreeable. Make them actually disagree about something substantive." Real experts disagree. That's where the good requirements come from.

Save the entire panel transcript. You will submit it as part of your deliverable.

2 Write the Specification

The panel gave you a set of requirements — some from each expert, with some disagreements. Now turn those into a precise specification: what the component takes as input, what it produces as output, what counts as success, and what the edge cases are.

Do this: Send a message like this to the AI:

Adapt This to Your Exercise Based on the panel discussion, write a specification for this component. Include: - INPUT: What does the component receive? (be specific about types) - OUTPUT: What does it produce? (what does the user see?) - SUCCESS CONDITIONS: How do we know it worked correctly? - EDGE CASES: What unusual inputs should it handle? Use the actual field names from the context file.

You may adapt this — it's a template, not a script.

Read the spec the AI produces. Does it account for the panel's key disagreements? If the panel argued about whether to use additive vs. multiplicative scoring (Exercise C), the spec should state which approach you chose and why. If it argued about two-column vs. force-directed layout (Exercise B), the spec should commit to one.

If something is missing, tell the AI what to add. This is your specification — you own it.

3 Write Test Cases Before Any Code

This is the step most people want to skip. Don't. Writing tests before code forces you to think about what "working" actually means — before you have code to be distracted by.

Do this:

Copy & Paste This Now write at least 4 acceptance tests for this component. Each test must specify: - INPUT: What exactly do we feed it? - EXPECTED OUTPUT: What should appear? - HOW TO VERIFY: How do we check it worked? Include at least: - One normal case (typical input, should work perfectly) - One edge case (unusual but valid input) - One failure case (invalid or missing input — should fail gracefully) - One visual check (does it look right?)

The AI will produce tests like "Input: 'natural light', 'task performance'. Expected: a directed graph with at least 2 intermediate nodes." Each test is a bet about what the component should do. When the code is built, you will check whether the bet was right.

You have now finished the thinking part. Steps 1–3 are complete. You have a panel transcript, a specification, and test cases — all before a single line of code. This is the method. Everything that follows is execution.
4 Ask the AI to Build It

Now — and only now — you tell the AI to write the code.

Copy & Paste This Build the component according to the specification above. Create a single, self-contained HTML file that I can open in a browser. For the data, embed 20-30 realistic sample records directly in the JavaScript as a const SAMPLE_DATA array. Base the sample records on the schema from the context file — use realistic field values, not placeholders. Use the same visual style as the Knowledge Atlas site: cream background (#FAF6F0), Georgia headings, system-ui body text.

The "embed sample data" instruction is important — it means the HTML file works anywhere, no server needed.

The AI will produce a complete HTML file. It may be long — that's fine. You don't need to read the code. You need to run it.

How to run it:

If your AI has a built-in preview (Claude shows "artifacts," ChatGPT shows "canvas"), the component should appear right there in the chat. You're done — move to Step 5.

If not: ask the AI to show you the complete HTML. Copy everything, paste it into a plain text file, save it with a .html extension (e.g., my_exercise.html), and double-click to open it in your browser.

5 Run Your Tests

Go back to the test cases you wrote in Step 3. Run each one against the component. For each test, record:

PASS — the component did what the test predicted.
FAIL — the component did something else. Write down exactly what happened.

Be honest. If a test is ambiguous and you're not sure whether it passed, write that down too. The grading rubric rewards honest verification, not a perfect score.

Most components will fail at least one test. That is expected. That is the point. If everything passes on the first try, either your tests were too easy or you got lucky. The next two steps — diagnosis and iteration — are where the learning happens.
6 Diagnose the Failures

For each failed test, you need to figure out why it failed. There are exactly two possibilities:

The spec was wrong. You asked for the wrong thing. The test expected behavior that doesn't make sense given the data, or your specification was ambiguous and the AI interpreted it differently than you intended.

The implementation was wrong. Your spec was correct, but the AI built it incorrectly — a bug, a misused field name, a calculation error.

This distinction matters. If the spec was wrong, you need to fix the spec and then rebuild. If the implementation was wrong, you tell the AI what went wrong and ask it to fix the code.

Do this: Tell the AI about the failure and ask it to diagnose:

Adapt This Test 2 failed. I expected [what you expected], but instead I got [what actually happened]. Look at the specification and the code. Is this a problem with the spec (I asked for the wrong thing) or the code (you built the right thing incorrectly)?
7 Fix and Re-Test

Based on the diagnosis, ask the AI to fix either the spec or the code. Then re-run the failing test. If it passes, move on. If it fails differently, go back to Step 6.

You do not need to achieve a perfect pass rate to get a good grade. You need to show that you tried, diagnosed, and iterated. A component with one unresolved failure and a clear explanation of why is worth more than a component with no failures and no evidence of testing.

8 Write Your Reflection (Two Paragraphs)

Paragraph 1: Identify one requirement that came out of the panel discussion that you would not have thought of on your own. Explain why it surprised you and how it changed the specification.

Paragraph 2: Describe one specific failure — a test that didn't pass — and how you diagnosed it. Was it a spec problem or an implementation problem? What did you do about it? What did you learn?

Avoid generic statements. "The AI was really helpful" is not a reflection. "The AI generated a search filter that looked perfect but silently dropped all claims with empty defeat_type fields, which is 93% of the dataset" — that is a reflection. Be specific. Name the fields. Describe the failure concretely.

What You Submit

One document (Google Doc, Word, or PDF) containing everything in this order:

1. Panel transcript — the full simulated expert discussion from Step 1.

2. Specification — inputs, outputs, success conditions, edge cases from Step 2.

3. Acceptance tests — all test cases with expected outcomes from Step 3.

4. All prompts you sent to the AI — including ones that didn't work. This is important: we want to see the failed attempts, not just the successes.

5. The AI's code output — the final HTML file (or a link to it).

6. Test results — PASS/FAIL for each test, with notes on what happened.

7. Two-paragraph reflection — from Step 8.

How This Is Graded

Each component is worth 25% of the grade. Notice that none of them are "does the code work perfectly."

Panel quality (25%) — Did the experts actually disagree, or did they just list features? A real argument between panelists about what matters most is worth far more than three experts who politely agree.

Specification completeness (25%) — Are inputs, outputs, edge cases, and success conditions defined before the code was written? If the spec is vague, the code will be too.

Verification rigor (25%) — Did you run the tests? Did you catch real failures? Did you diagnose them? A component that fails one test with a clear explanation beats an untested component that "seems to work."

Reflection honesty (25%) — A genuine surprise and a genuine failure, described concretely. Not "the AI is amazing" or "I learned a lot."
