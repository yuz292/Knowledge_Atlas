Track 2 · Week 7

Freeze a reproducible demo and say what the system cannot yet do

The final week is not about pretending the system is finished. It is about turning the strongest current path into a reproducible demonstration, documenting limits clearly, and leaving the next team with something they can trust.

This week's question

What can we demonstrate reliably now, under controlled conditions, and what should we explicitly refuse to claim?

Why this week matters

A final demo that hides uncertainty damages the project. A final demo that is smaller but reproducible, honest, and well documented moves the project forward. This week is about confidence through constraint.

Do not oversell

If the pipeline only works reliably on a few topics, say that. If the question filter only has one drafted constitution, say that. The class should be able to trust your limits statement.

What you need to understand before building

Reproducibility beats surface polish

A plain demonstration that another student can rerun is worth more than a polished story no one can verify.

Claims require evidence pointers

Every claim in the final presentation should point to a command, report, screenshot, benchmark, or bundle.

Limitations are part of the deliverable

The system is unfinished. Your job is to state precisely where it is strong, where it is weak, and what next work is required.

Handoff quality includes future usability

A final runbook, benchmark, and failure log are part of the artifact you hand to the next team, not optional extras.

How to use AI this week

Ask AI for

  • a final demo checklist with explicit evidence slots
  • a concise capabilities-versus-limitations table
  • a red-team pass over your runbook: what would confuse a new student?
  • likely demo failure modes and fallback plans

Do not ask AI to

  • inflate the claim language for presentation value
  • write a limitations section that hides the real weak points
  • replace the actual demo path with a hypothetical ideal future path

In-class activities

  1. 20 minutes: pick the one demonstration path you trust most.
  2. 20 minutes: rehearse the run from a clean note or runbook, not from memory.
  3. 20 minutes: have another student red-team the path and note where they became uncertain.
  4. 20 minutes: finalize the capabilities/limitations table and decide what you will explicitly not claim.

At-home work

  • freeze the final demo branch or commit and record it
  • finish the runbook and evidence bundle
  • prepare the final handoff note: what works, what does not, and what the next team should do first

Deliverables

Deliverable A runbook

Reproducible runbook and demo script

Rationale
The most valuable final artifact is a path that another student or TA can follow without guessing what you meant.
What to submit
A clean runbook listing repo, branch or commit, setup assumptions, commands, expected outputs, and what to inspect at each step.
What good looks like
A teammate can execute or audit the path without additional oral explanation.
Required verification
At least one other person must read or follow the runbook and confirm where it was clear or unclear.
Common false positives
A personal memory aid instead of a real runbook; missing commit hash; commands with unstated local assumptions.
Example
"Activate venv; run python cli/main.py doctor; run the saved topic command; inspect accepted IDs; run build-jobs; open the generated paper.json."
Deliverable B limits

Capabilities and limitations table

Rationale
The class needs to know what the current AF system can do confidently and what remains fragile or partial.
What to submit
A concise table with columns for capability, current evidence, confidence, limitation, and next recommended improvement.
What good looks like
Each line is tied to evidence rather than aspiration.
Required verification
Every capability line must point to a benchmark, run report, screenshot, or bundle. Every limitation line must describe an observable weak point.
Common false positives
Turning unfinished goals into current capabilities; describing limitations so vaguely that they cannot guide the next team.
Example
"Can build AE-ready bundles for selected accepted papers; evidence: Week 5 bundle inspection. Limitation: question-constitution coverage is still sparse and not representative across topics."
Deliverable C handoff

Final handoff pack

Rationale
The project should not reset to confusion after your demo.
What to submit
A compact pack containing the final runbook, benchmark, failure log, one validated bundle example, and your next-step recommendation.
What good looks like
A new contributor can see where to start, what to trust, and what remains unsolved.
Required verification
The handoff pack references concrete files, commands, and branches, not only verbal summaries.
Common false positives
A slide deck with no operational details; omitting the failure log because it feels negative; no recommendation for next work.
Example
A README-style handoff note plus links to the benchmark CSV, failure log, bundle directory, and the final branch.

Verification checklist

  • ✓
    Demo path frozen: the final path points to a specific repo state.
  • ✓
    Evidence-backed claims: every claimed capability has an evidence pointer.
  • ✓
    Honest limits: the system's current weaknesses are plainly stated.
  • ✓
    Future usability: the next contributor would know where to begin.

Exit condition for the week

The track is finished when the strongest AF path is both reproducible and honestly bounded. If the next student can follow your runbook, inspect your evidence, and see exactly what remains to be improved, you have done the right kind of final work.
