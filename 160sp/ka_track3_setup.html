Official Track 3 Contract

Use this rule: Track 3 students may inspect and edit assets in Blender, but the final course submission path is a browser-runnable A-Frame / WebXR scene. Unity and Unreal are optional prototype environments, not the required student baseline.

Primary tools

Blender for asset inspection and cleanup. A-Frame for the eventual submitted scene. Git and a browser for review and sharing.

Not required now

No Unreal starter repo, no headset pairing, and no large game-engine install just to begin the course.

First real goal

Build a strong benchmark room shortlist and prove that the rooms can support controlled architectural edits.

Why The Course Uses A-Frame

Advantages

A-Frame scenes run in the browser, are easy to inspect, easy to submit through Git, and much easier for TAs and future students to review than a heavy engine project.

Limitations

A-Frame is not the best path for maximal photorealism, dense physics, or the most ambitious headset-specific interaction design. Those are real limits.

Course judgment

For this class, shareability, experimental clarity, and reviewability matter more than maximal rendering power. That is why Blender plus A-Frame is the official submission path.

Prerequisites

  • A laptop that can run Blender 4.x reasonably well. A dedicated GPU helps, but many current laptops are adequate for the early weeks.
  • Python 3.11+ and Git installed.
  • Chrome or Firefox for browser scene review.
  • About 10 GB of free disk for downloaded room models, exports, and working files.
  • Optional: a Meshy or Tripo account for generating isolated props or exploratory room candidates.
If you are choosing between tools: install Blender first, not a game engine. Blender is needed immediately; Unreal and Unity are not.

Step-By-Step

1Install Blender

Download Blender 4.x from blender.org, install it, and confirm that it opens. You do not need mastery at once; you need enough fluency to import a model, navigate the viewport, inspect hierarchy, and read object dimensions.

2Verify Git, Python, and your browser workflow

Track 3 work still lives in a normal repo workflow. Confirm that Git and Python are available and that you can run a small local server when you need browser-based scene checks later.

python3 --version git --version
3Clone the course site and run it locally

You need local access to the Track 3 pages, workbook, and shared resources.

git clone https://github.com/dkirsh/Knowledge_Atlas.git cd Knowledge_Atlas python3 -m http.server 8000

Then open http://localhost:8000/160sp/ka_track3_today.html or http://localhost:8000/160sp/ka_track3_hub.html.

4Download 3 to 4 candidate room models

Your first academic task is not to build a flashy scene. It is to find room models that can support controlled architectural transformations without collapsing into unusable geometry.

Sketchfab

Best general source for downloadable interiors and room models. Prefer .glb when available.

TurboSquid

Useful when the hierarchy is clean enough to inspect and edit. Good for more professional room shells.

Poly Haven

Excellent for HDRIs, textures, and some interior assets. Useful even when the room shell comes from somewhere else.

Meshy / Tripo

Use these for isolated props or exploratory room candidates, not as the only source for the benchmark corpus.

5Import one room into Blender and inspect it

For each candidate room, check the properties that matter for later transformation work.

  • Scale: does a door look like a door rather than a tower?
  • Hierarchy: are walls, floors, ceilings, windows, and openings inspectable, or fused into one useless mesh?
  • Materials: can you identify and change meaningful material groups?
  • Openings: are windows and doors real openings or just texture illusions?
  • Editability: can you imagine changing ceiling height, wall curvature, enclosure, light, or materials without breaking everything else?
6Create a simple working folder structure

You do not need a large engine project. You need a clean workspace that maps to the weekly deliverables.

mkdir -p track3_workspace/{benchmark_rooms,generated_assets,variants,logs} cd track3_workspace
7Read the student pages in the right order

Use this sequence:

  1. Track 3 Today for the immediate task list.
  2. Week 1 for the first deliverables and evidence standards.
  3. Workbook for the long-range contract and the final scene path.

What Makes A Good Benchmark Room

Geometrically plausible

The room has realistic scale and obvious architectural structure. You can tell what is wall, floor, ceiling, window, and opening.

Transformable

The room supports controlled edits to one factor at a time, rather than requiring a total rebuild for every change.

Portable

The asset can be opened and reused outside one fragile engine path. Portable formats make the studio more robust and the handoffs cleaner.

A beautiful render is not enough. If the room is one fused mesh with no meaningful hierarchy, it is a poor benchmark regardless of how impressive it looks.

How To Use AI During Setup

Useful AI requests

  • Ask for concise explanations of scene hierarchy, pivots, materials, UVs, and export formats.
  • Ask for a checklist to judge whether a room asset is editable.
  • Ask for failure modes to watch when importing meshes into Blender or the browser.

Bad AI habits

  • Do not ask AI to decide whether a room is good without inspecting the asset yourself.
  • Do not let AI push you into a heavy toolchain just because it sounds powerful.
  • Do not confuse plausible language with verified geometry or a working scene.

Common Snags

  • Blender opens to an empty or confusing view: use Home to frame the scene or Numpad . to focus on the selected object.
  • The model is tiny or enormous: inspect object dimensions and correct scale before judging the room.
  • Textures are missing: many downloads place textures in a sibling folder rather than embedding them in the mesh.
  • Meshes from text-to-3D systems are chaotic: use them sparingly and expect cleanup before any serious evaluation.
  • Browser assets fail to load later: do not open scene files directly from the filesystem; use a small local web server.
Raise your standard: if another student cannot open the asset, inspect it, and understand your notes, the artifact is not yet ready for the shared corpus.

Optional Later Prototypes

Unity and Unreal are still intellectually relevant. If a student later wants to prototype a headset-specific extension, that can be sensible. But the baseline course path should remain lighter and clearer:

  1. benchmark rooms first,
  2. controlled transformations next,
  3. one browser-runnable research scene after that.

That order is not arbitrary. It is what makes the studio academically coherent and actually doable in the time available.
