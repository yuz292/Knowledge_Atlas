Deliverable submission

All deliverables should be submitted as files in your fork of the Article Finder repository, organized by week (e.g., deliverables/week4/). Use GitHub Issues for questions and Pull Requests for code contributions. Before merging any PR, run python scripts/verify_af_record.py locally to confirm your changes pass the functional verification harness.

Region 1
Environment setup

Before claiming any task, every Article Finder needs a working local environment. The pipeline assumes Python 3.11+, a Zotero library pointed at the shared group, and a GitHub account linked to the course organisation.

Required tooling

  • Python 3.11+ (verify with python3 --version)
  • Zotero 7 desktop + Zotero Connector browser extension
  • Git and a GitHub account enrolled in the course org
  • DOI/CrossRef access via your UCSD VPN for closed-access PDFs

One-shot bootstrap

git clone https://github.com/dkirsh/Article_Finder_v3_2_3.git
cd Article_Finder_v3_2_3
python3 -m venv .venv && source .venv/bin/activate
pip install -e .
pip install -r requirements.txt
cp config/settings.yaml config/settings.local.yaml
python3 cli/main.py doctor
python3 cli/main.py stats

Start with doctor and stats. If either one fails, post in Region 6 (Help & queue) before trying to claim tasks.

Region 3
Ways of working

The team operates on a weekly cadence with one synchronous check-in and asynchronous pair review in between.

Weekly rhythm

  1. Monday: claim a task from the queue (one at a time; complete before claiming another).
  2. Wed/Thu: submit a draft article record, assign a pair reviewer.
  3. Friday: 30-minute stand-up; defended records merge to the staging branch; contested records discussed with instructor.

Pair-review rule

No article record merges on a solo review. A second Finder must load the record, run the verification recipe, and sign off. This is structured peripheral participation (Lave & Wenger, 1991): every record is a small apprenticeship event.

Communication channels

  • #track2-article-finder Slack — quick questions, daily status
  • GitHub Issues — anything that blocks a merge
  • Weekly office hours — Thursdays 2–3 pm, CSB 155
Region 4 · Most important
Task context & assignment manifests

Each Article Finder task is specified by a manifest: a small, testable contract stating what goes in, what comes out, and what counts as "done." Click any task below to expand its manifest.

AF-001 · Seed search for "daylight and cognitive performance" Well-specified
Goal
Produce a 25-item seed corpus of empirical studies on daylight exposure and cognitive/attentional outcomes, 2010–present.
Inputs
Query templates in resources/queries/daylight_cognition.md; inclusion rubric in rubrics/voi_banding.md.
Outputs
data/af/af001_daylight_cognition.json — 25 records with DOI, title, authors, venue, year, abstract, VOI band, and inclusion justification (<80 words).
Acceptance
Pair reviewer runs scripts/verify_af_record.py af001 → exit 0. At least 18 of 25 records have full-text PDFs in Zotero group. VOI-band distribution is plausible (not all "high", not all "low").
Estimated effort
6–8 hours spread over a week.
Verification
Functional: the record file loads, schema validates, all DOIs resolve via CrossRef, ≥70 % have full text attached. No prose scoring.
AF-002 · Backward citation sweep for Ulrich (1984) Well-specified
Goal
Enumerate all articles citing Ulrich (1984, Science) that themselves contain a measured occupant outcome.
Inputs
Seed DOI 10.1126/science.6143402; Semantic Scholar API key.
Outputs
data/af/af002_ulrich_forward.json with ≥ 60 records.
Acceptance
verify_af_record.py af002 passes; at least 40 records carry a measured outcome (not review / theory-only).
Verification
Automated: count records flagged has_outcome: true and whose measurement_field is one of the approved vocabulary terms.
AF-003 · Scope note for "embodied architecture" (disambiguation) Partially specified
Goal
Separate the philosophical / phenomenological strand (Pallasmaa, Mallgrave, Robinson) from the empirical sensorimotor-simulation strand (Gallese, Jelić, Djebbara) into two corpora, with a scope note explaining the boundary.
Inputs
Glossary entries in resources/glossaries/embodied.md.
Outputs
Two record files + a 400-word scope note (markdown).
Acceptance
Scope note reviewed by instructor; corpora each have ≥ 15 records passing schema validation.
Verification
Functional on the record files; instructor sign-off on the note. Open question for this task: how to functionally verify the disambiguation itself. Currently under discussion with Prof. Kirsh.
AF-004 · Negative-result / failed-replication pass Speculative
Goal
Actively seek failed replications and negative findings in architectural-cognition studies (e.g., ceiling-height / construal, thermal priming).
Why speculative
Negative results are under-published; the yield from a single week of search is unknown. This is an exploratory task — low-yield outcomes are themselves informative.
Outputs
Whatever is found + a short note on search strategies that did / did not surface results.
Verification
Procedural: queries logged; records schema-valid. No minimum-count target.

Region 5
Functional verification

Every Article Finder deliverable is graded by whether it works, not by prose quality. The verification harness is a Python script that loads your output file, validates schema, resolves every DOI, and runs task-specific checks.

Run locally before submitting

python3 scripts/verify_af_record.py af001
# → AF-001: 25 records, schema OK, 24/25 DOIs resolve, 19/25 have PDFs
#   VOI bands: high=8 medium=13 low=4  → PASS

What the harness checks

  • JSON schema validity against contracts/af_record.schema.json
  • Every DOI resolves on CrossRef (HTTP 200, metadata returned)
  • Task-specific minima (record count, full-text coverage, VOI distribution)
  • No duplicate DOIs across records
  • Dates, venues, and author strings are well-formed

A record that passes the harness is considered mergeable. Prose review is advisory, not gating. This is deliberate: the Knowledge Atlas needs reliable structured data, and the grader needs to be able to say "yes, this works" without reading 60 pages of JSON.
