Phase 1 — Who Uses Knowledge Atlas?

Start with the user types already proposed below. Your job is to evaluate them, extend them, and assign yourself one or more types to develop in depth. A user type is defined by what that person is trying to accomplish in their professional or personal context — not by their job title alone. Use the AI prompt at the bottom of this section to generate additional candidate types.

Your deliverable for Phase 1: Select at least one user type to develop in depth. If you believe an important user type is missing from the list below, add it with a description and at least two starter scenarios. Submit your selection before beginning Phase 2.
Phase 1 · AI Brainstorm Prompt
🤖 Suggested prompt for Claude or another LLM
"I am a student in a cognitive science course designing a research tool called Knowledge Atlas (KA). KA is an AI-powered evidence-synthesis system for human-factors research on cognitive performance in workplace environments. It stores claims extracted from peer-reviewed articles, weights them by epistemic quality, and lets users query them using natural language. I have identified the following user types so far: [paste the six types from the assignment page]. Please suggest additional user types I may have missed, focusing on people who would have a plausible reason to interact with such a system — including non-research users. For each new type you suggest, provide: a one-sentence description of who they are, what primary goal brings them to KA, and two concrete scenarios (larger tasks they would be trying to accomplish). Think carefully about system architects, funders, regulators, science journalists, and members of the general public."
Phase 2 — User Scenarios & Use Cases

A user scenario is the larger task a person is trying to accomplish — a coherent episode of purposeful activity that may span multiple sessions. A use case is a specific subgoal or step within that scenario: a discrete thing the user needs to do that the system must support. For each user type you selected, define at least two scenarios and three use cases per scenario.

Worked example. User type: Undergraduate Student Researcher (fall course). Scenario S1: Choosing a research question — the student must identify a plausible CNFA topic, assess whether there is genuine uncertainty in the literature, and converge on a question their experiment could address within a quarter. Use cases within S1: (1) Browse the KA concept map to see which constructs are clustered near their intuitive topic; (2) Ask KA "what are the open questions in attention and lighting?"; (3) Check the Value of Information score to confirm the question is genuinely uncertain; (4) Save candidate questions to a personal draft list.
Phase 2 · AI Use-Case Expansion Prompt
🤖 Round 1 — Seed generation
"I am designing use cases for Knowledge Atlas, an AI evidence-synthesis system for human-factors research. My user type is [USER TYPE] and the scenario I am analysing is [SCENARIO NAME AND DESCRIPTION]. A use case has three parts: (1) a trigger — what situation or question makes this user want to interact with the system; (2) a query or action — exactly what they would type, click, or request; (3) a success criterion — what a satisfying system response looks like, and what would constitute a frustrating or misleading one. Please generate 10 use cases for this user type in this scenario. Do not list generic tasks — think about what this user genuinely cares about, what would surprise them, what would delight them, and what would make them abandon the system."
🤖 Round 2 — Self-reflection: where does KA fall short?
"Now I want you to evaluate your own output. Looking at the 10 use cases you generated, please assess each one: can the current version of Knowledge Atlas handle this use case well, partially, or not at all? For those it cannot handle well, what would need to change — in the data, the interface, the reasoning engine, or the API — to close the gap? Be specific. Think about what kinds of queries KA structurally cannot answer, and why."
Phase 3 — Site Walkthrough & Workbook

Pick one of the scenarios you defined in Phase 2. Walk through it on the live KA site, pretending to be the user type you chose. Before you start, ask an AI to suggest an optimal navigation path through the system for that scenario. Then try the path yourself. Record your observations, friction points, and questions below. You will shoot your key findings to the shared specification wall at the end.

Before you begin your walkthrough: Use the AI path-planning prompt below to generate a suggested strategy, then compare what the AI predicts with what you actually experience. Discrepancies are especially valuable findings.
Walkthrough Workbook
Phase 3 · AI Path-Planning Prompt
🤖 Ask the AI to plan your walkthrough path first
"I am about to walk through Knowledge Atlas pretending to be [USER TYPE]. My scenario is: [SCENARIO DESCRIPTION]. The KA site has the following pages: Home, Ask a Question (natural-language Q&A), Evidence Table, Gap Explorer, Experiment Maker (Hypothesis Builder), Sensor Selection, Dashboard, and Image Tagger. Given this scenario, what is the optimal path through the system? At each step, tell me: which page the user would visit, what action they would take, what they would hope to get from the system, and what could go wrong. Then tell me: which parts of this scenario do you think the current system handles well, and which are likely to feel incomplete or frustrating for the user, based on your understanding of the system's current capabilities?"
Phase 4 — Shared Specification Wall

This is the class knowledge base — the emerging usability specification for Knowledge Atlas built from every student's walkthrough findings. Each card below represents one student's scenario, use cases, and key friction observations. Your submissions from Phases 1–3 appear here once you shoot them to the wall. Read your classmates' entries and look for patterns: which user types keep appearing, which use cases cluster together, where does friction concentrate?

Stuck? Take a screenshot and ask AI.

How to get help from ChatGPT, Claude, or Gemini:

  2. Open ChatGPT, Claude, or Gemini
  3. Say: "I'm a COGS 160 student on Track 4 (GUI Evaluation). I see this screenshot. What should I do next to succeed at Phase {N}?"

This is the methodology — AI as a directed instrument, not a chatbot.

Do I need to write code?
No backend code required. You write user scenarios, evaluation reports, and experiment design specifications. Your contribution is observational and analytical, not technical.
How do I test pages without a backend?
Open the HTML files directly in Chrome. Most pages have sample data embedded. If you need fresh data, check the student database (data/student/web_persistence_v5_student_copy.db) for test fixtures.
What does a good user scenario look like?
Task (what the user wants) → optimal path (how they should do it) → actual path (what they really did) → friction points (where they got stuck) → severity (does it matter?). See the Phase 2 examples for templates.
Where do I submit my findings?
Commit to your branch, push, and open a Pull Request to track/4-staging. The instructor will review all findings and integrate them into the shared spec wall. See STEP 0 above for details.
