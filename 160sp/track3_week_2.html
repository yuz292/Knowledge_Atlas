Week 2

Question for the week: How do you translate research language into factors that can actually drive room edits?

Use AI well

Before you ask AI for help, make sure you can state what you are trying to learn this week and what evidence would count as a useful answer.

Class meetings

Meeting A — Turn latent variables into operational factors
Start here: Choose one latent variable from the table and ask what visible or editable room properties could plausibly influence it.

In class

  • Take two rows from the architectural factor table and break them into manipulable parameters, not just verbal paraphrases.
  • Ask AI for help distinguishing closely related ideas such as perceptual fluency, soft fascination, or refuge; then decide what would count as evidence for each in a room transformation workflow.
  • Draft a template for a factor sheet: name, target latent variables, parameters, allowed range, side effects, validation checks, and example transformations.
  • Exchange sheets with another team and see whether they are implementable or still too abstract.

Home

  • Complete factor sheets for the reduced core set.
  • Build a glossary of accepted vocabulary, synonyms, and ambiguous terms for each factor.
  • Note which terms sound plausible but are too vague to use as instructions without clarification.
Meeting B — Specify operators before you build them
Start here: Pick one transformation you think is easy and one you think is hard, then say why.

In class

  • Choose three priority transformations that the class must support early.
  • For each one, describe the input, the assumptions about the room, the scene elements it touches, what should remain unchanged, and what could fail.
  • Ask AI to suggest edge cases or failure modes you may have missed. Add only the ones that are concrete and testable.
  • Turn the discussion into an operator specification sheet rather than jumping straight into code or interface work.

Home

  • Revise the operator sheets after critique.
  • Make a dependency map showing which team needs assets, schemas, or tests from which other team.
  • Prepare a short comparison of which operators are likely to be easy, moderate, and hard in your chosen toolchain.

Week deliverables

D3 Factor ontology pack

Rationale
This pack is the bridge between research concepts and room edits. It gives the studio a common language for what can be changed and why.

What to submit
One factor sheet per core factor. Each sheet should include the factor name, target latent variables, manipulable parameters, acceptable ranges or states, side effects, validation checks, typical room conditions where it works well, and one example request in both research language and edit language.

What good work looks like
The sheets are operational. They define actual knobs and checks rather than repeating the literature in softer words. They also show where ambiguity remains and what extra information would be needed.

Example
A strong wall-curvature sheet names spline curvature, corner angle, alcove depth, and opening refit as relevant variables, and notes that preserving doors and windows is a validation problem rather than an afterthought.

D4 Operator specification sheets

Rationale
The specification sheet helps you decide what an operator owns before implementation begins. This reduces thrashing later.

What to submit
A one-page sheet for each priority operator with purpose, parameters, preconditions, touched scene elements, things that must remain stable, export expectations, validation logic, and known failure modes.

What good work looks like
Another team could build or test the operator from the sheet. The sheet predicts breakage rather than discovering it only by accident.

Example
A strong ceiling-height specification states whether the ceiling mesh moves, whether windows and lights move with it, how collision is updated, and what counts as a failed transformation.
