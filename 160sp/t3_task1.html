TRACK 3 · TASK 1

Collect & Catalog VR-Ready 3D Interior Models

Curate a library of at least 20 VR-ready 3D interior models, evaluate each against eight manipulation classes, segment the meshes so every wall is individually addressable, and produce a mesh_roles.json that the Task 3 AI front-end will consume.

75 points 4 phases No prerequisites

The big picture

The Knowledge Atlas catalogues evidence that specific environmental features alter human cognition and affect. To test these relationships in VR, a researcher needs 3D rooms where every factor — ceiling height, wall materials, lighting, window size, furniture density — can be varied independently while holding the others constant. Most 3D models on the open web cannot do this: they are single fused meshes, or they have baked lightmaps that cannot be swapped, or they are at arbitrary scale. Your job is to collect models that can.

What makes a model "VR-ready"

Five hard constraints separate models you can use from models you cannot. The first three are about how the model is structured; the last two are about file size and licence.

ConstraintWhat it meansHow to verify
Separable named meshes The ceiling is a separate mesh from the walls; each wall is individually identifiable. Open in Blender → check the Outliner. Named objects like Ceiling, Wall_North, Floor = viable. One object called mesh_001 = not viable.
PBR materials Materials use base-colour, roughness, metalness slots — not a single pre-baked lightmap. Look at the material panel. If you see Principled BSDF, you are good. If you see only a baked diffuse texture, you cannot swap materials.
Architectural scale Real-world units, in metres. Ceiling-height manipulation requires knowing the actual height. Measure the model in Blender. A 6 m × 5 m office should read as 6 and 5 in the dimensions panel, not 0.06 or 600.
VR-compatible polygon count Less than 500K polygons per scene; less than 20 unique materials; textures ≤ 4096×4096. Blender shows the polygon count in the top bar. Check texture sizes in the material panel.
Open licence CC0, CC-BY, or CC-BY-SA. No "editorial use only" or "All Rights Reserved." Read the licence on the source page before downloading. Record it in your catalog.

Advanced source: World Labs Marble

World Labs Marble generates explorable 3D worlds from a single photograph or text prompt. This is directly relevant to neuroarchitecture research — Champalimaud Foundation and King’s College London already use Marble to generate patient-specific VR environments for OCD exposure therapy. Use Marble for at least three of your twenty models so you demonstrate the photo → 3D → parametric pipeline.

The free plan gives four worlds, the Standard plan gives twelve, and the API plan is pay-per-use. Generation times: text or image → pano about 30 seconds, pano → full world about five minutes, high-quality mesh export about an hour.

Generate from a text prompt via the API:

curl -X POST 'https://api.worldlabs.ai/marble/v1/worlds:generate' \
  -H 'Content-Type: application/json' \
  -H 'WLT-Api-Key: YOUR_API_KEY' \
  -d '{
    "display_name": "Clinical Office Baseline",
    "model": "marble-1.1",
    "world_prompt": {
      "type": "text",
      "text_prompt": "A clinical white office with 3m ceilings, fluorescent lighting, one window on the south wall, a desk and two chairs"
    }
  }'

In the Marble web viewer, click Export → High-quality mesh (GLB). This runs server-side for about an hour and produces a 600K-triangle textured mesh plus a 1M-triangle vertex-colour mesh. The exported mesh is a single fused object — all walls, ceiling, and floor are one continuous surface. Phase 2 separates it into individual objects.

Deliverable for this phase: model_sources.json with at least five sources, three of them tested.

Phase 2: Segment Marble exports into named architectural elements

A fused mesh is not parametrically modifiable. You must separate it into ceiling, walls, floor, windows, and furniture, and assign PBR materials to each. This is a core part of the assignment — you are expected to use at least one AI-assisted approach, not do everything by hand. You must document which approach you used and evaluate its accuracy.

The four approaches

Approach Tooling API cost Viewport-aware Accuracy Best for

Recommended workflow: use Approach D (headless script) to batch-process all your models, then visually inspect every output in Blender and use Approach A or B to fix the errors. Document every correction — that is your ruthless validation data and earns the grade.

The headless segmentation script

Approach D classifies faces by their normal direction. Save this as segment_marble_export.py in your branch: 

import bpy, bmesh, sys, math

argv = sys.argv[sys.argv.index("--") + 1:]
input_path, output_path = argv[0], argv[1]

bpy.ops.wm.read_factory_settings(use_empty=True)
bpy.ops.import_scene.gltf(filepath=input_path)
obj = [o for o in bpy.context.scene.objects if o.type == 'MESH'][0]
bpy.context.view_layer.objects.active = obj

bpy.ops.object.mode_set(mode='EDIT')
bm = bmesh.from_edit_mesh(obj.data)

def classify_face(face):
    n = face.normal
    if n.z >  0.7: return 'Floor'
    if n.z < -0.7: return 'Ceiling'
    angle = math.atan2(n.y, n.x)
    if -0.785 < angle <= 0.785: return 'Wall_East'
    if  0.785 < angle <= 2.356: return 'Wall_North'
    if  angle >  2.356 or angle <= -2.356: return 'Wall_West'
    return 'Wall_South'

for role in ['Ceiling', 'Floor', 'Wall_North', 'Wall_South', 'Wall_East', 'Wall_West']:
    bpy.ops.mesh.select_all(action='DESELECT')
    bm = bmesh.from_edit_mesh(obj.data)
    for face in bm.faces:
        if classify_face(face) == role:
            face.select = True
    bmesh.update_edit_mesh(obj.data)
    bpy.ops.mesh.separate(type='SELECTED')
    new_obj = [o for o in bpy.context.scene.objects
               if o.name.startswith(obj.name) and o != obj][-1]
    new_obj.name = role

bpy.ops.object.mode_set(mode='OBJECT')

for obj in bpy.context.scene.objects:
    if obj.type != 'MESH': continue
    mat = bpy.data.materials.new(name=f"PBR_{obj.name}")
    mat.use_nodes = True
    bsdf = mat.node_tree.nodes["Principled BSDF"]
    bsdf.inputs['Roughness'].default_value = 0.7
    obj.data.materials.clear()
    obj.data.materials.append(mat)

bpy.ops.export_scene.gltf(filepath=output_path, export_format='GLB')

Run it on every Marble export:

for f in marble_exports/*.glb; do
  blender --background --python segment_marble_export.py -- "$f" "segmented/$(basename $f)"
done

The script fails on L-shaped rooms, angled walls, curved surfaces, and rooms where furniture faces look like wall faces. Visual inspection is non-negotiable — every output must be opened in Blender and manually checked. Misclassified faces get fixed with Approach A or B.

Phase 4: Create mesh-annotation files

For each model, create a mesh_roles.json that maps every mesh name to its semantic role. This is the file the Task 3 AI front-end will read to know which mesh to modify when the user says “change the north wall.” 

{
  "model_id": "SKF_office_001",
  "meshes": {
    "Cube.003":  { "role": "ceiling", "notes": "single plane, height=3.0m" },
    "Cube.004":  { "role": "wall", "wall_id": "north", "notes": "back wall" },
    "Cube.005":  { "role": "wall", "wall_id": "south", "notes": "front wall with window cutout" },
    "Cube.006":  { "role": "wall", "wall_id": "east" },
    "Cube.007":  { "role": "wall", "wall_id": "west" },
    "Plane.001": { "role": "floor" },
    "Cube.010":  { "role": "window", "parent_wall": "south", "position": "center" },
    "Cube.011":  { "role": "furniture", "subtype": "desk" },
    "Cube.012":  { "role": "furniture", "subtype": "chair" },
    "Lamp":      { "role": "light", "light_type": "point" }
  },
  "room_dimensions": { "width_m": 6.0, "depth_m": 5.0, "height_m": 3.0 }
}
Each wall must have a wall_id (north / south / east / west or left / right / back / front). This is what enables per-wall material and geometry changes in Tasks 2 and 3. A model without per-wall IDs cannot be used for the AI front-end.

Phase 5: Write a ruthless validation prompt

Before declaring a model "ready," test it with an adversarial AI prompt designed to break things. Instead of asking “does it work?”, you ask “in exactly which ways does it fail?”

You are a hostile QA engineer. Your job is to find every way this 3D model catalog entry could be wrong, misleading, or unusable. For model {model_id}:
  1. Open the model in Blender. Count the actual meshes and compare to the mesh_roles.json. Are any missing or mislabelled?
  2. Try to move the ceiling mesh up by 1 metre. Does the room still look correct? Do walls gap?
  3. Try to change the north wall material to a different colour. Does only that wall change, or do others change too (shared material)?
  4. Check every material — is it PBR or is it a baked texture that cannot be swapped?
  5. Measure the room in Blender. Does it match room_dimensions in the JSON?
  6. Is the model oriented correctly (Y-up or Z-up)? Does the "north" wall actually face north?
Report every discrepancy found.

Write a ruthless validation prompt for your catalog and run it on at least five of your models. Document what broke; document what you fixed.

Why this matters

Your model library and mesh annotations are infrastructure. In Task 2 you will manually test modifications in A-Frame. In Task 3 you will build the AI front-end that uses your mesh_roles.json files to know which mesh to modify. Accurate annotations now save hours of debugging later.
