Track 2 Setup — Article Finder

Track 2 works on one repository, the Article Finder (AF). AF generates Value-of-Information queries, searches Semantic Scholar and OpenAlex, downloads candidate PDFs, and hands them off to the downstream extraction pipeline. Track 2 students do not clone the extraction repo (Article Eater); that is a different track. Plan for about an hour — thirty minutes to fork, clone, and install, another thirty for the GitHub-workflow walk through at the bottom of this page.

Scope

One repository only. Everything on this page refers to kirsh-lab/Article_Finder. The sibling repository Article_Eater_PostQuinean_v1_recovery is consumed by a different track and should not be cloned for Track 2. The Finder writes a well-defined handoff artefact (data/handoff/*.json) that the Eater reads; the contract between them is the only thing Track 2 needs to honour.

Prerequisites

  • GitHub account with SSH keys configured. If you have never set up SSH, follow GitHub's official instructions at docs.github.com/en/authentication/connecting-to-github-with-ssh before Step 1. Verify with ssh -T git@github.com.
  • Python 3.11+ and Git 2.30+ — confirm with python3 --version and git --version.
  • Semantic Scholar API key — free; request at semanticscholar.org/product/api.
  • OpenAlex access — no key required; set OPENALEX_EMAIL in your .env so the polite pool accepts your requests.
  • 15 GB free disk for PDFs harvested during the quarter.

Part A — The GitHub Fork Workflow (do this first)

Everyone on Track 2 works from their own fork of the Article Finder repository, not from a direct clone. A fork is your personal server-side copy of the lab's repository: you can push freely to your fork, and when your work is ready you open a pull request back to the lab's copy, which we call upstream. This pattern is the standard open-source contribution model and it is what the rest of the class will evaluate your work through.

1Fork the repository on GitHub (once per person)
  1. Sign in to GitHub, then open github.com/kirsh-lab/Article_Finder.
  2. Click the Fork button in the top-right corner.
  3. Accept the defaults. GitHub creates a copy at github.com/<your-handle>/Article_Finder.
  4. Leave the "Copy the main branch only" box checked — it is faster and you will not need legacy branches.
Why fork instead of clone directly? You do not have write access to kirsh-lab/Article_Finder, which is deliberate: it prevents accidental pushes to main and it gives every student a private space to break things. All of your commits will live on your fork until you open a pull request.

Replace <your-handle> with your GitHub username. The clone creates a remote called origin that points at your fork.

3Add the lab repository as upstream
git remote add upstream git@github.com:kirsh-lab/Article_Finder.git
git remote -v
# You should see:
#   origin    git@github.com:<your-handle>/Article_Finder.git (fetch)
#   origin    git@github.com:<your-handle>/Article_Finder.git (push)
#   upstream  git@github.com:kirsh-lab/Article_Finder.git       (fetch)
#   upstream  git@github.com:kirsh-lab/Article_Finder.git       (push)

You will pull from upstream whenever the lab's main advances, and push only to origin. Never push directly to upstream; the server-side protection rules will reject it anyway.

4Disable accidental pushes to upstream
git remote set-url --push upstream no_push

This belt-and-braces step makes git push upstream fail at the local level so a mistyped command cannot reach the lab's copy.

Part B — Install and Run the Article Finder

5Create and activate a virtual environment
# macOS / Linux
python3 -m venv .venv
source .venv/bin/activate

# Windows (PowerShell)
py -3 -m venv .venv
.venv\Scripts\Activate.ps1
6Install dependencies
pip install --upgrade pip
pip install -r requirements.txt
7Configure your API credentials
cp .env.example .env
# Edit .env and fill in:
#   SEMANTIC_SCHOLAR_API_KEY=...
#   OPENALEX_EMAIL=you@ucsd.edu
Never commit .env. It is already in .gitignore; keep it that way. If you accidentally push a key, rotate it immediately on the provider's website and then remove it from git history with git filter-repo. Ask a TA if you have already pushed — it is recoverable, but do not keep committing on top of it.
8Initialise the local pipeline database
python scripts/init_pipeline_db.py

Creates data/pipeline_lifecycle.db (SQLite) with empty tables for candidates, downloads, and VOI scores.

9Run a sanity-check query
python -m article_finder.cli discover \
    --query "ceiling height creativity" \
    --max-results 10

You should see ten candidate articles with DOIs, abstracts, and VOI scores logged to the database. If the command fails with ModuleNotFoundError: article_finder, your virtual environment is not active — re-run Step 5 and try again.

Part C — Your Daily GitHub Workflow

Every unit of work — a new VOI heuristic, a fixed parser, a reworded prompt — follows the same six-step loop. Doing it the same way every time is how a class of thirty students can collaborate on one repository without stepping on each other.

10Sync your fork with upstream
git checkout main
git fetch upstream
git merge --ff-only upstream/main
git push origin main

Run this every time you start a work session. --ff-only refuses to merge if your local main has diverged from upstream/main; if it does diverge, ask a TA — it usually means something was committed to main directly by mistake, and the fix is not something to improvise.

11Create a feature branch
git checkout -b track2/<your-handle>/<short-topic>

For example: track2/dkirsh/voi-prior-calibration. The branch name is not cosmetic — our CI reads the track2/ prefix and runs the Track 2 test suite only. Branches that do not match the prefix get no CI coverage.

12Commit in small, readable increments
git add path/to/file.py
git commit -m "voi: widen prior on ceiling-height studies (fixes #123)"

One concern per commit; one sentence per commit message, in the imperative. The rest of the commit message is for the why, not the what — the diff already shows what. If you are changing more than one thing at a time, split it into two commits.

13Push your branch to origin (your fork)
git push -u origin track2/<your-handle>/<short-topic>

The -u flag sets the upstream tracking reference so future git push and git pull on this branch work without extra arguments.

14Open a Pull Request against upstream/main
  1. Visit your fork on GitHub. GitHub will display a yellow banner offering to open a PR against kirsh-lab/Article_Finder:main. Click Compare & pull request.
  2. Title: one sentence describing the change. Body: a short paragraph on motivation, a bullet list of what changed, and a Testing section showing the command you ran to verify the change.
  3. Request review from your track lead.
  4. Wait for CI (about three minutes). If CI fails, read the log, fix, push again — the PR updates automatically.
15After the PR merges, clean up
git checkout main
git fetch upstream
git merge --ff-only upstream/main
git push origin main
git branch -d track2/<your-handle>/<short-topic>
git push origin --delete track2/<your-handle>/<short-topic>

Your feature branch's job is done the moment the PR merges. Keeping merged branches around creates noise for the next contributor reviewing your fork.

A visual summary. upstream/main is the canonical history; your origin/main is a mirror of it; your feature branch sprouts from main, lives on origin, is reviewed as a PR, and is deleted on merge. Sync at the start of every session, branch before every change, one PR per concern, clean up after merge.

Google AI Scholar Manual Workflow

Google AI Scholar does not yet expose an API. Until it does, we capture its result pages by hand:

  1. Open Google AI Scholar, run your query, and scroll through the first two pages of results.
  2. Use the browser's Save as → Web Page, Complete to preserve the HTML locally.
  3. Drop the saved page into data/manual_captures/<date>/ and run python scripts/ingest_manual_capture.py — it parses titles, DOIs, and abstracts into the same schema the API crawlers use.

Common Snags

  • Permission denied (publickey) on git clone — your SSH key is not attached to your GitHub account. Run ssh -T git@github.com; if it fails, follow the SSH-key guide linked in Prerequisites.
  • fatal: refusing to merge unrelated histories — almost always means you cloned upstream directly instead of your fork. Delete the directory and start over from Step 2.
  • Updates were rejected because the remote contains work you do not have — someone (possibly your earlier self from another machine) pushed to the same branch. git pull --rebase origin <branch> and re-push.
  • HTTP 429 from Semantic Scholar — rate-limit; the CLI already backs off, but without an API key you are hard-capped. Complete Step 7.
  • Slow PDF downloads — publishers throttle anonymous GETs. Use the UCSD library VPN; the pipeline respects HTTPS_PROXY.
  • article_finder not a package — you ran the CLI from the wrong directory. cd into the repo root and re-activate the venv.
Never commit to main. The only commits that should appear on your local main are fast-forward merges from upstream/main. If you ever find yourself on main with uncommitted changes, stash them (git stash), branch (git checkout -b track2/…), and unstash (git stash pop).
