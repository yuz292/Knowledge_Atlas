TRACK 3 · TASK 3 · FLAGSHIP

AI Front-End, Literature-Grounded Presets, and the Curved-Wall Worship Room

Task 3 ships three deliverables. First, a thin LLM wrapper that translates a natural-language prompt plus the room’s parameter manifest from Task 1 into a validated JSON patch. Second, per assigned room type, a small library of named presets each citing a specific environment-psychology study so the room can be set to a published, replication-ready configuration. Third — the cohort flagship — a single curved-wall worship room published as a citable open-source parametric module, the only room of the fifteen that breaks Infinigen’s axis-aligned-box assumption, with every cohort member listed as co-author.

5 phases 75 points Weeks 7–10 Cohort flagship deliverable

The big picture

Tasks 1 and 2 produced parametric rooms callable through JSON. Task 3 makes them callable through natural language. A researcher who wants to inspect a high-ceiling Meyers-Levy office on a sleep-warm Münch bedroom should not have to remember which slider sets CCT or which key sets ceiling_height_m; she should type “raise the ceiling and warm the lighting” and watch the room update. The wrapper this task ships is small — a thin LLM call, a JSON-patch grammar, and a validation gate — but it is the layer that makes the parametric library usable by humans rather than only by other code.

The presets close a different gap. A parametric room is only replicable if its named configurations come from somewhere defensible. A “cozy living room” preset that the author invented is decoration; a kaplan_restorative preset whose values reproduce the natural-light condition in Kaplan & Kaplan (1989) is a citable experimental stimulus. Every student authors at least four such presets across her two assigned room types. Each preset ships as a JSON parameter patch plus a citation block (full APA + DOI) explaining which published study’s condition it reproduces and what hypothesis it represents.

The flagship is the curved-wall worship room. Of the fifteen room types in this cohort’s library, fourteen fit comfortably inside Infinigen’s axis-aligned-box assumption: floor plans are rectangles, ceilings are flat, walls are perpendicular. Worship spaces — cathedrals, mosques, chapels — are the exception. They use Bezier curves for floor plans, vault profiles for ceilings, and fractal-rich surface ornament. The published environmental- psychology effect is also the strongest in the library: Vartanian et al. (2015) report large reliable affective responses to high-vault contour-rich interiors; Joye (2007) argues a biophilic-fractal route to architectural awe; Bermudez et al. (2017) show contemplative-architecture-specific activation in fMRI. Stiny (1980) provides the formal apparatus — shape grammars — that justifies treating Bezier-based parametric room generation as an in-principle tractable computational problem rather than a one-off art project. The cohort closes a gap the parametric-models literature explicitly identifies (Raistrick et al., 2024; Khanna et al., 2024) by shipping one shared, citable, open-source parametric module at Knowledge_Atlas/3d_rooms/worship_curved/ with every cohort member as co-author.

Real-cathedral lighting via splat augmentation (Sprint 4 addendum). The Vartanian et al. (2015) cathedral-effect preset is materially stronger when the parametric Bezier walls are rendered under HDRI lighting captured from an actual cathedral interior. The cohort flagship integration step should call scripts/track3/splat_to_hdri.py on a Polycam capture of a real worship space (e.g., Notre-Dame de Paris, Hagia Sophia replica scans, or any local cathedral / temple / mosque your team can reach with a phone). Apply the resulting .hdr environment to the integration render. The parametric Bezier curvature stays under cohort control; the lighting stays photoreal. This is the cleanest available reproduction of the Vartanian condition the open-source parametric-rooms literature can currently mount.

The five phases below carry these three deliverables. Phase 1 is the LLM wrapper and parameter-patch grammar. Phase 2 is the validation gate. Phase 3 is the literature-grounded presets. Phase 4 is the curved-wall worship room flagship. Phase 5 is the cross-room evaluation that audits the wrapper against the entire cohort library.

Phase 1: LLM wrapper + parameter-patch grammar

Contract objective: “I want a Python (or JavaScript) wrapper that takes (a) the user’s natural-language prompt, (b) the room’s parameter manifest from Task 1, and emits a JSON patch that Task 2’s wrapper can apply.”

The patch is a list of operations. Three operation types cover everything Task 3 needs: set assigns a parameter to a value; delta adds to a parameter’s current value; preset applies a named preset by name. Each operation carries a target parameter, a value, and (for set) an optional unit string the validator will check against the manifest.

Patch grammar (JSON Schema)

{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "Room Parameter Patch",
  "type": "object",
  "required": ["room_id", "operations"],
  "properties": {
    "room_id":   { "type": "string" },
    "manifest_version": { "type": "string" },
    "operations": {
      "type": "array",
      "minItems": 1,
      "items": {
        "oneOf": [
          {
            "type": "object",
            "required": ["op", "param", "value"],
            "properties": {
              "op":    { "const": "set" },
              "param": { "type": "string" },
              "value": {},
              "unit":  { "type": "string" }
            }
          },
          {
            "type": "object",
            "required": ["op", "param", "delta"],
            "properties": {
              "op":    { "const": "delta" },
              "param": { "type": "string" },
              "delta": { "type": "number" },
              "unit":  { "type": "string" }
            }
          },
          {
            "type": "object",
            "required": ["op", "preset"],
            "properties": {
              "op":     { "const": "preset" },
              "preset": { "type": "string" }
            }
          }
        ]
      }
    }
  }
}

Parameter validation is delegated to the room’s manifest: for every operation the wrapper consults the manifest’s per-parameter schema (declared type, declared range, declared unit) and only emits the operation if the value or delta would stay inside the documented bounds.

System prompt

System prompt for the LLM

You are a parameter editor for a 3D room. The user describes a change in plain language. Your job is to read the room’s parameter manifest and emit a JSON patch consisting of one or more operations from {set, delta, preset}. Do NOT invent parameters not in the manifest. Do NOT exceed the documented ranges. If the user’s request cannot be expressed in the manifest, emit a structured error with code: "out_of_schema" and a one-sentence explanation.

Worked example

User prompt: “warm up the lighting and raise the ceiling 30 cm.” Manifest excerpt: light_cct_k (range 2700–6500), ceiling_height_m (range 2.0–6.0, current 2.7). Expected patch: 

{
  "room_id": "office_seed_004",
  "manifest_version": "1.0.2",
  "operations": [
    { "op": "set",   "param": "light_cct_k",       "value": 3000, "unit": "K" },
    { "op": "delta", "param": "ceiling_height_m",  "delta": 0.30, "unit": "m" }
  ]
}

Note the wrapper translated “warm” to a CCT value inside the manifest’s range, and translated “raise 30 cm” to a delta rather than a set — preserving whatever ceiling height the user had previously dialled in.

Deliverable

A working CLI demo (nl_patch_demo.py or nl_patch_demo.js) that takes a natural-language prompt and a manifest path, calls the LLM, prints the JSON patch, and pipes the patch through Task 2’s wrapper to render the modified room.

Phase 2: Validation gate

A validation gate is mandatory because LLMs hallucinate parameter names, exceed declared ranges, and emit malformed JSON. The gate catches all three classes of failure before the patch reaches the renderer. Without it, an LLM that confidently writes ceiling_height_m: 47.0 will crash the geometry generator; an LLM that targets ceiling_h_meters will silently no-op; an LLM that omits a closing brace will throw deep inside the parser.

Specification

  • Range check. Every value falls within [min, max] as declared in the manifest.
  • Type check. Every value matches the schema’s declared type (number, integer, string, boolean, hex colour).
  • Name check. Every targeted parameter exists in the manifest at the declared manifest_version.
  • Refusal mode. If the LLM emits an operation the manifest cannot express — for example, targeting wall_curvature on a rectangular room — the wrapper refuses with a structured error ({code: "out_of_schema", param: …, reason: …}) rather than guessing a substitute.

Building the validator

Ask your AI

“Write me a Python validator that takes (a) a JSON parameter patch and (b) a JSON-Schema manifest, and returns either OK or a structured list of violations. Use jsonschema for the schema check and add custom range/name checks on top.”

Return shape: {ok: bool, violations: [{op_index, code, param, message}, …]}. Codes: schema_invalid, unknown_param, type_mismatch, out_of_range, out_of_schema.

Deliverable

validation_gate.py plus a test suite of at least eight cases — five valid (single set, single delta, single preset, multi-op patch, edge-of-range value) and three invalid (unknown param, value above max, type mismatch).

Phase 3: Literature-grounded presets

For each of your two assigned room types, define 2–3 named presets. Each preset lives at presets/<room_type>/<preset_name>.json; its content is a valid parameter patch plus a citation block (full APA reference + DOI) explaining which study the preset reproduces and what hypothesis it represents. Be conservative: when a study does not specify a value for one of your manifest’s parameters, leave it at default rather than guess.

Phase 4: THE FLAGSHIP — Curved-Wall Worship Room

This is the cohort’s flagship deliverable. Of the fifteen room types in the library, the curved-wall worship room is the only one that breaks Infinigen’s axis-aligned-box assumption; it is also the room type with the strongest published environmental-psychology effect (Vartanian et al., 2015; Bermudez et al., 2017). A clean implementation is genuinely citable. The cohort ships one shared parametric module at Knowledge_Atlas/3d_rooms/worship_curved/, with every cohort member listed as co-author. Each student owns one sub-component; the components compose into the published module.

Authoring rationale

Ask your AI

“I’m contributing the [sub-component] to a cohort-published parametric curved-wall worship room. Read [Vartanian 2015 / Joye 2007 / Bermudez 2017] and tell me: (a) which geometric or parametric features in those studies most directly motivate my sub-component, (b) what range of values is defensible based on the data in those studies, (c) what cross-cultural variance the studies acknowledge, (d) what the construct-validity risks are if my sub-component is implemented carelessly. Be specific about study findings, not generic about architecture.”

Deliverable per student

One sub-component module + unit tests + a 500-word rationale citing the literature + a contribution commit to the integration PR. The integration lead additionally owns the term-end cohort PR.

Phase 5: Cross-room evaluation

Take the full library of parametric rooms — everyone’s manifests from Task 1, everyone’s pipelines from Task 2, everyone’s presets from Task 3 — and evaluate the AI front-end against at least 10 prompts per room. For each prompt, record:

  1. What the user asked (the verbatim natural-language prompt).
  2. What JSON patch the LLM emitted.
  3. Whether the validation gate passed, degraded, or refused.
  4. Whether the rendered result matches the user’s intent (a 0–5 subjective rating, with one-sentence rationale).

Aggregate the findings into a final report that answers four questions: which room types does the AI handle reliably; which parameters are systematically misinterpreted; where does the validation gate fire most often; and what improvements would have the largest yield?

Deliverable

evaluation_report.md — 1500–2000 words, with per-room tables, a summary of the four findings, and a prioritised list of improvements.

The cohort flagship PR (integration lead only)

At term end, the integration lead opens a separate cohort-wide PR titled Track 3 Flagship: Curved-Wall Worship Room — Cohort SP 2026 from flagship/worship_curved into main. Do not commit API keys. The LLM wrapper reads its key from an environment variable. Repository secrets and .env files are never committed. A single leaked key has cost prior cohorts real money; the gate that catches this is your own discipline.
