TRACK 3 · TASK 2

VR Conversion & Manual Factor Testing

Load at least ten of your models in a browser-based A-Frame viewer, build real-time slider controls for all eight manipulation classes, and produce a factor viability matrix supported by before/after screenshots.

75 points 3 phases Prerequisite: Task 1 120+ fps target

The big picture

In Task 1 you cataloged twenty-plus 3D models and annotated every mesh. Now you must prove those annotations are correct by actually loading the models in a browser and manually testing every manipulation. This is not a demo. It is validation engineering — and the matrix you produce is the input to Task 3’s validation gate.

You already know HTML and JavaScript from Tracks 1 and 2. A-Frame wraps Three.js, a glTF loads with one HTML tag, and modifications are JavaScript property changes that execute at 120 fps in our prototype.

Per-wall selection is non-negotiable. The user must be able to click "North Wall" and change only that wall’s material. This is what your mesh_roles.json wall_id annotations from Task 1 enable. A viewer without per-wall selection cannot be used by the AI front-end.

How wall selection works

// mesh_roles.json tells us which mesh is which wall
const roles = await fetch('mesh_roles/SKF_office_001.json').then(r => r.json());

const walls = Object.entries(roles.meshes)
  .filter(([name, info]) => info.role === 'wall')
  .map(([name, info]) => ({ meshName: name, wallId: info.wall_id }));

// When user selects "north" and clicks a material swatch:
function setWallMaterial(wallId, color, roughness) {
  const target = wallId === 'all' ? walls : walls.filter(w => w.wallId === wallId);
  target.forEach(w => {
    const mesh = scene.getObjectByName(w.meshName);
    mesh.material.color.set(color);
    mesh.material.roughness = roughness;
  });
}

How ceiling height works (the geometry challenge)

function setCeilingHeight(newHeight) {
  const ceiling = scene.getObjectByName(ceilingMeshName);
  const originalHeight = roles.room_dimensions.height_m;
  ceiling.position.y = newHeight;

  walls.forEach(w => {
    const mesh = scene.getObjectByName(w.meshName);
    mesh.scale.y = newHeight / originalHeight;
    mesh.position.y = newHeight / 2;
  });

  lights.forEach(l => { l.position.y = newHeight - 0.3; });
}

These are JavaScript property changes — no re-rendering, no re-compilation. Slider dragging is perceptually instant and measured at 120 fps in our prototype.

Lighting quality requirement

Basic Three.js lights (DirectionalLight + AmbientLight) produce flat, unrealistic scenes with no light bounce, no colour bleeding, no ambient occlusion. For research stimuli, this is unacceptable — 33 of the registry lighting tags describe properties of indirect light. Your viewer must implement at least one of these lighting upgrades:

Level Technique What it adds Implementation
Level 2 (minimum) HDRI environment map + tone mapping Realistic ambient from all directions; reflections on glossy surfaces Load a .hdr from Poly Haven, set scene.environment, use ACESFilmicToneMapping. ~20 lines.
Level 3 (expected) + Screen-space ambient occlusion (SSAO) Corners and crevices darken naturally; large realism boost Add SSAOPass from Three.js post-processing. ~40 lines.
Level 5 (stretch) Path tracing via three-gpu-pathtracer Full global illumination: light bounces between surfaces, colour bleeds, soft shadows, caustics Replace renderer.render() with pathTracer.renderSample(). ~20 lines. See Interior Scene demo.

The three-gpu-pathtracer Interior Scene demo runs on standard laptop GPUs (MacBook Air M1 and up). It uses WebGL 2 (not WebGPU), renders at half resolution during interaction, and reaches a visually clean 38 samples in about 30 seconds at rest. At 370+ samples the result is publication-quality. The library is actively maintained, MIT-licensed, 1.7k GitHub stars.

The interactive UX for path tracing:

SLIDER DRAG:     Scene goes noisy (like a Blender viewport) — still responsive
SLIDER RELEASE:  pathTracer.updateMaterials() + pathTracer.reset()
                 Image converges to clean GI in ~2 seconds
EXPORT:          Capture converged frame as the research stimulus

HDRI sources (all CC0, free)

  • Poly Haven HDRIs — filter "indoor". Download ≥ 3 at different colour temps (warm interior, neutral studio, cool overcast).
  • Ambient CG — CC0 HDRIs and matching PBR material textures.

Testing protocol for each factor

  1. Set the slider to its lowest value (e.g. ceiling = 2.0 m)
  2. Take a screenshot
  3. Set to the highest value (e.g. ceiling = 6.0 m)
  4. Take a screenshot
  5. Record: did it work? what broke? how long did the modification take?

Phase 3: Write ruthless validation tests

For each model you tested, write a ruthless prompt that an AI cannot pass unless the viewer actually works. Run this on every model. The results go into the factor viability matrix.

Load model SKF_office_001 in the viewer. Execute these 12 tests in sequence:
  1. Set ceiling to 2.0 m. Is the ceiling visually at desk height? If not, the scale is wrong.
  2. Set ceiling to 5.0 m. Do the walls extend to meet it, or is there a gap?
  3. Select only the north wall. Change it to brick. Verify: south, east, and west walls are unchanged.
  4. Set colour temperature to 2700 K. Does the room look warm? Set to 6500 K. Does it look clinical?
  5. Set window scale to 3.0×. Does the window mesh overlap adjacent walls?
  6. Set window scale to 0.2×. Is the window still visible or does it disappear?
  7. Change floor to marble (roughness 0.15). Does it look reflective?
  8. Hide all furniture meshes. Is only architecture remaining?
  9. Check the FPS counter during slider drag. Is it ≥ 60 fps throughout?
  10. Set all sliders to extreme values simultaneously. Does anything z-fight or overlap?
  11. Reset to defaults. Does the model return to its original state exactly?
  12. Open the browser console. Are there any JavaScript errors?
Report pass / fail for each test.
