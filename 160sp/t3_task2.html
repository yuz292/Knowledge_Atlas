TRACK 3 · TASK 2

Render-to-Web Pipeline and Parameter-Sweep Gallery

A Python wrapper turns the Task 1 JSON manifest into a glTF file; a browser viewer renders it; a static gallery sweeps each parameter at low/mid/high; an HSSD divergence note documents what your parametric room captures and what an artist-authored room captures that yours cannot.

4 phases 75 points Weeks 5–7

The big picture

A parametric room is only useful for research if a researcher can see the result of varying a parameter. In Task 1 you defined the manifest — the dictionary of knobs the generator exposes. Task 2 closes the loop. JSON in, glTF out, render in the browser, in seconds. Once that loop is fast, a sweep is just a script and a grid of thumbnails.

Splat-augmented gallery (Sprint 4 addendum). If you completed Task 1 Phase 2.5 you have an HDRI at 3d_rooms/<your_room_type>/lighting.hdr and at least two PBR material packs in 3d_rooms/_materials_library/. Pass the HDRI to the parametric renderer (Infinigen accepts HDRI environment maps directly via the environment_image argument; see scripts/track3/infinigen_wrapper.py) and apply your material packs to the corresponding parametric surfaces. Render your gallery twice — once with synthetic lighting and stock materials, once with the splat-derived HDRI and your extracted materials — and include both in the deliverable. The visible jump in realism without losing parameter editability is the point of the splat-augmentation path (Kerbl et al., 2023).

The closing deliverable is not a polish exercise. It is an HSSD divergence note. The Habitat Synthetic Scenes Dataset (HSSD-200; Khanna et al., 2024) gives us 211 artist-authored full houses. The same paper reports a striking finding: 122 well-modelled HSSD scenes outperformed ten thousand procedurally generated scenes on real-world transfer. That result is itself a teaching moment. A researcher who uses your tool needs to know what a parametric room buys them (controlled sweeps, factorial designs) and what it costs them (idiosyncratic detail, lived-in clutter, asymmetric layouts).

So Task 2 has two faces. One face is engineering — a working pipeline from JSON to a rendered scene in a browser tab. The other face is epistemic — an honest written comparison between what your parametric room captures and what an artist-authored room captures, for the same room type. The grade rewards both.

Phase 1: Build the Python wrapper

Contract objective: “I want a Python script that takes a JSON parameter dict matching my Task 1 manifest, calls Infinigen Indoors, exports the result as a glTF file, and writes a sidecar JSON with the resolved parameters and render time.”
Contract is with: the Task 1 manifest (manifests/<room>.schema.json), the Infinigen Indoors Python API, and the Blender 4.x glTF exporter.
The wrapper is shipped, not assigned. You do not author infinigen_wrapper.py from scratch. The course ships a working version at Knowledge_Atlas/scripts/track3/infinigen_wrapper.py; you extend its ROOM_BUILDERS dict with your assigned room types and customise per-room parameter mapping. Same for scripts/track3/validation_gate.py (Task 3) and 3d_rooms/_scaffold/viewer_starter.html (the model-viewer starter you adapt for the parameter-sweep gallery).

Wrapper signature

The script is invoked from the command line, not from a notebook:

python3 scripts/track3/infinigen_wrapper.py \
  --manifest manifests/living_room.schema.json \
  --params   params/living_room_default.json \
  --out      renders/living_room_default.gltf

It produces two files alongside the glTF:

  • renders/living_room_default.gltf — the scene with embedded textures.
  • renders/living_room_default.meta.json — the resolved parameters (after defaults and clamping), the manifest version, the Infinigen commit hash, and the wall-clock render time.

Infinigen → glTF: the export step

Infinigen Indoors ships with an Omniverse export path (Raistrick et al., 2024). glTF is one Blender-headless step beyond that: bake the scene, then call Blender's glTF exporter from a script. The minimum command, run from your repo root: 

blender --background --python scripts/track3/infinigen_wrapper.py -- \
  --manifest manifests/living_room.schema.json \
  --params   params/living_room_default.json \
  --out      renders/living_room_default.gltf

The double-dash (--) tells Blender to pass everything after it to your Python script rather than to Blender itself. Inside the script, use Blender's Python API to build the Infinigen scene and then call bpy.ops.export_scene.gltf(...) with export_format='GLTF_EMBEDDED' so the textures travel with the geometry in a single file.

“I have an Infinigen LivingRoom instance. Show me the exact Python code to (a) instantiate it with parameter values from a JSON file, (b) bake the scene into Blender, (c) export to a single glTF file with embedded textures. Use Blender 4.x's bpy.ops.export_scene.gltf API. Include error handling for parameters that fall outside the manifest ranges.”

Deliverable

Working scripts/track3/infinigen_wrapper.py plus three test renders per assigned room (default, perturbed-1, perturbed-2), each with its .meta.json sidecar. Six glTFs total across your two assigned rooms.

Phase 2: Build the browser viewer

Use Google's open-source <model-viewer> web component. It is free, CDN-hosted, no install, and ships with realistic image-based lighting (IBL) and orbit controls out of the box. We do not reach for Three.js here because this task does not need custom shaders, post-processing, or path tracing — we need a clean turntable view of a single glTF, in eight lines of HTML.

Minimum viable viewer

<!DOCTYPE html>
<html>
<head>
  <script type="module"
    src="https://ajax.googleapis.com/ajax/libs/model-viewer/3.5.0/model-viewer.min.js">
  </script>
</head>
<body>
  <model-viewer
    src="renders/living_room_default.gltf"
    alt="Procedurally generated living room"
    camera-controls
    auto-rotate
    shadow-intensity="1"
    exposure="1"
    style="width: 100%; height: 600px;">
  </model-viewer>
</body>
</html>

That is the entire baseline. Drop in a glTF, open the page, drag to orbit. Now extend it.

Ask your AI

“Modify this <model-viewer> viewer so that (a) it loads a different GLB based on a query-string parameter (e.g. viewer.html?scene=perturbed-1), (b) it shows the parameter values from the sidecar JSON (.meta.json) in a small overlay panel in the top-right corner, (c) it switches between three pre-rendered parameter settings via three buttons labelled default, perturbed-1, perturbed-2. Keep the page to one HTML file with inline CSS and JS.”

Deliverable

A working viewer.html per assigned room, committed under track3/<your-name>/viewer/. Each viewer must (a) load the three test renders from Phase 1, (b) display the resolved parameters from the sidecar, (c) orbit-control with mouse and touch.

Phase 3: Generate the parameter-sweep gallery

For each of your two assigned rooms, generate one render at three settings of each exposed parameter (low / mid / high), holding all other parameters at their defaults. With six parameters per manifest × three settings each, that is 18 renders per room — 36 renders total across your two rooms.

The gallery is a single static HTML page that lays the 18 renders out as a grid. Each tile is a thumbnail (use the glTF rendered to a PNG, or embed a small <model-viewer> per tile if your machine can run them) with the parameter name and the parameter value labelled underneath.

Grid specification

RowsOne row per parameter (six rows)
ColumnsThree columns: low, mid, high
Tile contentsRender thumbnail; parameter name; parameter value
Tile size320 × 240 px (or larger if your screen allows)
HeaderRoom type, date generated, link to the manifest used
FooterTotal render time, Infinigen commit, your name
Ask your AI

“I have a parameter manifest with 6 parameters and a Python wrapper that turns a JSON dict into a glTF. Write me a build script that emits 18 renders for one room (each parameter at low/mid/high, all other parameters at default) plus a static HTML gallery page that displays all 18 with their parameter labels. The script should be idempotent — re-running it should skip any render whose .meta.json sidecar is already up to date.”

Hold all other parameters constant. The point of a sweep is to isolate the effect of one knob. If you accidentally vary two parameters between adjacent tiles, the sweep no longer reads as a sweep. The build script must drive this discipline mechanically — do not hand-edit the JSON dicts.
Deliverable

One gallery.html per assigned room (two total) plus 36 renders + sidecars committed under track3/<your-name>/gallery/.

Phase 4: HSSD divergence note

What HSSD-200 is

The Habitat Synthetic Scenes Dataset (HSSD-200; Khanna et al., 2024) is an artist-authored dataset of 211 full houses with 18,656 individual object instances, distributed under CC-BY-NC. The 2024 paper makes one finding worth memorising: agents trained on just 122 well-modelled HSSD scenes outperformed agents trained on ten thousand procedurally generated scenes on a real-world transfer task. Quality of authoring beat quantity of generation, by a wide margin.

HSSD is the right comparison object for your parametric room. Same room types. Different generation philosophy. Different strengths.

What you write

For each of your two assigned rooms, find the closest HSSD scene by room type. Open both the HSSD scene and your parametric default render side-by-side in a browser. Then write a 600–1000 word divergence note that covers the following four questions, in order:

SectionWhat to address
(a) Shared coverage What does your parametric room capture that the HSSD scene also captures? Architectural geometry, basic materials, plausible lighting, recognisable furniture categories.
(b) Parametric-only What does your room capture that HSSD does not? The sweeps themselves. Your room exists in 18 versions; the HSSD scene exists in one. Factorial designs, between-condition comparisons, controlled stimulus variation — these are parametric-only affordances.
(c) HSSD-only What does HSSD capture that your room does not? Idiosyncratic detail, asymmetric layouts, lived-in clutter, wear and tear, the small inconsistencies that make a real room read as real. Be specific — name three concrete items in the HSSD scene that your generator could not produce.
(d) Implications For a researcher choosing stimuli: which kind of study wants which kind of room? Sweeps and dose-response designs prefer parametric. Naturalistic scene perception, ecological validity, and real-world transfer prefer HSSD. State your recommendation as a rule a colleague could follow.
This is an epistemic deliverable, not a polish exercise. The point is not to make your parametric room look as good as HSSD. It will not. The point is to write honestly about the divergence so the next researcher who picks up your tool knows what they are buying and what they are giving up. Khanna et al.'s 122-vs-10,000 finding is the background result; your divergence note is the local instance of it.
Deliverable

One divergence_note.md per assigned room (two total), 600–1000 words each, committed under track3/<your-name>/divergence/. Include screenshots of both the HSSD scene and your parametric default in a screenshots/ subfolder, referenced by relative path from the note.
