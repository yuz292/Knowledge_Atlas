Region 1
Environment setup

Track 3 has one official student baseline: Blender for inspecting and cleaning assets, plus a browser-first A-Frame workflow for final course scenes. Unity and Unreal remain prototype environments only.

Software

  • Blender 4.x for room inspection, scale correction, and cleanup
  • Python 3.11+ and Git for local work
  • Chrome or Firefox for browser scene review and debugging
  • Optional: Meshy or Tripo for isolated props, not for the whole benchmark corpus

First-run checklist

  1. Install Blender and confirm it opens.
  2. Clone Knowledge_Atlas and open the Track 3 setup guide.
  3. Download at least three candidate room models in .glb, .fbx, or .obj.
  4. Inspect one room in Blender for scale, hierarchy, openings, and materials.
  5. Review the workbook so you understand that the final submission path is A-Frame.

Region 3
Ways of working

Ten students is a lot for one studio. Expect pair or triple sub-teams with clear handoffs.

  • Sub-teams of 2–3; each owns one asset, one operator, one scene, or one documentation slice.
  • Every admitted asset must carry a source note, format note, and licence note.
  • Normalise rooms to portable formats such as .glb whenever possible.
  • No scene or asset is treated as “working” until another student can open it and reproduce the basic check.
  • Browser review is the default; headset testing is optional, not the first gate.
Region 4
Task context & assignment manifests

Track 3 work now breaks into four explicit deliverable lanes. They are ordered roughly the way the course expects them to mature.

T3-001 · Benchmark room evaluation pack Well-specified
Goal
Build a shared shortlist of room assets that are worth transforming later.
Outputs
Room sheets with source, format, scale sanity, hierarchy notes, supported transformations, and risks.
Acceptance
Another student can read the sheet and understand why the room belongs in or out of the corpus.
Verification
Manual inspection in Blender plus a short written justification, not just a screenshot.
T3-002 · Controlled transformation variant Well-specified
Goal
Take one base room and create one variant that changes a single architectural factor cleanly.
Outputs
Base room, transformed variant, and a short change log stating what changed and what was held constant.
Acceptance
The intended factor shift is visible and the room remains structurally usable.
Verification
Peer review plus a checklist for scale, openings, hierarchy, and material continuity.
T3-003 · Instrumented A-Frame scene Well-specified
Goal
Turn the selected paradigm into one browser-runnable scene with explicit event hooks and condition labels.
Outputs
vr/scene.html plus vr/LOG_SCHEMA.md.
Acceptance
The scene loads in browser without console errors and the logging plan is explicit enough for another person to inspect.
Verification
Local server run, browser console check, and peer walkthrough on desktop or headset.
T3-004 · Researcher handoff pack Partially specified
Goal
Package the work so a new user can understand the room choice, transformation logic, and experimental purpose.
Outputs
Quick-start instructions, variant notes, limitations note, and one end-to-end example request.
Open question
How much of this should be a web interface versus a tightly structured document. That will be resolved in the final workflow stage.
Acceptance
Another student can follow the package without verbal rescue.
Region 5
Functional verification

Track 3 deliverables are graded by does the asset or scene remain usable and intelligible, not by visual polish. Use three checks:

  1. Asset sanity check in Blender — scale, hierarchy, openings, materials, and obvious breakage.
  2. Browser scene check — the scene loads from a local server, assets resolve, and there are no fatal console errors.
  3. Peer walkthrough — another student can interpret the condition and explain what the scene is supposed to test.

A scene or asset that fails the first two checks is not “done,” regardless of how promising it looks.

Region 6
Help & queue
Where to ask for help
  • Blender import or scale problems → bring the asset and its source link.
  • Browser or A-Frame load failure → bring the exact console error and the asset path.
  • Asset licensing uncertainty → add the source URL to the resource sheet before reuse.
  • Experimental-design ambiguity → escalate to the workbook and Friday demo, not just to a code fix.
