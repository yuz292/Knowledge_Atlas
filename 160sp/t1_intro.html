TRACK 1 — IMAGE TAGGER

Track 1: Image Tagger

Build the visual annotation pipeline for the Knowledge Atlas — collect 500 images, write detectors for 6 latent tags, run them at scale, and ship a public viewer.

3 tasks 10 weeks 75 + 75 + 75 points

What you’ll build

The Knowledge Atlas studies how buildings shape the people inside them — how ceiling height changes thinking, how daylight shifts mood, how clutter raises stress. To study those effects computationally you need photographs, and you need each photograph to carry both a licence and a measurement.

You will start with nothing and finish with a working pipeline. In Task 1 you gather 500 openly-licensed interior photos across fifteen room types, with full provenance for every file. In Task 2 you write six detectors that read those images and output latent tags — numerical scores for social-spatial features like sociopetal seating, prospect, and interactional visibility. In Task 3 you run the detectors over the whole collection and ship a standalone browser that lets anyone search the images by tag or by human effect. The collection, the detectors, and the viewer all become long-lived infrastructure for the Atlas.

The three tasks

TASK 1
Build an Image Collection
Find five openly-licensed image sources, build a search pipeline, and gather 500 architectural-interior photographs with full provenance — source URL, photographer, licence, and room type for every file.
4 phases · 75 points
TASK 2
Build Latent Tag Detectors
Pick six latent tags from the Tagging Contractor registry and implement detectors for them. Each detector ships with a design note, a typed contract that matches the registry, and a pytest suite that passes the semantic-audit gate.
7-step scaffold · 75 points
TASK 3
Annotate Collection & Build Image Viewer
Run your six detectors across all 500 images, write the results to a schema-conformant annotations.json, and build a single-file HTML viewer that lets users browse the collection by tag domain or by human effect.
3 phases · 75 points

Repos you’ll need

Track 1 spans four repositories. Fork the ones you will commit to; read the others.

Knowledge_Atlas
The Atlas itself — this is where your collection page, annotations, and image viewer ship to students and researchers.
Tagging_Contractor
The latent-tag registry: 424 tags across 44 domains, each with a definition_long, method_family, and extractability. The source of truth your detectors must match.
Article_Eater
Evidence extraction from the published literature, including the PNU templates that link environmental tags to outcomes — useful for Task 3’s effect-mode browser.
atlas_shared
Shared schemas and contracts used across Atlas repos. Reference when you need to produce JSON that other Atlas tools will consume.

How submission works for Track 1

Track 1 uses a single git branch per student — track1/your-name — across all 3 tasks. Each completed task is submitted as its own Pull Request to the instructor. The full mechanics are on each task page; here’s the lifecycle in one paragraph:

  1. Once at the start of the track: fork the track’s repos, clone, set the upstream remote, create track1/your-name off upstream/main.
  2. Continuously while working: commit small, push at the end of each session.
  3. At the end of each task: open a Pull Request from your branch to dkirsh:main, title it Track 1 · Task N: ... — your name, paste your file manifest, tag @dkirsh, do not merge.
  4. Across tasks: keep using the same branch; additional task PRs are opened against the previous task’s merge commit.

The detailed commands and the per-task PR title format are on each task page under “Submit Task N on GitHub.”
