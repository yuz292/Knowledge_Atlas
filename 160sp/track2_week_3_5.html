Why this week matters

Teams often lose two weeks by talking as if the system already exists. In Track 2 that failure is especially easy, because AF has a lot of real components and a lot of surrounding prose. Week 3.5 is therefore a discipline week: identify the real commands, the real boundary, the real tests, and the real missing pieces.

Use the repo as ground truth

The current AF repo exposes real entry points such as python cli/main.py doctor, stats, classify, discover, and build-jobs. Work from those. Do not inherit imaginary helper scripts from older site text without verifying that they exist.

What you need to understand before building

AF is not AE

AF owns discovery, import, deduplication, classification, intake, and bundle construction. AE owns downstream extraction. Your week fails if you confuse those responsibilities.

The handoff is an artifact

The relevant AF → AE contract is the job bundle. If the bundle is malformed, mislabeled, or duplicate-ridden, the handoff is bad even if the PDF itself looks fine.

Question filtering is a second opinion

The question constitution layer can promote rejects and surface edge cases, but it does not replace the broader AF triage logic.

The right unit of progress is a proved behavior

"I worked on the discovery pipeline" is not progress. "The same query run twice makes zero extra API calls and creates no duplicate DOI rows" is.

How to use AI this week

Ask AI for

  • the smallest plausible slice of the repo to inspect first
  • a list of concrete pass/fail tests for your chosen component
  • likely false positives for a discovery or dedup path
  • a tighter version of your success condition
  • an issue template or PR checklist that forces evidence

Do not ask AI to

  • guess which files or commands exist without you giving them the repo facts
  • decide that something works because it "sounds right" architecturally
  • replace reading the actual job-bundle or question-filter contract
  • write a large subsystem before you have named three tests it must pass
Useful prompt pattern this week:
You are looking at Article_Finder_v3_2_3. The exact command/file I need to
understand is: ________. My success condition is: ________. Give me the
smallest implementation path, 3 pass/fail tests, 3 likely false positives,
and the evidence I should capture if the tests pass.

In-class activities

  1. 15 minutes: open the repo and identify the real entry points: python cli/main.py --help, doctor, stats, ui.
  2. 20 minutes: read the Track 2 context docs on AF → AE handoff, question constitutions, and unified registry fields.
  3. 20 minutes: choose one owned component for the week. Examples: intake dedup, question filter, benchmark design, bundle validation, or review workflow.
  4. 20 minutes: write the first success condition and at least three tests for that component before anyone writes fresh code.

At-home work

  • set up the AF repo from your own fork and verify the environment with real commands
  • read docs/PRODUCTION_RUN.md, docs/QUESTION_FILTER_WITH_AG_2026-04-07.md, and the relevant KA contextual notes
  • write a one-page boundary note stating what your chosen component owns, what it emits, and how you will know it worked

Deliverables

Deliverable A environment + verification

Environment proof packet

Rationale
If your local repo, Python environment, and base commands are uncertain, every later claim is weak.
What to submit
A short markdown note or issue comment containing your repo path, branch name, Python version, and the output of python cli/main.py doctor, python cli/main.py stats, and one focused test command such as pytest tests/test_question_relevance_gate.py.
What good looks like
Another student can read the packet and know you are on the correct repo, with the correct CLI, and that at least one real verification command executed in your environment.
Required verification
Include exact commands and outputs, not prose summaries. If something failed, include the failure and your current explanation.
Common false positives
A screenshot of a terminal without the command; an output block from the wrong repo; saying "doctor worked" without showing anything; using stale site commands that are not in AF.
Example
python cli/main.py doctor
python cli/main.py stats
python -m pytest tests/test_question_relevance_gate.py
Deliverable B boundary + ownership

Boundary and success-condition memo

Rationale
Track 2 will drift immediately unless each student knows exactly what part of the pipeline they own and what the handoff boundary is.
What to submit
One page naming the component you are working on, its inputs, outputs, downstream consumer, three explicit tests, and one false positive you expect to encounter.
What good looks like
Someone outside your pair can tell what "done" means without asking you to explain it orally.
Required verification
At least one test must be about integration, not only a local helper function. At least one test must guard against a false positive.
Common false positives
Describing a domain aspiration instead of a pipeline behavior; listing generic goals such as "improve discovery"; omitting the downstream consumer of your output.
Example
"Component: dedup after intake. Pass if the same DOI never creates two accepted candidate rows, and if two title variants with the same authors are flagged for review rather than silently duplicated."
Deliverable C test discipline

Initial test matrix

Rationale
This class is using AI heavily. That makes explicit tests more important, not less.
What to submit
A small table with columns for test name, command or procedure, expected behavior, evidence captured, and pass/fail result.
What good looks like
The matrix contains at least three tests, one edge case, and one integration check. It is written before large implementation work begins.
Required verification
At least one row must be repeatable by a teammate without private local assumptions.
Common false positives
Treating "the page rendered" or "the script exited 0 once" as sufficient evidence; writing test names with no commands or expected outputs.
Example
"Run the same query twice; expected result: second run creates no duplicate DOI row and shows no additional external fetches in the log."

Verification checklist

  • ✓
    Exact repo: you can name the repo URL and branch you are actually using.
  • ✓
    Exact boundary: you can state one AF output and the AE-side artifact or consumer it feeds.
  • ✓
    Exact commands: you ran at least two real AF CLI commands and recorded the outputs.
  • ✓
    Exact tests: your chosen component has at least three named pass/fail tests.
  • ✓
    Exact false positive: you can name one plausible way the system could look correct while being wrong.

Exit condition for the week

You are ready to move on only if your team has stopped talking in generalities. By the end of Week 3.5, each student should be able to say: which component they own, what command or path they are proving, how success will be tested, and what evidence they already have or still lack.
