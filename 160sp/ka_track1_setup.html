Track 1 Setup — Image Tagger

Track 1 uses two required repositories and one optional companion. The Image Tagger produces the per-image latent-variable tags; the Tagging Contractor manages the contracts (schemas, acceptance criteria, validation sweeps) that tell the Tagger what a legitimate tag set looks like. An optional third repository, the Outcome Contractor, layers in outcome-defined search: it lets a user query the tagged image base for photographs that are predicted to produce a particular psychological outcome (say, restoration, or threat-perception). Plan for about an hour — thirty minutes to fork, clone, and install, another thirty for the GitHub-workflow walk through at the bottom of this page.

Scope

Two repositories required, one optional. Everything in Parts A–B refers to kirsh-lab/image_tagger and kirsh-lab/Tagging_Contractor. A third repository, kirsh-lab/Outcome_Contractor, is needed only if your sub-team is working on outcome-defined image retrieval. The Tagger writes a JSON tag set; the Contractor decides whether that tag set is admissible against a contract; the Outcome Contractor (if cloned) layers outcome-prediction queries on top of the admitted tag base. The contract between the Tagger and the Contractor is the only thing Track 1 needs to honour.

Prerequisites

  • GitHub account with SSH keys configured. If you have never set up SSH, follow GitHub's official instructions at docs.github.com/en/authentication/connecting-to-github-with-ssh before Step 1. Verify with ssh -T git@github.com — you should see a message that begins "Hi <your-handle>! You've successfully authenticated…". If you see "Permission denied (publickey)", your key is not yet associated with your GitHub account; fix that before proceeding. Without a working SSH key, every command in Parts A–C will fail in the same opaque way.
  • Python 3.11+ and Git 2.30+ — confirm with python3 --version and git --version.
  • 8 GB free disk for the sample image corpus and the cached vision models.
macOS ships with an older Python. Install a modern one via Homebrew: brew install python@3.12. Windows users should install Python from python.org and tick "Add Python to PATH" in the installer, or use winget install Python.Python.3.12.

Part A — The GitHub Fork Workflow (do this first)

Everyone on Track 1 works from their own fork of each required repository, not from a direct clone of the lab's copy. A fork is your personal server-side copy of the lab's repository: you can push freely to your fork, and when your work is ready you open a pull request back to the lab's copy, which we call upstream. This pattern is the standard open-source contribution model and it is what the rest of the class will evaluate your work through. Because Track 1 spans two required repositories, you will perform the fork-and-wire-up dance twice (and a third time if you also need the Outcome Contractor).

1Fork both required repositories on GitHub (once per person)
  1. Sign in to GitHub.
  2. Open github.com/kirsh-lab/image_tagger and click the Fork button in the top-right corner. Accept the defaults. GitHub creates a copy at github.com/<your-handle>/image_tagger.
  3. Open github.com/kirsh-lab/Tagging_Contractor and click Fork again. Accept the defaults. GitHub creates github.com/<your-handle>/Tagging_Contractor.
  4. Optional, only if your sub-team needs outcome-defined retrieval. Open github.com/kirsh-lab/Outcome_Contractor and click Fork.
  5. For all forks, leave the "Copy the main branch only" box checked — it is faster and you will not need legacy branches.
Why fork instead of clone directly? You do not have write access to kirsh-lab/image_tagger or kirsh-lab/Tagging_Contractor, which is deliberate: it prevents accidental pushes to main and it gives every student a private space to break things. All of your commits will live on your forks until you open a pull request. Track 1 is the only track that operates across two coupled repositories, so keeping the fork hygiene tidy on both is essential — a tag set on your image_tagger branch only validates if the matching contract on your Tagging_Contractor branch is also pushed.
2Clone your forks (not the lab's copies)
mkdir -p ~/REPOS && cd ~/REPOS

# Required fork A — your copy of the tagger itself
git clone git@github.com:<your-handle>/image_tagger.git

# Required fork B — your copy of the contracts that govern admissible tag sets
git clone git@github.com:<your-handle>/Tagging_Contractor.git

# Optional fork C — only if you forked it in Step 1
git clone git@github.com:<your-handle>/Outcome_Contractor.git

# Part A: Fork both required repositories on GitHub
# Part B: Install and run
# Part C: GitHub workflow
# Part D: Wire up the site

Replace <your-handle> with your GitHub username in every line. Each clone creates a remote called origin that points at your fork of that repository. Verify after each clone with git remote -v from inside that repository's directory.

3Add the lab repositories as upstream

For each fork you cloned, add the lab's copy as a second remote called upstream. You will pull from upstream whenever the lab's main advances, and push only to origin.

cd ~/REPOS/image_tagger
git remote add upstream git@github.com:kirsh-lab/image_tagger.git
git remote -v
# You should see:
#   origin    git@github.com:<your-handle>/image_tagger.git (fetch)
#   origin    git@github.com:<your-handle>/image_tagger.git (push)
#   upstream  git@github.com:kirsh-lab/image_tagger.git       (fetch)
#   upstream  git@github.com:kirsh-lab/image_tagger.git       (push)

cd ~/REPOS/Tagging_Contractor
git remote add upstream git@github.com:kirsh-lab/Tagging_Contractor.git
git remote -v
# Same pattern — origin points at your fork, upstream at the lab's copy.

# Only if you cloned the Outcome Contractor:
cd ~/REPOS/Outcome_Contractor
git remote add upstream git@github.com:kirsh-lab/Outcome_Contractor.git
git remote -v

Never push directly to upstream on any of these repositories; the server-side protection rules will reject it anyway, but the next step prevents the command from leaving your machine.

4Disable accidental pushes to upstream (do this for every fork)
cd ~/REPOS/image_tagger        && git remote set-url --push upstream no_push
cd ~/REPOS/Tagging_Contractor  && git remote set-url --push upstream no_push

# Only if you cloned the Outcome Contractor:
cd ~/REPOS/Outcome_Contractor  && git remote set-url --push upstream no_push

This belt-and-braces step makes git push upstream fail at the local level so a mistyped command cannot reach the lab's copies. Run it once per repository at the start of the quarter and forget about it.

Part B — Install and Run the Tagger + Contractor

5Create and activate a shared virtual environment

Create one shared environment at the parent level so both repositories import against the same dependency set:

# macOS / Linux
cd ~/REPOS
python3 -m venv .venv
source .venv/bin/activate

# Windows (PowerShell)
cd $HOME\REPOS
py -3 -m venv .venv
.venv\Scripts\Activate.ps1
6Install dependencies for every cloned repository
pip install --upgrade pip

# Tagger — vision models and rule-based extractors
pip install -r image_tagger/requirements.txt

# Contractor — JSON-schema tooling and validation sweeps
pip install -r Tagging_Contractor/requirements.txt

# Outcome Contractor (only if cloned)
pip install -r Outcome_Contractor/requirements.txt

First install pulls OpenCV, PyTorch, and the COCO-133 segmentation weights. Expect 5–10 minutes on a reasonable connection.

7Download the sample image corpus
python image_tagger/scripts/fetch_sample_images.py --set cogs160_week3

Roughly 120 photographs (living rooms, offices, corridors, outdoor transitional spaces) pre-licensed for classroom use.

8Run the tagger on one image to confirm install
python -m image_tagger.cli tag \
    --image image_tagger/data/samples/cogs160_week3/room_014.jpg \
    --output out/room_014.tags.json

Open out/room_014.tags.json and you should see object-material detections, abstract visual statistics (fractal dimension, visual complexity, openness), and any L-code latent variables the rule-based extractors currently emit. If the JSON looks empty, the model download failed — re-run Step 6.

9Validate the tag set against the contract
python -m tagging_contractor.validate \
    --tags out/room_014.tags.json \
    --contract Tagging_Contractor/contracts/cogs160_week3.contract.json

This is the round trip that makes Track 1 work: the Tagger produces a JSON object and the Contractor either admits it (all schema, coverage, and consistency checks pass) or returns a list of violations that tell you exactly what the Tagger got wrong. Your weekly contributions will usually be either (a) a new rule-based extractor in the Tagger or (b) a tightened contract clause in the Contractor — rarely both in the same PR.

Part C — Your Daily GitHub Workflow

Every unit of work — a new visual feature in the Tagger, a tightened acceptance clause in the Contractor, a fixed parser, a reworded prompt — follows the same six-step loop. Doing it the same way every time is how a class of thirty students can collaborate on coupled repositories without stepping on each other.

Coupled-repo rule for Track 1. If a change touches both the Tagger and the Contractor (for example: you added a new tag class in the Tagger and need a matching contract entry to admit it), open two pull requests with matching branch names — one in each repository — and link them in the PR descriptions. Reviewers will merge them in the order that keeps the round trip in Step 9 passing: usually the Contractor PR first (admit the new class), then the Tagger PR (start producing it).
10Sync your forks with upstream

Run this every time you start a work session, in every fork you have cloned.

cd ~/REPOS/image_tagger
git checkout main
git fetch upstream
git merge --ff-only upstream/main
git push origin main

cd ~/REPOS/Tagging_Contractor
git checkout main
git fetch upstream
git merge --ff-only upstream/main
git push origin main

# Repeat for Outcome_Contractor if cloned.

--ff-only refuses to merge if your local main has diverged from upstream/main; if it does diverge, ask a TA — it usually means something was committed to main directly by mistake, and the fix is not something to improvise.

11Create a feature branch (matching name in every repo you change)
# In whichever repository you are about to change:
git checkout -b track1/<your-handle>/<short-topic>

# If the change spans both repos, repeat with the SAME branch name in the other:
cd ../Tagging_Contractor
git checkout -b track1/<your-handle>/<short-topic>

For example: track1/dkirsh/openness-extractor. The branch name is not cosmetic — our CI reads the track1/ prefix and runs the Track 1 test suite only, which includes the Tagger ↔ Contractor round-trip test. Branches that do not match the prefix get no CI coverage. Using the same branch name in both repos lets a reviewer find the paired PR with one click.

12Commit in small, readable increments
git add path/to/file.py
git commit -m "tagger: add openness extractor for indoor scenes (closes #45)"

One concern per commit; one sentence per commit message, in the imperative. The rest of the commit message is for the why, not the what — the diff already shows what. If you are changing more than one thing at a time, split it into two commits. For Contractor changes, use the contract: prefix in your subject line so PR reviewers can tell at a glance which side of the round trip the change lives on.

13Push your branch(es) to origin (your fork)
git push -u origin track1/<your-handle>/<short-topic>

# If you also branched the Contractor:
cd ../Tagging_Contractor
git push -u origin track1/<your-handle>/<short-topic>

The -u flag sets the upstream tracking reference so future git push and git pull on this branch work without extra arguments. For coupled changes, push the Contractor branch first so the Tagger PR's CI can resolve the matching contract entry it needs.

14Open a Pull Request against upstream/main
  1. Visit your fork on GitHub. GitHub will display a yellow banner offering to open a PR against the lab's main. Click Compare & pull request.
  2. Title: one sentence describing the change. Body: a short paragraph on motivation, a bullet list of what changed, and a Testing section showing the command you ran to verify the change — for Track 1, that is almost always the validate command in Step 9.
  3. If the change spans both repositories, open the second PR now and link it from the first PR body with a line such as "Pairs with kirsh-lab/Tagging_Contractor#<n>".
  4. Request review from your track lead.
  5. Wait for CI (about three minutes). If CI fails, read the log, fix, push again — the PR updates automatically.
15After the PR merges, clean up
cd ~/REPOS/image_tagger
git checkout main
git fetch upstream
git merge --ff-only upstream/main
git push origin main
git branch -d track1/<your-handle>/<short-topic>
git push origin --delete track1/<your-handle>/<short-topic>

# Repeat in Tagging_Contractor (and Outcome_Contractor) if you branched there.

Your feature branch's job is done the moment the PR merges. Keeping merged branches around creates noise for the next contributor reviewing your fork.

A visual summary. upstream/main on each repository is the canonical history; your origin/main is a mirror of it; your feature branch sprouts from main, lives on origin, is reviewed as a PR, and is deleted on merge. Sync at the start of every session, branch before every change, one PR per concern, clean up after merge. For Track 1's coupled changes: same branch name in both repositories, Contractor PR opened first, link them in the descriptions.

Common Snags

  • SSL errors on pip install — usually a stale pip. Rerun pip install --upgrade pip certifi.
  • Torch install hangs on Windows — swap to the CPU wheel: pip install torch --index-url https://download.pytorch.org/whl/cpu.
  • "image_tagger not a package" — you ran the CLI from the wrong directory. cd back to ~/REPOS (the parent that contains both repositories) before re-running.
  • Contractor says "contract file not found" — the --contract path is relative to your current working directory. Use an absolute path or run the validator from the ~/REPOS parent.
  • git push rejected with "Permission denied" on either kirsh-lab repository — you cloned the lab copy directly instead of your fork. Re-do Steps 1–3 from your forked URL. Do not try to "fix" this with git remote set-url origin … until you understand which copy you have; ask a TA.
  • Tagger PR's CI fails on the round-trip test — the matching contract change is missing or unmerged. Push the Contractor branch and open its PR (Step 14) before worrying about the Tagger CI failure; once both PRs exist, the CI will pass against the paired branch.
