VR Environment Production

You are the environment team. Your job is to translate experimental protocols from the research fronts into navigable, instrumented virtual scenes that could actually be used to run the missing studies. ● Phase 1 — In Progress
Why this phase matters

Knowledge Atlas tracks research fronts — clusters of open empirical questions ranked by Value of Information (VOI). Track 3's job is to pick one paradigm from those fronts where a VR implementation would genuinely close a gap: a study design that cannot be run cheaply in a standard lab but becomes tractable in VR. The selection memo produced in this phase is the justification your design rests on for the rest of the quarter.

Start here: Browse Knowledge Atlas Topics and filter for high-VOI entries. Look for paradigms tagged "VR-feasible" or "lab-constrained." The Research Gaps page lists unfilled paradigm slots explicitly.
Candidate paradigms (select one)

The following paradigms have been flagged as high-value VR targets based on ecological validity constraints or participant burden that VR can reduce. Select the one you will build for this quarter.

Attention Restoration
Lab versions (slideshow images) severely compress the restorative pathway. VR allows navigation through immersive natural vs. urban scenes with dwell-time and gaze tracking as DVs.
VOI: HIGH · ART, Kaplan & Kaplan 1989
Stress Inoculation Environments
Workplace stress protocols require real confederates and actors. VR allows controlled stress induction (social evaluative threat) with repeatable, parametric scene variation and no deception risk.
VOI: HIGH · Dickerson & Kemeny 2004
Spatial Navigation & Working Memory
Naturalistic wayfinding in real buildings is prohibitively expensive to replicate across labs. VR building environments with identical floor-plans allow cross-site replication of hippocampal navigation protocols.
VOI: MEDIUM · O'Keefe & Dostrovsky 1971
Social Presence & Embodied Cognition
Full-body illusion and avatar embodiment studies require motion capture rigs. A-Frame + WebXR hand-tracking provides a lower-cost equivalent suitable for pilot work on body-ownership and social proximity effects.
VOI: MEDIUM · Blanke & Metzinger 2009
Distraction & Focused Attention (Open Office)
Ecological open-plan studies require an actual office. A VR office allows parametric manipulation of distractor density, distance, and type (visual/acoustic) while measuring task performance and gaze allocations.
VOI: HIGH · Banbury & Berry 2005

○ Phase 2 — Not Started
What is protocol translation?

Every published experiment describes an IV–DV structure, a participant task, a stimulus set, and a procedure. Your job is to map each of those elements onto a VR scene. This is not straightforward: lab stimuli are often 2D images, audio clips, or confederate actors — none of which translate directly into a navigable 3D space. The translation requires explicit design decisions about spatial geometry, navigation affordances, task timing, and logging architecture.

Core constraint: The scene must preserve the causal structure of the original protocol. A rendered forest that the participant can walk through is ecologically richer than a slideshow — but only if the experimenter can still manipulate and measure the same variables. Every translation decision must be justified against this constraint.

○ Phase 3 — Not Started
Platform choice

All COGS 160 VR scenes must be built as standalone HTML files using A-Frame. This guarantees they run in any browser without a game engine installation, can be submitted to the KA repo, and are accessible to reviewers and future contributors without a headset (desktop + mouse also works for scene review). Unity and Unreal projects are not accepted for Track 3 deliverables, though students may prototype in those environments and port to A-Frame for submission.

🌐
A-Frame
Required for submission
Web-based VR framework built on top of Three.js. Scenes are HTML files. No build step, no install required for reviewers.
  • Runs in browser (desktop + VR)
  • Entity–component architecture
  • WebXR controller support
  • Gaze cursor and raycaster components
  • GLTF model loader built in
✓ Required format
🔺
Three.js (direct)
Advanced option
Lower-level than A-Frame; more control over rendering pipeline. Acceptable if the scene cannot be built with A-Frame's component system.
  • Full WebGL access
  • Requires more boilerplate
  • Must still export as single HTML file
🎮
Unity / Unreal (prototyping only)
Not submittable
Useful for rapid scene prototyping and visual design, but cannot be submitted to KA repo. Port to A-Frame before PR.
  • Rich asset marketplace
  • Better physics and rendering
  • Requires instructor sign-off to use

○ Phase 4 — Not Started
What must be measured

A VR scene without data collection is scenery, not a research instrument. Phase 4 wires at least three of the DVs from your design spec into the scene's event-logging system. All events must be timestamped relative to scene start, labelled with a structured event name, and written to a downloadable JSON log at session end.

Minimum DV set (pick three or more): Gaze dwell time on tagged objects / zones · Navigation path length (metres) · Time-on-task per trial · Interaction events (button presses, object selections) · Zone entry/exit timestamps · Self-report rating (5-point Likert, rendered in-scene on a visible panel)
Standard event log format

Every event written by the logging component must conform to this structure. Downstream analysis scripts assume it.

Scene start
event: ka_scene_start
t: 0 (always zero)
data: paradigm, condition, participant_id
{"event":"ka_scene_start","t":0,"data":{"paradigm":"art","condition":"natural","pid":"P01"}}
Gaze dwell
event: ka_gaze_dwell
t: ms since scene start
data: target_id, dwell_ms, position
{"event":"ka_gaze_dwell","t":4210,"data":{"target":"tree_cluster_A","dwell_ms":1820,"pos":[1.2,-2.1]}}
Zone entry/exit
event: ka_zone_enter / ka_zone_exit
t: ms since scene start
data: zone_id, camera_pos
{"event":"ka_zone_enter","t":7840,"data":{"zone":"task_zone_1","cam":[0.8,1.6,-4.9]}}
Path position
event: ka_position_tick
t: ms since scene start
data: x, y, z (camera world position)
{"event":"ka_position_tick","t":2000,"data":{"x":0.3,"y":1.6,"z":-2.1}}
Likert rating
event: ka_rating
t: ms since scene start
data: scale_id, value (1–5)
{"event":"ka_rating","t":62000,"data":{"scale":"restoration","value":4}}
Scene end / download
event: ka_scene_end
t: ms since scene start
data: total_events, log_file_name
{"event":"ka_scene_end","t":90200,"data":{"total_events":142,"file":"ka_log_P01.json"}}
Logging component code

Add the log-events A-Frame component to your scene. Paste this into your scene's <script> block and attach the log-events attribute to <a-scene>.

AFRAME.registerComponent('log-events', { init: function () { this.log = []; this.t0 = Date.now(); this.record('ka_scene_start', { paradigm: 'PLACEHOLDER', condition: 'PLACEHOLDER', pid: (new URLSearchParams(location.search)).get('pid') || 'anon' }); // Position tick every 2 s setInterval(() => { const cam = document.getElementById('rig').object3D.position; this.record('ka_position_tick', {x: +cam.x.toFixed(2), y: +cam.y.toFixed(2), z: +cam.z.toFixed(2)}); }, 2000); // Download button (optional — or call downloadLog() from console) document.addEventListener('keydown', e => { if (e.key === 'd') this.downloadLog(); }); }, record: function (event, data) { this.log.push({ event, t: Date.now() - this.t0, data }); }, downloadLog: function () { const pid = (new URLSearchParams(location.search)).get('pid') || 'anon'; const blob = new Blob([JSON.stringify(this.log, null, 2)], {type: 'application/json'}); const a = document.createElement('a'); a.href = URL.createObjectURL(blob); a.download = `ka_log_${pid}.json`; a.click(); } });

○ Phase 5 — Not Started
Why naïve-user validation is required

Scenes that the builder can navigate fluently often break on first contact with a new user. Naïve-user walkthroughs catch: missing orientation cues (users don't know where to go), invisible task triggers, cybersickness from poor locomotion design, and logging gaps (events not firing when they should). Two walkthroughs are the minimum; three or more significantly increase the reliability of the validation report.

Walkthrough protocol
Standard observer protocol (read before each walkthrough):
Seat the participant. Open the scene. Say: "I won't help you — just explore and do whatever seems right." Start a screen recording. Do not narrate or gesture. After 5 minutes, ask: (1) What did you think the task was? (2) Was anything confusing? (3) Did the scene feel natural? Note their verbal answers and your observations in the walkthrough log below.

Do I need a VR headset?
No. A-Frame scenes run in any modern browser. Desktop mode works perfectly for development and peer review. If you want to test on an actual headset, many are available in the UCSD labs (check with your TA).
Can I use Unity instead of A-Frame?
You can build in Unity if you want, but you must submit as a browser-runnable A-Frame HTML scene. The course requires scenes that open directly in Chrome without an installer or build process.
What if my scene runs slowly?
Reduce polygon count, compress textures to <100kb each, test in Chrome with DevTools Performance tab. A-Frame's performance profiler (built-in) shows which entities are expensive. Keep FPS ≥ 60 for comfortable VR.
Where do I submit my scene?
Commit to your branch, push, and open a Pull Request to track/3-staging. Include the HTML file, a README explaining how to run it, and the validation report. See STEP 0 above for details.
