Why this week matters

A weak literature system can always make its numbers look better by being permissive. Week 5 is where you stop rewarding permissiveness and start rewarding discrimination: fewer duplicates, cleaner metadata, clearer reasons, and a bundle that AE can process without guessing.

What you need to understand before building

Deduplication is multi-signal

DOI matching is strongest, but title and author overlap matter too. A system that only catches exact DOI matches will miss real duplicates.

Metadata quality affects every later stage

Bad title, DOI, year, or abstract fields make relevance, bundling, and later extraction less reliable.

The handoff schema is not decoration

paper.json is the formal contract between AF and AE. If required fields are missing or misleading, the handoff is bad.

Borderline cases teach you about scope

The question filter can surface near misses, topic expansion candidates, and new-topic candidates. Those categories should not be flattened into a single "maybe."

How to use AI this week

Ask AI for

  • adversarial duplicate examples that stress DOI, title, and author matching
  • a benchmark design for accepted, rejected, and edge-case papers
  • a schema-validation checklist for job bundles
  • failure modes in title normalization, DOI parsing, and APA formatting

Do not ask AI to

  • invent missing metadata instead of marking the uncertainty
  • erase edge cases because they complicate the demo
  • claim that a bundle is contract-compliant without inspecting the generated files

In-class activities

  1. 20 minutes: assemble a 20-paper benchmark set containing obvious accepts, obvious rejects, duplicates, and borderline cases.
  2. 20 minutes: run the current dedup and triage path over that set.
  3. 20 minutes: inspect one or more generated bundles and compare the metadata against the schema and source paper records.
  4. 20 minutes: decide where the current errors come from: matching, enrichment, constitution boundaries, or bundle construction.

At-home work

  • improve one quality-control component: dedup, metadata repair, question gating, or bundle validation
  • re-run the same benchmark and record before/after behavior
  • build at least one real AE-ready bundle from a paper you would currently keep

Deliverables

Deliverable A benchmark

Twenty-paper benchmark with labels

Rationale
You cannot improve quality without a fixed set of cases that includes things the pipeline should reject or flag.
What to submit
A labeled benchmark table with at least 20 entries spanning accept, review, reject, duplicate, and edge-case categories.
What good looks like
The benchmark includes hard cases, not only obvious wins. It is suitable for rerunning after repairs.
Required verification
Each label must have a brief justification grounded in title, abstract, article type, or metadata quality.
Common false positives
Using a benchmark made only of clearly in-domain papers; leaving duplicates out because they are inconvenient; labeling with no rationale.
Example
A mixed set containing two DOI duplicates, one fuzzy title duplicate, three review articles, and two adjacent restoration papers that trigger edge-case discussion.
Deliverable B before/after repair

Quality repair with before/after evidence

Rationale
The class needs to see that a concrete repair reduced a known weakness.
What to submit
A branch or patch plus a short note showing the old behavior, the change, and the new behavior on the same benchmark or run.
What good looks like
The note isolates one improvement clearly enough that someone else can say whether it truly helped.
Required verification
Use the same benchmark or topic run before and after. Do not change the test set midstream unless you say why.
Common false positives
Changing the query and the dedup logic at the same time; showing only the new output; claiming success because counts changed without explaining why.
Example
Title normalization now catches one fuzzy duplicate pair that previously slipped into the accepted set.
Deliverable C handoff proof

One valid AF → AE bundle

Rationale
Track 2 cannot claim progress unless it can hand AE something structurally trustworthy.
What to submit
A generated job bundle plus a short inspection note showing the contents of paper.json, PDF hash presence, and why the paper belongs.
What good looks like
The bundle is not merely present on disk. Its metadata lines up with the paper record, the schema, and the reason for inclusion.
Required verification
Show the command used to build it, the bundle path, and the specific fields you inspected.
Common false positives
Handing over a PDF with no trustworthy metadata; assuming one successful file write means the bundle is valid; not checking that the paper is actually one you would send onward.
Example
python cli/main.py build-jobs --status send_to_eater --output data/job_bundles followed by inspection of the created paper.json.

Verification checklist

  • ✓
    Benchmark exists: you have a fixed test set with labels.
  • ✓
    False positives are named: at least one paper or duplicate case exposed a real weakness.
  • ✓
    Repair is evidenced: you can compare before and after on the same or clearly comparable cases.
  • ✓
    Bundle inspected: you looked inside the handoff artifact, not just its folder name.
  • ✓
    Scope is honest: if the question filter or bundle path is still partial, you say so explicitly.

Exit condition for the week

Move on only when you can show that the system is becoming more selective and more trustworthy, not merely busier. A valid bundle, a named benchmark, and one before/after repair are the minimum evidence.
