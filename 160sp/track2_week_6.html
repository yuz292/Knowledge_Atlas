Make the pipeline robust across topics

Week 6 is where you find out whether your improvements generalize. The question is no longer "does it work for one convenient topic?" but "does it stay sensible when we vary the topic, the research front, and the failure modes?"

This week's question

When we run AF on multiple topics or research fronts, does it remain disciplined about relevance, duplication, logging, and uncertainty?

Why this week matters

A pipeline that works on one hand-picked topic may simply be overfit. Week 6 tests whether your logic survives topic changes, adjacent literatures, messy metadata, and the real ambiguity that appears when a corpus grows.

Near miss is not the same as topic expansion

If your filter surfaces a paper that almost fits the current topic, that is one kind of signal. If it surfaces a paper that suggests the topic boundary itself should expand, that is another. Keep those signals separate.

What you need to understand before building

Generalization requires comparison

You need at least two topics, bundles, or research fronts if you want to say the pipeline behaves consistently.

Logging is part of correctness

If a rerun duplicates papers or silently changes decisions and you cannot tell why, the system is not yet trustworthy.

Failure taxonomy matters

Separate API failure, metadata absence, duplicate miss, borderline topic fit, and schema failure. Those are different repair problems.

Cross-topic drift is expected

The goal is not zero drift. The goal is to detect it, characterize it, and decide whether it is acceptable.

How to use AI this week

Ask AI for

  • a comparison matrix for two topics or research fronts
  • a failure taxonomy for the logs you are seeing
  • a minimal logging schema: what counts, thresholds, and event fields should be recorded
  • possible topic-expansion interpretations of your edge cases

Do not ask AI to

  • average away real disagreements between two topics
  • turn every edge case into a topic-expansion candidate by default
  • write "robustness" language before you have comparative evidence

In-class activities

  1. 20 minutes: choose two distinct topics, questions, or research fronts.
  2. 20 minutes: run the same AF path on both and record counts, top candidates, duplicates, and reject reasons.
  3. 20 minutes: inspect differences and name which ones reflect true topical difference and which ones look like pipeline weakness.
  4. 20 minutes: write a short failure log with categories and proposed repairs.

At-home work

  • repeat the two-topic comparison cleanly and store the results in one place
  • improve one logging or review surface so that failures are easier to diagnose next time
  • document any genuine near miss, topic expansion, or new-topic signal that emerged

Deliverables

Deliverable A comparison

Two-topic comparison report

Rationale
You need evidence that the system does not only behave sensibly on one favored topic.
What to submit
A report comparing two runs: commands, counts, accepted papers, borderline papers, rejects, duplicates, and notable differences.
What good looks like
The report identifies both topic-specific effects and pipeline-specific weaknesses.
Required verification
Use comparable settings across the two runs and say when you intentionally changed something.
Common false positives
Comparing two totally different workflows and calling the result "robustness"; using only aggregate counts with no qualitative inspection.
Example
One topic produces many review-only papers because it contains more theory literature; another produces more empirical accepts. That difference is documented rather than hidden.
Deliverable B failure log

Failure log with repair loop

Rationale
Mature systems improve because failures are categorized, not merely complained about.
What to submit
At least five logged failures, each with category, evidence, probable cause, and next repair step.
What good looks like
The failures are concrete and differentiated: duplicate miss, metadata gap, API issue, edge-case ambiguity, bad bundle field, and so on.
Required verification
Each logged failure includes one observable symptom such as a count mismatch, missing field, unexpected status, or wrong routing decision.
Common false positives
A vague "it was inconsistent"; listing only bugs that were already fixed; no evidence attached.
Example
"Title-only intake created two borderline papers with inconsistent year parsing; symptom: duplicate titles but different years in the DB."
Deliverable C instrumentation

One improved review or logging aid

Rationale
Week 6 should leave the team with better visibility into what the pipeline is doing.
What to submit
A small code change, dashboard view, report template, or review worksheet that makes one class of failure easier to detect.
What good looks like
The aid shortens the gap between "something looks wrong" and "we know what to inspect next."
Required verification
Show the aid in use on one real comparison or failure case.
Common false positives
A cosmetic chart with no decision value; extra logging that nobody can interpret; a doc page unrelated to the observed failures.
Example
A compact table showing, per run, total candidates, accepted, review, reject, duplicates, and edge-case counts.

Verification checklist

  • ✓
    Two comparable runs: you can compare them without hand-waving away major differences.
  • ✓
    Failure categories: at least five failures are named with evidence and next steps.
  • ✓
    Uncertainty preserved: near misses, topic-expansion signals, and new-topic signals remain distinct where relevant.
  • ✓
    Instrumentation improved: one aid now makes debugging or review easier than it was in Week 5.

Exit condition for the week

You can move to Week 7 once the team has comparative evidence, a structured failure log, and at least one better way to see what the pipeline is doing. If all you have is a second pile of papers, you are not ready yet.
