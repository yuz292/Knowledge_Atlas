◎Goalwhat this deliverable teaches

You will take the top three findings from your T4.d backlog — filtered through T4.e to keep only those that reproduced — and produce a mid-fidelity redesign for each. A mid-fidelity redesign is more committal than a sketch but less committal than working code: a wireframe or annotated mockup that names every new affordance, explains every changed label, and declares a pre-registered success metric that a future usability pilot could measure.

This is a medium-hardness deliverable because the temptation is to let yourself off the hook by producing a vague "looks better" mockup. Three disciplines prevent this. First, each redesign must be tied by ID to a backlog row, so the problem it solves is non-negotiable. Second, each redesign carries a pre-registered success metric — before you have evidence it works. Third, each redesign is paired with a GOMS-style task-time estimate that an observer could falsify.

›What you'll dothe steps

  1. Load your T4.d backlog.csv. Filter out any rows rated NONE or UNDERSPECIFIED in T4.e. From what remains, pick the three highest-RICE rows.
  2. For each of the three, produce a mid-fidelity mockup. Acceptable formats: an annotated screenshot edited in Figma / Sketch / Keynote / PowerPoint, a hand-drawn wireframe photographed cleanly, or a raw HTML mockup in a standalone file. The bar is "someone else could implement this without talking to you."
  3. Write a 250-word design note per redesign covering: the problem it solves (cite the backlog ID), the mechanism by which it helps (cite a principle or heuristic), a declared success metric (what a future pilot should measure and what threshold counts as success), and one trade-off you are accepting.
  4. For at least one redesign, produce a GOMS-style task-time estimate: list the goals, operators, methods, and selections for the revised flow; sum the operator times from Card, Moran & Newell's (1983) keystroke-level model.
  5. Submit one redesign_{id}.md per redesign plus the mockup asset, under 160sp/tracks/t4/{your_id}/redesign/. Three redesigns total.

✗Example of a naive approachso you can see what scores low

The classic weak redesign is visual restyling without an argument. Read the block below and notice how quickly the prose shifts from "the user cannot" to "it would look nicer if".

band 0–1 example — do NOT submit a redesign note like this

Redesign 1 — make the home page better

I think the home page could be a lot better. I would change the colours to something more modern and add some icons to the buttons. I would also make the font larger. I think this would help users understand the site faster. The metric would be that users like it more. I would measure this by asking them.

Why this scores band 0 or 1. Every sentence is an aesthetic preference presented as a design claim. There is no backlog ID cited, so we cannot tell which finding is being answered. "Make the colours more modern" is not a mechanism; "add some icons" is not a mechanism; "make the font larger" is a mechanism only if you name which text and cite a readability result (e.g. Dyson, 2004 on line-length and comprehension). The metric is "users like it more" — this is a failure of the pre-registration requirement twice over: like it more than what baseline, and measured how, and at what threshold counts as success? A band-3 redesign names a specific baseline (a pilot time-on-task median or an error rate from T4.c), declares a falsifiable threshold, and states what the designer would conclude if the threshold were missed. Crucially, no trade-off is acknowledged: every design choice buys one thing and costs another, and a proposal that claims only upside is not an engineering document, it is marketing copy.

The redesign-note template. One file per redesign:

# Redesign {id} — {short title}

**Backlog row**: {F-xx from T4.d}  (RICE {value}, severity {n}, replication verdict {FULL|PARTIAL})
**Mockup**: ./redesign_{id}.png  (or .html, .fig, .pdf)

## Problem
{30–60 words, literal, citing pilot quote or audit observation}

## Mechanism — why this proposal addresses the problem
{40–80 words; cite a heuristic, principle, or empirical result}

## Pre-registered success metric
- **Baseline**: {what the current site measures on this task, from T4.c}
- **Threshold**: {numeric threshold that would count as success}
- **Measurement protocol**: {exactly how a follow-up pilot would measure it}
- **Falsification condition**: {what would count as "this redesign did NOT work"}

## Trade-off accepted
{one explicit thing this redesign costs; a design with no costs is not a design}

## GOMS estimate (if applicable)
{Goals / Operators / Methods / Selections list + keystroke-level time}

✓Verification

how your work is graded
  • Automated: the AI grader parses three redesign_*.md files and a mockup asset per redesign. Missing asset caps Evidence at band 1.
  • Automated: the Backlog row field must match a real row in your T4.d CSV; mismatch caps Traceability at band 0.
  • Automated: absence of a Pre-registered success metric section with all four sub-fields caps Rigor at band 1.
  • TA audit: a TA sketches out the proposed flow and challenges one assumption; ability to respond to the challenge in writing is assessed in T4.g (the final report).

Scoring bands (0–3 per criterion, rescaled to the 10-point value):
