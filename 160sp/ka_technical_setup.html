Set Up Your Development Environment

This page walks you through everything you need to start working on your track. No prior experience with AI tools, Git, or programming is assumed. Follow the steps in order — each one builds on the last.

The Big Picture: How This Course Works

Understand the workflow before you install anything

In COGS 160, you build real software tools that contribute to the Knowledge Atlas — a research system that organizes scientific evidence about how built environments affect human cognition and well-being. The key difference from a normal programming class is that you work with an AI coding assistant, not alone.

The workflow is: you think, the AI writes code, and you verify and improve. You are the scientist-designer; the AI is the fast typist. Your job is to define what to build, evaluate whether it works, and fix what goes wrong.

You are not expected to be a programmer. Many students in this course have never written code before. That is fine. The AI tools handle the syntax; you handle the thinking. What you need is curiosity, willingness to experiment, and the ability to describe what you want clearly.

Here is what each piece of the system does:

The Knowledge Atlas Website

This is the public-facing product — a website where anyone can explore scientific evidence about how architecture and environments affect people. It has pages for topics, evidence, argumentation structures, and knowledge gaps. You can browse it right now from the navigation above.

The GitHub Repositories

The code and data behind the website live in GitHub repositories (repos). Think of a repo as a shared folder that tracks every change anyone makes. There are two main repos you will encounter:

Knowledge_Atlas — contains the website pages, data payloads (JSON files with evidence and argumentation data), and course materials (including this page).

Article_Eater — contains the AI pipeline that extracts evidence from scientific papers. Track 2 and Track 5 students work with this repo directly.

Your AI Coding Assistant

You will use an AI tool to help you write, debug, and test code. The AI reads a context file that tells it everything about the specific exercise you are working on — the data format, the expected output, and the constraints. You paste this context in at the start of each session.

Install Basic Software

You need a code editor and a web browser — that is all for most exercises

The Week 2 exercises produce single HTML files that run in your browser. You do not need to install Python, databases, or anything complicated for now. Here is the minimum:

Access the Course Repository

Where the code, data, and context files live

All the data and context files you need for the exercises are stored in a GitHub repository. There are two ways to access them, depending on your comfort level.

Option A — Browse Online (Easiest)

Read Files on GitHub

You can read any file directly on the GitHub website without installing anything. Navigate to the repository, find the file you need, and click on it to view its contents. To use a file, click the Raw button and copy the text.

This is the simplest approach for Week 2. You only need to download files when an exercise requires local data (most don't).

View Knowledge_Atlas on GitHub →

Option B — Clone to Your Computer

Fork and Clone the Repository

If you plan to make changes to the code (especially for Tracks 1, 2, and 4), you will eventually need your own copy of the repository on your computer. Here is how:

Step 1: Create a GitHub account (if you do not have one) at github.com. Use your UCSD email.

Step 2: Fork the repository. Go to the repo page and click the Fork button in the top right. This creates your own copy on GitHub.

Step 3: Clone your fork to your computer.

# Replace YOUR_USERNAME with your GitHub username
git clone https://github.com/YOUR_USERNAME/Knowledge_Atlas.git

# Move into the folder
cd Knowledge_Atlas

# Verify you are in the right place
ls

Step 4: Keep your fork up to date. As the course progresses, the main repository will change. To get updates: 

# Add the original repo as "upstream" (do this once)
git remote add upstream https://github.com/dkirsh/Knowledge_Atlas.git

# Pull latest changes (do this weekly)
git fetch upstream
git merge upstream/main
Which option should I use? For Week 2, Option A (browsing online) is all you need. When your track requires you to create and submit code files, switch to Option B. Your track page will tell you when.

Using Context Files with Your AI Tool

This is the key technique — how you give AI the knowledge it needs

Every exercise in this course comes with a context file. A context file is a plain-text document (a .md file) that tells your AI everything it needs to know about the exercise: what data format to expect, what the output should look like, what constraints to follow, and what mistakes to avoid.

Without a context file, the AI will guess — and guess wrong. With one, it produces code that actually works with our data.

How to use a context file

The Three-Step Process

1. Find the context file. Context files live in the 160sp/context/ folder of the Knowledge Atlas repository. Each exercise page tells you exactly which file to use. The files are also listed on the Context File Index.

2. Copy its contents. Open the file on GitHub (or in your local clone). Select all the text and copy it to your clipboard.

3. Paste it into your AI as the very first message. Start a new conversation with your AI tool. Paste the entire context file as your first message. Then say something like: 

I am a COGS 160 student working on [Exercise X]. I have pasted the
context file above. Please read it carefully before we begin.

My first task is: [describe what you want to build]
Start a new conversation for each exercise. AI tools have limited memory within a conversation. If you switch exercises, start a fresh conversation and paste the new context file. Mixing contexts confuses the AI.

Exercise A context_exA_trust_panel.md “Why Trust This?” evidence panel with confidence indicators

Accessing the Data

The JSON files your exercises will read

The exercises do not ask you to read scientific papers directly. Instead, they work with pre-extracted data — JSON files that contain evidence, argumentation structures, and article metadata already pulled from over 760 research papers. Your job is to build tools that visualize and analyze this data.

The two main data files are:

evidence.json

Contains approximately 1,900 evidence records. Each record links a scientific finding to an environment variable (e.g., daylight level) and an outcome variable (e.g., attention restoration), with metadata about effect sizes, sample sizes, and the source paper. Used by Exercises A, C, and D.

argumentation.json

Contains 38 debate clusters with approximately 4,800 edges linking claims that support or attack each other. Each cluster represents a real scientific disagreement — for example, whether natural environments restore attention (Attention Restoration Theory) or reduce stress (Stress Recovery Theory). Used by Exercises 0 and B.

These files are served directly from the K-ATLAS website. Your code will fetch them with a URL like: 

// In your JavaScript, the context file will tell you the exact URL
const response = await fetch('data/ka_payloads/evidence.json');
const data = await response.json();
You do not need to download these files manually. Your HTML file will fetch the data automatically when it runs in the browser, because the exercises are designed to be served from the same site. If you are working locally and encounter a CORS error, see the troubleshooting section below.

Working on Exercises Locally

The step-by-step cycle for completing an exercise

Here is the complete cycle for working on a Week 2 exercise:

Step 1

Open the Exercise Page

Go to the Week 2 Exercises page. Read the description for the exercise assigned to your team. Note which context file you need.

Step 2

Get the Context File

Navigate to 160sp/context/ in the repository (online or local). Open the context file for your exercise and copy its full contents.

Step 3

Start an AI Session

Open your AI tool (Claude, ChatGPT, etc.). Start a new conversation. Paste the context file as your first message. Then describe what you want to build, following the 7-Step Method described on the exercise page.

Step 4

Save the Code

When the AI generates code, copy it and save it as an .html file in your cogs160 folder. Name it descriptively: exA_trust_panel_v1.html

Step 5

Open in Your Browser

Double-click the HTML file to open it in your browser. Does it work? Does it show data? If not, go back to the AI, describe what went wrong, and iterate.

Step 6

Verify and Reflect

Steps 5–7 of the 7-Step Method ask you to verify the output, diagnose problems, and reflect on what you learned. Write up your verification and reflection — this is 50% of your grade.

Troubleshooting Common Problems

Solutions to the issues students hit most often
My HTML file is blank when I open it
Open the browser console (right-click → Inspect → Console tab) and look for red error messages. Copy the error and paste it back to your AI — it can usually fix the problem.
I get a “CORS error” when loading data
This happens when you open an HTML file locally (file://) and it tries to fetch data from the K-ATLAS server. Fix: use a local server. In your terminal, navigate to your folder and run python3 -m http.server 8000, then open localhost:8000/your_file.html.
The AI writes code that does not work
This is normal and expected. Copy the error message from your browser console and paste it back into the AI conversation. Say: “This is the error I see. Please fix the code.” Iteration is part of the process — not a failure.
I ran out of free AI messages
Both Claude and ChatGPT have daily limits on free tiers. Options: (a) wait a few hours and try again, (b) switch to the other tool, (c) start a new conversation (sometimes limits reset per conversation).
I do not know what to tell the AI
Start with the 7-Step Method. Step 1 is “Assemble a Panel” — ask the AI to play the roles of experts who discuss the problem. This gets the conversation going even if you do not know the technical details yet.
My code looks different from other teams
That is expected. Every team will produce a different implementation. The grading rubric cares about the process (panel quality, spec completeness, verification rigor, reflection honesty), not whether your code matches anyone else's.
What is the “7-Step Method”?
Panel → Spec → Tests → Build → Verify → Diagnose → Iterate. It is the development workflow for every exercise. See the Week 2 Exercises page for the full explanation.
Can I use a different AI tool?
Yes. Any AI that can generate JavaScript/HTML code will work. The context files are tool-agnostic. If you use a tool not listed here, just make sure it can handle long input (the context files are ~2,000 words).

Track 5 focuses on quality assurance: verifying evidence claims, evaluating warrant strengths, and building trust indicators. The Week 2 exercises (A and C) are your starting point.
