TRACK 2 — ARTICLE FINDER

Track 2: Article Finder

Build a PRISMA-compliant article discovery pipeline — fix the contribute page, extract knowledge gaps, generate targeted queries, execute searches, and triage results at the abstract level.

3 tasks 10 weeks 75 + 60 + 75 points +20 contract bonus on Tasks 2 & 3

What you’ll build

The Knowledge Atlas needs a steady inflow of papers. Researchers working on environment-and-cognition need to know which questions the literature has answered, which it has only half-answered, and which it has not yet asked. Without a disciplined discovery pipeline, the Atlas drifts on whatever its contributors happen to have read this week.

You will build that pipeline end-to-end, and you will build it to the PRISMA reporting standard — the gold-standard checklist for documenting how a literature search found, screened, and selected its papers. PRISMA is the difference between a search someone else can audit and a pile of links nobody can defend.

You start in Task 1 with a broken contribute page that accepts PDFs and does nothing with them. You diagnose what is missing, write a contract that specifies the fix, hand the contract to your AI, and verify that submitted papers actually get classified, stored, and reported back to the user. In Task 2 you turn to discovery proper: you read the Article Eater’s PNU mechanism templates, identify the knowledge gaps the Atlas most needs filled, score them by Value of Information, and translate each gap into a paired Google AI Citation query and a Boolean query. In Task 3 you run those queries through SerpAPI, collect abstracts via a multi-source fallback chain, triage the returned papers at the abstract level, and report every step of the funnel on a PRISMA dashboard.

The throughline is the contract. Every task requires a written contract that specifies inputs, processing, outputs, success conditions, and a test checklist. A weak contract produces weak work; Tasks 2 and 3 each award twenty bonus points for contract quality, because the contract is the artefact that lets a collaborator (human or AI) build the right thing without watching you do it.

How the pipeline is organised

The pipeline has a specific shape, and the shape matters. Candidate references arrive from two channels. The first is review papers already in the Atlas corpus that cite other work; an Article Eater script (extract_neuro_key_review_references.py) already harvests these and produces a co-citation count per DOI. The second channel is search engines that respond to the queries you write in Task 2. Search runs through a coordinated set of three harvest scrapers, plus a fourth scraper that operates only at PDF retrieval:

  • SerpAPI — paid Google Scholar channel; fast, reliable, capped at 250 free credits per month.
  • scholarly — free Python package that scrapes Google Scholar HTML; the quota-exhausted fallback. Brittle; CAPTCHA-prone; document its results as best-effort.
  • paper-scraper — free preprint-server scraper covering arXiv, bioRxiv, medRxiv, and chemRxiv; the right tool for high-VOI gaps where the literature is moving fast.
  • scidownl — PDF acquisition tool of last resort; Sci-Hub-mirror-backed; UCSD library policy unresolved. Opt-in only, after Unpaywall and OpenAlex both fail, with instructor sign-off. Treated as a Stage-3 source, never a default scraper.

Every candidate, regardless of channel, walks through the same three-stage triage funnel. Stage 1 is metadata-only screening on title and venue — a cheap classifier pass that rejects 30 to 50 per cent of search noise without spending a Semantic Scholar request or a SerpAPI credit. Stage 2 is abstract collection through the four-source fallback chain (Semantic Scholar → CrossRef → PubMed → OpenAlex), classifier scoring on the abstract, and a four-way decision: ACCEPT, EDGE_CASE, REJECT, or MISSING_ABSTRACT. Stage 3 is PDF retrieval, run only for ACCEPT rows and only via the source cascade (Unpaywall first, OpenAlex OA second, scidownl third when the policy flag is set). The cardinal rule of the funnel: never download a PDF to decide whether you want the paper.

Where harvested references live: the article_references table

Every candidate reference your pipeline harvests — whether from a review-PDF reference list or from one of the three search scrapers — lands as a row in the article_references table in the article lifecycle database (pipeline_lifecycle_full.db). This table is the candidate buffer between harvesting and acquisition: it holds papers the system knows about but does not yet have a PDF for.

The table carries provenance (discovered_via, discovered_from_paper_id, discovered_query), triage state (triage_stage, triage_decision, voi_score, classifier_score), and forward links into the papers table once the PDF is acquired (acquired_paper_id). The PRISMA dashboard you build in Task 3 reconstructs the entire funnel from one SQL group-by on this table; no spreadsheets, no parallel state, no free-floating JSON. The full DDL lives in scripts/coordination/lifecycle/schema.sql; the reconnaissance document at docs/AF_PIPELINE_RECON_2026-04-27.md §3 walks through the schema, the indexes, and the v_acquisition_queue convenience view.

Free-floating outputs do not count. A reference list extracted into a JSON file that never joins back to the lifecycle DB is invisible to the rest of the pipeline. Every harvested candidate must land as an article_references row; otherwise the next harvester will rediscover it, the PRISMA dashboard cannot count it, and your work is unverifiable.

The three tasks

Fix the Contribute Page

Search Execution & Triage

Repos you’ll need

Track 2 spans four repositories. Fork the ones you will commit to; read the others.

Knowledge_Atlas
The Atlas itself — this is where the contribute page you fix in Task 1 lives, and where your discovery dashboard ships.
Article_Finder
The discovery half of the system — the pipeline that turns knowledge gaps into queries, runs them, and triages the returned abstracts.
Article_Eater
The processing half — evidence extraction from the published literature, including the PNU mechanism templates whose unfilled slots are the gaps your queries target.
atlas_shared
Shared schemas and the classifier subsystem that the contribute page wires into. Install with pip install -e . from the repo root.

How submission works for Track 2

Track 2 uses a single git branch per student — track2/your-name — across all 3 tasks. Each completed task is submitted as its own Pull Request to the instructor. The full mechanics are on each task page; here’s the lifecycle in one paragraph:

  1. Once at the start of the track: fork the track’s repos, clone, set the upstream remote, create track2/your-name off upstream/main.
  2. Continuously while working: commit small, push at the end of each session.
  3. At the end of each task: open a Pull Request from your branch to dkirsh:main, title it Track 2 · Task N: ... — your name, paste your file manifest, tag @dkirsh, do not merge.
  4. Across tasks: keep using the same branch; additional task PRs are opened against the previous task’s merge commit.

The detailed commands and the per-task PR title format are on each task page under “Submit Task N on GitHub.”
