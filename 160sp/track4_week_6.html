Why this week matters

Finding problems is necessary but not sufficient. This week you demonstrate that you can fix real usability issues in production code, submit them as reviewable Pull Requests, and verify that fixes for one persona do not create new problems for another.

Cross-persona review is mandatory

Every PR gets reviewed by a student holding a different persona. A navigation fix that helps the Practitioner (who wants to get in and out fast) might remove contextual information that the Researcher needs. The cross-review catches these trade-offs before they ship.

Types of fixes

Navigation & information architecture

Missing breadcrumbs, inconsistent back-links, dead-end pages, unclear active-state indicators, broken or misleading labels. These are often CSS + HTML fixes with high impact.

Interactivity

Broken or missing search/filter, non-functional buttons, localStorage issues, keyboard navigation gaps, screen reader failures. These require JS work and testing.

Visualization improvements

Axis labels, colour contrast, responsive sizing, tooltip clarity, data-ink ratio. Apply Tufte's principle: every pixel of ink should convey information.

Content hierarchy

Portal page routing, card layout, progressive disclosure of complexity. The question is always: does the page reveal the right amount of information for the user's current task?

Accessibility

Colour contrast (WCAG 2.1 AA: 4.5:1 for text, 3:1 for large text), keyboard navigability, focus indicators, alt text, ARIA labels. These are non-negotiable on any merge.

How to use AI this week

Ask AI for

  • implementation — "write the CSS change that fixes this breadcrumb inconsistency"
  • accessibility checking — "does this colour combination meet WCAG AA?"
  • test generation — "write a checklist I can use to verify this fix works in Chrome and Firefox"
  • PR description — "summarise this change in terms of which heuristic it addresses"

Do not ask AI to

  • generate a fix you do not understand — you must be able to explain every line in the PR
  • skip the cross-persona review — "it looks fine to me" is not a review
  • merge without testing on the local server

In-class activities

Tuesday — Fix sprint planning
Start here: Bring your audit report's top 5 findings ranked by priority.

In class

  • Present your top 5 fixes with mockups, wireframes, or code sketches.
  • Instructor approves or redirects — some fixes may be too large; scope down.
  • Begin implementing fix #1 and fix #2.

Home

  • Complete fixes #1–#3. Test each on the local server.
  • Take before/after screenshots for each fix.
  • Open PRs with clear descriptions citing the heuristic addressed.
Thursday — Cross-persona review
Start here: Each student reviews one of the other students' PRs.

In class

  • Walk the fixed page through your persona's eyes (not the fixer's persona).
  • File review comments: does the fix help your persona, hurt them, or have no effect?
  • Discuss trade-offs as a group. If a fix helps one persona but hurts another, how do you resolve it?

Home

  • Complete fixes #4 and #5 based on feedback.
  • Merge approved PRs.
  • Update friction inventory to reflect resolved issues.

Week deliverables

D7 Five merged Pull Requests

Rationale
PRs are the tangible output of the fix sprint. They prove you can move from diagnosis to treatment in production code.

What to submit
5 PRs, each with: a title naming the page and heuristic, before/after screenshots, a description explaining the fix, and at least one cross-persona review comment.

What good work looks like
Each PR addresses a specific, documented finding from the audit report. Before/after screenshots make the improvement visible. The cross-review confirms no persona regression.

Example
"Fix H4 violation: standardise sidebar link colours across content-nav pages. Before: 3 different teal shades. After: consistent --teal (#2A7868) on all 13 pages. Cross-review: Practitioner persona confirms faster scanning."
