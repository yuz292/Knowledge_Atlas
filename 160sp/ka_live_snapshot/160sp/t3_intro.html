TRACK 3 — VR STUDIO

Track 3: VR Studio

Build the VR pipeline for the Knowledge Atlas — collect 20+ VR-ready 3D models, convert and validate them in an interactive viewer, then ship an AI front-end that modifies scenes from natural-language prompts.

3 tasks 10 weeks 75 + 75 + 75 points

What you’ll build

The Knowledge Atlas stores evidence that specific environmental features — ceiling height, light spectrum, surface materials, biophilic content — alter human cognition and affect. To test those relationships in virtual reality, researchers need 3D rooms they can change one factor at a time. They need to raise a ceiling without warping the walls, swap the north wall’s material without touching the other three, and dim the lights without reloading the scene. Almost no off-the-shelf 3D model supports that.

Track 3 builds the VR Studio — the in-house name for the project’s end-to-end VR content pipeline — that makes it possible. You will start with nothing and finish with three working pieces that compose into a single tool. In Task 1 you curate a library of at least twenty VR-ready interior models in glTF or GLB format, scoring each against eight manipulation classes and shipping a mesh annotation file that maps every mesh to its architectural role (ceiling, north wall, window, floor, and so on). In Task 2 you build a WebXR viewer that loads any of those models and exposes real-time slider controls for every manipulation class, then test all ten of your strongest models against all eight factors to produce a factor viability matrix. In Task 3 you wrap that viewer in an AI front-end so a researcher can type “a warm cozy library with 4-meter ceilings and exposed brick on the north wall” and watch the scene reconfigure in seconds.

The architectural innovation is in what the LLM does not do: it never generates 3D geometry. Instead it maps natural language onto a constrained JSON schema of parametric scene modifications — named parameters (ceiling height, wall colour, light intensity) that runtime code can edit without rebuilding the model. Your Task 2 viewer applies the JSON; a validation gate checks every modification against your viability matrix before it touches the scene. The result is the first tool that lets a neuroarchitecture researcher dial up a Meyers-Levy high-ceiling condition or an Appleton prospect-dominant scene on their own model in seconds rather than weeks.

The three tasks

TASK 1
Collect & Catalog VR-Ready 3D Interior Models
Find and curate at least 20 VR-ready interior models in glTF/GLB format, evaluate them against 8 manipulation classes, and ship structured mesh annotation files plus a ruthless validation prompt.
4 phases · 75 points
TASK 2
VR Conversion & Manual Factor Testing
Build an interactive WebXR viewer with real-time slider controls for each manipulation class, then test every model against every factor and write ruthless validation tests.
3 phases · 75 points
TASK 3
AI Front-End for Parametric Scene Modification
Build a scene introspector and an LLM parameter extractor so users modify scenes by typing natural-language prompts; add a validation gate, presets from the literature, and tests on 10+ models.
5 phases · 75 points

Repos you’ll need

Track 3 spans four repositories. Fork the ones you will commit to; read the others.

Knowledge_Atlas
The Atlas itself — this is where your model catalog, WebXR viewer, and AI front-end ship to students and researchers.
KA_VR_Studio
The VR pipeline repo — the home for your model library, mesh_roles/ annotations, Blender segmentation scripts, the A-Frame viewer, and the AI front-end code.
Tagging_Contractor
The latent-tag registry — 424 tags across 8 manipulation classes (geometry, lighting, materials, colour, biophilic, furniture, acoustic, complexity) that define your viability scoring rubric and parameter space.
Article_Eater
Evidence extraction from the published literature, including the 166 PNU mechanism templates whose experimental conditions become the research presets in Task 3 (Meyers-Levy ceilings, Appleton prospect-refuge, biophilic restoration, and the rest).

How submission works for Track 3

Track 3 uses a single git branch per student — track3/your-name — across all 3 tasks. Each completed task is submitted as its own Pull Request to the instructor. The full mechanics are on each task page; here’s the lifecycle in one paragraph:

  1. Once at the start of the track: fork the track’s repos, clone, set the upstream remote, create track3/your-name off upstream/main.
  2. Continuously while working: commit small, push at the end of each session.
  3. At the end of each task: open a Pull Request from your branch to dkirsh:main, title it Track 3 · Task N: ... — your name, paste your file manifest, tag @dkirsh, do not merge.
  4. Across tasks: keep using the same branch; additional task PRs are opened against the previous task’s merge commit.

The detailed commands and the per-task PR title format are on each task page under “Submit Task N on GitHub.”
