Getting Started as a KA Contributor

This page tells you everything you need to set up your environment, claim your track, get your files, and make your first contribution — from a blank laptop to a working development setup.

OVERVIEW

What you are joining

A production knowledge system you will extend, not a toy project

Knowledge Atlas (KA) is a live, versioned software system. It has a git history, a database, automated tests, and a formal extraction pipeline. Your work this quarter will be committed to a real repository, reviewed against defined acceptance criteria, and — if it passes — merged into the main system.

This means the setup steps below are not optional formalities. A misconfigured Python environment will silently corrupt data. Working on the wrong branch will overwrite your teammates' code. Running the wrong script against the production database will corrupt records that took months to build. Follow the instructions in order.

One rule above all others: Never run any script that writes to the database unless you have confirmed you are pointing at the student copy (web_persistence_v5_student.db), not the operational database. Check the connection string before every session. This is the single most common first-week mistake.
STEP 1

Install your environment

Python 3.12 · git · VS Code · browser dev tools

All four tools are required. If you already have them, verify the versions below and move on.

macOS

# Install Homebrew if you don't have it
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# Install Python 3.12 and git
brew install python@3.12 git

# Verify
python3 --version   # should print Python 3.12.x
git --version       # should print git version 2.x

# Install VS Code (if not already installed)
brew install --cask visual-studio-code

Windows 11

# In PowerShell (run as Administrator)
winget install Python.Python.3.12
winget install Git.Git
winget install Microsoft.VisualStudioCode

# Restart your terminal, then verify
python --version
git --version

Required Python packages (all tracks)

# After cloning the repo (Step 2), run from the repo root
pip3 install -r requirements.txt --break-system-packages

# If requirements.txt is not yet present, install the core set manually
pip3 install sqlite3 requests pandas beautifulsoup4 \
             pytest black ruff --break-system-packages
Browser dev tools. All track 4 students (and any student who will test or evaluate KA pages) need to know how to open Chrome DevTools (⌘⌥I on Mac, F12 on Windows), inspect the DOM, read the Console tab, and monitor Network requests. There is a 10-minute walkthrough at the start of the Track 4 assignment page.
STEP 2

Clone the repository and get on your track branch

One repo, one branch per track, one personal branch per student

The KA repository lives at github.com/dkirsh/Knowledge_Atlas. You will work on a personal branch that is based on your track's staging branch. Your work accumulates there until the instructor reviews and merges it upstream.

  1. Clone the repo

    Run this once. You will do all subsequent work inside this directory.

    git clone https://github.com/dkirsh/Knowledge_Atlas.git
cd Knowledge_Atlas
  2. Configure your identity

    Git requires your name and email before you can commit. Use your UCSD email.

    git config user.name "Your Name"
git config user.email "you@ucsd.edu"
  3. Create your personal branch off your track staging branch

    Replace track-N with your assigned track number (1–4) and yourname with your UCSD username.

    # Fetch the latest staging branches
git fetch origin

# Switch to your track staging branch
git checkout track/N-staging       # e.g., track/2-staging

# Create your personal branch from it
git checkout -b track/N-staging/yourname   # e.g., track/2-staging/dkirsh
  4. Push your personal branch to the remote immediately

    This registers your branch on GitHub so the instructor can see it from Day 1.

    git push -u origin track/N-staging/yourname
  5. Never commit directly to master or to your track staging branch

    All your commits go on your personal branch. When a sprint is complete, you open a Pull Request from your personal branch into the track staging branch. The instructor reviews and merges. That is the only path work takes into the shared system.

Branch naming convention. The pattern track/N-staging/username is intentional: it groups all track-2 branches together in the GitHub branch list, making it easy for the instructor to see everyone's work at a glance.

STEP 3

Get your student database copy

Read from production — write to your copy only

The KA system's state lives in web_persistence_v5.db. You will never write to this file. The student copy, web_persistence_v5_student.db, is a snapshot made at the start of the quarter. It is safe to modify, break, and rebuild. 

# The student DB copy is in data/student/ — confirm it exists
ls data/student/web_persistence_v5_student.db

# If it's missing, ask the instructor to regenerate it before proceeding

Every Python script you run must use the student DB path. The canonical way to confirm this in any script is to check the connection string at the top of the file: 

# Correct — student copy
DB_PATH = "data/student/web_persistence_v5_student.db"

# WRONG — never use this path in your work
DB_PATH = "data/web_persistence_v5.db"
If you accidentally write to the production DB: Stop immediately, do not run any more scripts, and message the instructor. Do not attempt to undo or repair the damage yourself. The instructor has backup snapshots.

Track 1 Image Tagging & CNFA Annotation ▼

Your job: annotate research images with CNFA construct tags using the 5-algorithm-bundle protocol. Your code will read from the KA image registry and write tag assignments back to the student DB.

  • Read ka_tag_assignment.html in full — the 5 algorithm bundles (A–E) and Phase 0 table are your specification.
  • Open image_tag_service.py — skim its 41 attributes and 5 domain scores (spatial, material, lighting, social, nature).
  • Run the existing tag pipeline on a test image and confirm output: python3 scripts/tag_image_test.py
  • Read contracts/IMAGE_TAGGING_CONTRACT.md — this defines what an accepted tag assignment looks like.
  • Find a sample image in data/images/ and manually apply bundles A–E using the Phase 0 reference table. Write down your reasoning. This is your warm-up before the first sprint.
  • Post your personal branch URL in the Track 1 Slack/Teams channel.

Track 2 Article Finding Pipeline ▼

Your job: grow the KA article corpus. You will build search queries, run literature searches, screen articles, acquire PDFs, and ingest them through the extraction pipeline. Hard minimum: 150 citation records, 50 PDFs, by Week 8.

One explicit Track 2 front: help build the future Neural Underpinnings layer. Some article-finder teams will be assigned pathway families such as Predictive Processing, Salience Networks, Interoceptive / Constructionist Affect, Neuromodulatory Systems, Chronobiological Regulation, or Multisensory Integration. For that pathway, your job is to collect classic anchors, recent reviews, direct-measurement papers, and environment-relevant papers, while marking clearly what is directly measured versus only inferred.
  • Read ka_article_finder_assignment.html in full — the 5-phase pipeline and milestone table are your spec.
  • Open data/student/research_queue_state.json — these are your gap targets. You own a slice of this list (assigned in Week 1).
  • Run the duplicate check on one test DOI: python3 scripts/check_duplicate.py --doi=10.xxxx/xxxx
  • Configure your local AF API settings before heavy search work: use a valid email for OpenAlex / Crossref / Unpaywall polite-pool access, and follow the current class guidance on when the OpenAlex trial key may be used.
  • Run AF commands through the repo virtualenv, not bare python3, or you may miss installed scorer/search dependencies and get false failures.
  • Read the Google AI Query Guide in docs/GOOGLE_AI_QUERY_GUIDE.md before writing any queries.
  • Set up your Query Diary: create data/student/query_log_yourname.json using the schema in contracts/QUERY_LOG_SCHEMA.md.
  • Run one test query on Google AI or Semantic Scholar. Record it in your Query Diary. This is your Week 3 starting point.
  • If assigned a neural pathway family, start with 3–8 classic anchor papers before expanding to reviews and direct-measurement papers.
  • Post your personal branch URL in the Track 2 Slack/Teams channel.

STEP 5

How to submit work — the Pull Request workflow

Commit → push → open PR → await review

Every piece of work travels the same path: your personal branch → track staging branch → (after instructor review) → master. You do not merge your own PRs. You open them, describe what you did and what the acceptance criteria are, and wait for review. 

# 1. Stage your changes (list files explicitly, not git add -A)
git add data/student/query_log_dkirsh.json

# 2. Write a descriptive commit message
git commit -m "T2 Week 3: 15 queries across 4 tools, 3 gap targets (acoustic/spacing/light)"

# 3. Push to your remote branch
git push

# 4. Go to github.com/dkirsh/Knowledge_Atlas → Pull Requests → New PR
#    Base: track/2-staging   Compare: track/2-staging/yourname
#    Title: [T2 Week 3] 15 queries, 3 gap targets
#    Body: list what you did and paste the acceptance criteria from the assignment
Good commit messages include your track, the week, and a brief description of what changed and why. Bad commit messages ("updates", "fix", "wip") make it impossible for the instructor to review your work efficiently. Treat the commit message as the subject line of an email to a careful reader.

What the instructor checks when reviewing your PR

  • Does the code / data match the stated acceptance criteria for this sprint milestone?
  • Is the student DB path correct? (No writes to production path.)
  • Are tests passing? (pytest tests/track_N/ must pass before requesting review.)
  • Is the commit message descriptive enough to reconstruct what was done without reading the diff?
  • Is the change self-contained — i.e., would reverting this PR leave the codebase in a clean state?
STEP 6

Using AI assistance correctly

AI is a directed instrument in your pipeline — not a chatbot

This course teaches a specific methodology for AI-assisted development. The goal is not to "get good output" — it is to produce output that meets a formal specification, passes defined tests, and integrates correctly with the larger system. That requires a different approach than asking a chatbot for help.

The methodology has five components: role assignment, task definition, constraints and prohibitions, context loading, and output format specification. These are covered in full — with worked examples for each track — in the AI Methodology guide.
