Each Extractor Is a Scientific Claim. The Test Suite Is Your Evidence.

Implementing a CNFA tag is not a coding exercise. It is an argument: "this computation measures this psychological construct, because here is what it does to images we know the answer for." This week you write both the argument and its proof.

Deliverable: D2 — Test Suite + Math Implementations Weight: 15% of final grade Due: End of Week 5 (Sunday 11:59 PM) Team: Pair work — one shared GitHub branch per pair

In Week 3 you triaged 20 tags and augmented 10 registry entries with bilingual fields. In Week 4 you established what VLMs can and cannot do reliably. This week you write the deterministic extractors — the math-based ones that can be tested against known ground truth. VLM-based semantic extractors come in Week 6, after you understand what deterministic computation looks like and where it stops working.

The central requirement is test-before-code. Write the tests first. This is not a stylistic preference; it is a methodological one. Hagerhall et al. (2004) showed that fractal dimension predicts landscape preference — but only when computed correctly on images with sufficient resolution and appropriate edge detection. If you write the extractor first and then check whether the tests pass, you will unconsciously tune the extractor to pass the tests rather than to measure the construct. Writing the tests first, against the definition, prevents that drift.

tests/test_fractal_dimension.py
import numpy as np
import cv2
from backend.science.math.fractal import compute_fractal_dimension

def test_pure_white_near_zero():
    """A blank white image has no edges → fractal dimension ≈ 0."""
    img = np.ones((256, 256, 3), dtype=np.uint8) * 255
    result = compute_fractal_dimension(img)
    assert result < 0.10, f"Expected < 0.10 for blank image, got {result:.4f}"

def test_pure_black_near_zero():
    """A blank black image has no edges → fractal dimension ≈ 0."""
    img = np.zeros((256, 256, 3), dtype=np.uint8)
    result = compute_fractal_dimension(img)
    assert result < 0.10, f"Expected < 0.10 for blank image, got {result:.4f}"

def test_fine_checkerboard_moderate():
    """A 4×4 px checkerboard has regular edges at one scale → moderate D (0.35–0.65)."""
    img = np.zeros((256, 256, 3), dtype=np.uint8)
    for y in range(0, 256, 4):
        for x in range(0, 256, 4):
            if (y // 4 + x // 4) % 2 == 0:
                img[y:y+4, x:x+4] = 255
    result = compute_fractal_dimension(img)
    assert 0.35 < result < 0.65, f"Expected 0.35–0.65 for checkerboard, got {result:.4f}"

def test_single_horizontal_line_low():
    """One straight horizontal line is a 1D object → D close to 0 after normalization."""
    img = np.zeros((256, 256, 3), dtype=np.uint8)
    img[128, :] = 255   # single white row
    result = compute_fractal_dimension(img)
    assert result < 0.25, f"Expected < 0.25 for single line, got {result:.4f}"

def test_monotonicity_with_noise():
    """Adding Gaussian noise should monotonically increase fractal dimension."""
    base = np.zeros((256, 256, 3), dtype=np.uint8)
    prev = compute_fractal_dimension(base)
    for sigma in [5, 20, 60]:
        noisy = np.clip(base.astype(float) +
                        np.random.normal(128, sigma, base.shape), 0, 255).astype(np.uint8)
        curr = compute_fractal_dimension(noisy)
        assert curr >= prev, f"Expected monotone increase at sigma={sigma}"
        prev = curr
Common mistake: Writing tests that will pass regardless of whether your implementation is correct — for example, assert 0 <= result <= 1. That just tests that you normalized. The test must fail if your algorithm is broken. The blank-image test (no edges → D ≈ 0) is a hard constraint, not a range check.
STEP 3 Implement the Extractor
backend/science/math/fractal.py
import cv2
import numpy as np
from typing import Optional

def compute_fractal_dimension(image: np.ndarray,
                               canny_lo: int = 50,
                               canny_hi: int = 150) -> float:
    """
    Box-counting fractal dimension of image texture.

    Returns a value in [0, 1] where 0 = no texture (smooth wall),
    1 = maximally complex self-similar texture (dense foliage).

    Algorithm: Canny edge detection → box counting across 6 scales →
    log-log linear fit → slope = raw D → normalize to [0,1].

    References:
        Mandelbrot, B. B. (1982). The Fractal Geometry of Nature. Freeman.
        Taylor, R. P. et al. (1999). Fractal analysis of Pollock's drip paintings.
            Nature, 399, 422. https://doi.org/10.1038/20833
        Hagerhall, C. M. et al. (2004). Fractal dimension of landscape silhouette
            outlines as a predictor of landscape preference.
            J. Environmental Psychology, 24, 247–255.
    """
    if image.size == 0:
        return 0.0

    gray = cv2.cvtColor(image, cv2.COLOR_BGR2GRAY)
    edges = cv2.Canny(gray, threshold1=canny_lo, threshold2=canny_hi)

    # Box sizes: powers of 2 from 4px to 128px
    sizes = [4, 8, 16, 32, 64, 128]
    counts = []
    h, w = edges.shape
    for size in sizes:
        n_boxes = 0
        for y in range(0, h, size):
            for x in range(0, w, size):
                if edges[y:y+size, x:x+size].any():
                    n_boxes += 1
        counts.append(max(n_boxes, 1))   # avoid log(0)

    # Log-log fit: slope is the fractal dimension D
    log_inv_size = np.log(1.0 / np.array(sizes, dtype=float))
    log_count    = np.log(np.array(counts, dtype=float))
    slope, _     = np.polyfit(log_inv_size, log_count, 1)

    # Raw D is in [1, 2] for 2D images; normalize to [0, 1]
    d_raw = float(np.clip(slope, 1.0, 2.0))
    return (d_raw - 1.0) / 1.0
STEP 4 Wrap in the Spatial Bounding Utility

Every extractor must be called through the spatial wrapper. Do not return a single scalar — return the full regional structure. The wrapper is already in the codebase; you just need to use it.

backend/science/spatial.py (utility — do not modify)
from typing import Callable, Dict, Any
import numpy as np

def apply_regional(image: np.ndarray,
                    extractor_fn: Callable[[np.ndarray], float],
                    grid_rows: int = 3,
                    grid_cols: int = 3) -> Dict[str, Any]:
    """
    Apply a scalar extractor to the whole image and to each cell of a
    (grid_rows × grid_cols) spatial grid.

    Returns:
        {
          "whole_image": float,
          "grid_shape": [rows, cols],
          "cells": [
            {"row": r, "col": c, "value": float|None,
             "bbox_xywh": [x, y, w, h]},
            ...
          ]
        }
    Cells where the crop is smaller than 16×16 px receive value=None.
    """
    h, w = image.shape[:2]
    cell_h, cell_w = h // grid_rows, w // grid_cols
    cells = []
    for r in range(grid_rows):
        for c in range(grid_cols):
            y1 = r * cell_h;  y2 = (r+1) * cell_h
            x1 = c * cell_w;  x2 = (c+1) * cell_w
            crop = image[y1:y2, x1:x2]
            value = extractor_fn(crop) if min(crop.shape[:2]) >= 16 else None
            cells.append({"row": r, "col": c, "value": value,
                          "bbox_xywh": [x1, y1, cell_w, cell_h]})
    return {"whole_image": extractor_fn(image),
            "grid_shape": [grid_rows, grid_cols],
            "cells": cells}

Technical Background

Why Box-Counting for Fractal Dimension?

The box-counting method (Liebovitch & Toth, 1989) is the standard approach for estimating the fractal dimension of 2D images because it is computationally straightforward and well-validated. The core idea: if an object is self-similar across scales, the number of boxes of size r needed to cover it scales as N(r) ∝ r−D. Plotting log N against log(1/r) gives a straight line whose slope is D. For a smooth surface (no self-similar edges), D ≈ 1. For a maximally complex texture, D approaches 2. We normalize to [0, 1] by applying (D − 1) / 1.

The psychological significance is established by work from Richard Taylor and colleagues (Taylor et al., 1999; Hagerhall et al., 2004; Joye, 2007): human aesthetic preference shows an inverted-U curve peaking at D ≈ 1.3–1.5 (normalized ≈ 0.3–0.5). Dense urban environments cluster around D ≈ 1.2–1.6; natural landscapes around D ≈ 1.3–1.8. This is the scientific claim your implementation operationalizes.

Making a Tag Regional: The Core How-To

The apply_regional utility (Step 4 above) handles grid slicing. Here is what you need to understand about what it does internally:

# How array slicing defines spatial regions in NumPy:
# Image shape is (height, width, channels). Y is rows; X is columns.

h, w = image.shape[:2]
cell_h = h // 3          # e.g. 512 // 3 = 170 px per row
cell_w = w // 3

# Top-left cell (row=0, col=0):
top_left = image[0:cell_h, 0:cell_w]

# Upper-centre cell (row=0, col=1) — useful for ceiling queries:
top_centre = image[0:cell_h, cell_w:2*cell_w]

# Lower-right cell (row=2, col=2) — useful for floor queries:
bot_right = image[2*cell_h:, 2*cell_w:]

# Threshold query — find cells where value ≥ 0.6:
results = apply_regional(img, compute_fractal_dimension)
hot = [(c["row"], c["col"]) for c in results["cells"]
       if c["value"] is not None and c["value"] >= 0.6]
Minimum region size: Most algorithms behave badly on very small patches. The spatial wrapper enforces a 16×16 px minimum and returns None for smaller cells. For a 512×512 image with a 3×3 grid, each cell is ~170×170 px — well above the minimum. For the box-counting algorithm specifically, the largest box size (128 px) must be smaller than the shortest cell dimension, otherwise the fit uses fewer than four points and becomes unreliable. If your images are small, use fewer grid divisions (2×2 instead of 3×3).

Canny Edge Detection — Choosing Thresholds

Canny has two parameters, threshold1 (lower) and threshold2 (upper). Edges with gradient magnitude above threshold2 are always kept; those below threshold1 are discarded; those between are kept only if connected to a strong edge. The default (50/150) is appropriate for most interior images. However:

  • Low-contrast images (dim rooms, overcast natural light) may need a lower threshold (20/80).
  • HDR or strongly post-processed images may be oversaturated in edges — use a higher threshold (100/200).
  • Run cv2.Canny(gray, 50, 150) and display the result before computing D; inspect it. Does it pick up the texture you expect?

Known Limitations of This Approach

You must state these in your tag passport:

  • JPEG compression artifacts add spurious high-frequency edges that inflate D. Pre-filter with a small Gaussian blur (cv2.GaussianBlur(gray, (3,3), 0)) before Canny if you observe anomalously high scores on compressed images.
  • Depth cues are ignored: a room with many small objects in the foreground will score high even if the background is smooth. The score reflects what the camera sees, not the physical complexity of the space.
  • Lighting conditions: very low light collapses edge contrast. Very bright light (blown highlights) eliminates texture. Flag images with extreme exposure.

FAQ

The registry entry for my tag is very vague — it just says "estimate visually." What do I do? ▼

This is a genuine problem and worth raising in your tag passport. If extraction_notes_2d does not give you a computable procedure, you have three options. First, search the literature: is there an established computer vision proxy for this perceptual construct? (The references above are a starting point; also try Google Scholar with the tag name + "image analysis" or "texture feature.") Second, ask Claude: describe the construct and ask what image statistics would correlate with it — but treat the answer as a hypothesis, not a fact, and test it. Third, note in the triage table that this tag needs additional specification and flag it as requiring VLM-based extraction (Week 6). Do not invent an algorithm without justification and present it as authoritative.

My implementation passes all tests but gives nonsensical results on real images. What's wrong? ▼

This is more common than you might expect and reveals a genuine gap between synthetic test cases and real-world data. The most common causes: (1) Your tests did not cover the distribution of the real images — they were too clean or too ideal. (2) The algorithm has a parameter that needs tuning for architectural interiors specifically (e.g., Canny thresholds, or the range of box sizes). (3) The tag definition, when applied to real rooms, does not separate perceptually meaningfully. All three are important findings worth reporting. First, visualize the intermediate outputs — for fractal dimension, display the Canny edge map for a real image. Does it look right? If the edge map is nearly empty (low-contrast image) or completely full (JPEG artifacts), you have found the problem.

Why 3×3 grid? My images are landscape and the cells come out as wide rectangles. ▼

The 3×3 grid is the default, not a fixed requirement. You can pass any grid_rows, grid_cols to apply_regional. For architectural interiors at 4:3 or 16:9 aspect ratio, a 3×4 grid (3 rows, 4 columns) gives more nearly square cells, which matters for some algorithms. However: check whether your algorithm's behaviour changes with cell aspect ratio before changing from 3×3. For fractal dimension specifically, the box-counting algorithm is not aspect-ratio-sensitive because it tiles the cell independently in both axes. For edge-density or symmetry algorithms, rectangular cells may produce systematic bias. Document whatever grid you use and why.

Can I use scikit-image or scipy instead of OpenCV? ▼

Yes, with one constraint: the function signature must remain compute_[tag](image: np.ndarray) -> float where image is a BGR uint8 array (OpenCV convention), and the output must be in [0, 1]. Any library that produces that output is acceptable. If you use scikit-image, note that it expects RGB arrays — convert with image_rgb = image[:, :, ::-1]. Document your library choices in the tag passport.

How do I handle images that are smaller than 128×128 px (the largest box size)? ▼

Cap the largest box size at min(h, w) // 2. For a 90×90 image, use sizes [4, 8, 16, 32, 45]. The fit will have fewer points and be less reliable — flag this in the output as "low_resolution_warning": true. For the spatial wrapper applied to a 3×3 grid, a 90×90 image gives 30×30 px cells — below the recommended minimum for most algorithms. Switch to a 2×2 grid, or simply report the whole-image value only.

My tag is "perceived naturalness" — there is no formula for this. Is this a Week 5 or Week 6 tag? ▼

Week 6. Perceived naturalness requires scene understanding — it is not derivable from pixel statistics alone. It is a VLM-based semantic tag: you will prompt a vision-language model with something like "On a scale of 0 to 10, how natural does this space feel? Consider biophilic elements, material character, and the absence of industrial or synthetic surfaces. Respond with a single number." In Week 5, check whether any proxy for naturalness is extractable deterministically — for example, the proportion of green-hued pixels, or the fraction of the image that matches a vegetation colour range. That proxy would be a separate math-based tag (e.g., green_pixel_fraction) that can be computed this week, even though it is not the same as perceived naturalness.
