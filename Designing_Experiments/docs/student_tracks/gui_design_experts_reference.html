Expert Design Panel Reference

Verbatim advice from the canonical voices in HCI, UX, and interaction design — and how each argument shaped the ATLAS interface you are auditing.
Status: Permanent reference — use throughout Weeks 3–10 Relevance: All GUI track deliverables Companion: Visualization Design Guide

How to use this document. This page records, verbatim, the arguments that the expert panel made when consulted during the design of the ATLAS user-home and workflow-hub pages (March 2026). Each expert's position is stated in their own terms — not summarised into bullet points — because the reasoning matters as much as the conclusion. When you write your usability critiques and success conditions, you should cite these experts by name with APA references from the list at the bottom of this page. A critique that cites Norman on affordances or Shneiderman on closure is more persuasive than one that says "it was confusing."

A companion page, Visualization Design Guide, covers data display, chart choice, and Tufte's data-ink principle for the ATLAS dashboard and evidence cards.

Verbatim Panel Advice

The following records, as faithfully as possible, the argument each expert makes when applied to the specific design problem of a role-based research tool for architectural and environmental psychology. These are not summaries. They are the reasoning, stated in the terms each expert uses.

🧠
Don Norman — The Design of Everyday Things (1988, revised 2013)
Cognitive ergonomics · UCSD emeritus · co-founder, Nielsen Norman Group

Norman's central argument is that poor design is not a user failure but a design failure. When people cannot figure out how to use a system, the fault lies in the system's failure to provide adequate signifiers, mappings, and feedback — not in the user's cognitive limitations. Applied to ATLAS: when a student cannot figure out where to start, or submits an article and receives no feedback about whether it landed, the problem is the interface, not the student.

The concept of the gulf of execution — the gap between the user's intention and the available actions — is directly applicable to ATLAS's article submission flow. If the student's intention is "I want to contribute this paper to the evidence base," but the available actions on the page present a generic upload form with no context about how the paper will be processed, the gulf is wide. Norman argues that closing this gulf requires making the relationship between intention and action legible through what he calls natural mappings: the form should reflect the mental model of the task, not the data model of the database.

The corresponding gulf of evaluation — the gap between the system's state and the user's perception of that state — is equally present in ATLAS. After submitting an article, the student has no reliable signal about what happened. Norman is unambiguous: feedback must be immediate, informative, and matched to the action. A silent success is a design failure. A loading spinner with no progress indication is a design failure. The user's model of the system state must stay aligned with the actual system state at all times.

Norman also distinguishes between knowledge in the head (what the user has memorized) and knowledge in the world (information embedded in the environment that can be read off without prior learning). Good design minimizes the demand for knowledge in the head and maximizes the availability of knowledge in the world. Every workflow step in ATLAS should be legible to someone who has not memorized the system — because during a live class session, no one has.

Key principles for your ATLAS audit

  • Every action should produce a response the user can see within 100ms (perceived as immediate) or 1 second (perceived as responding), or show a progress indicator
  • The form should reflect the user's task model, not the database schema — label fields for what they mean to the user, not what they mean to the server
  • Error messages should explain what went wrong and what to do next — "Error 422" fails on both counts
  • Controls that cannot be used should be visibly unavailable (disabled state), not silently ignored
  • Reversibility: wherever possible, actions should be undoable — Norman calls this the most important single design heuristic
🔍
Jakob Nielsen — Usability Engineering (1993) & Progressive Disclosure (1990–2024)
Nielsen Norman Group · 10 Heuristics for User Interface Design

Nielsen's progressive disclosure principle holds that a system should show users only the information they need at the moment, deferring advanced or secondary features until the user actively requests them. This is not simplification — it is cognitive respect. Every additional element on a screen competes for the user's limited attention; elements that are not needed now are not neutral, they are actively harmful to the user's focus on the current task.

Applied to the ATLAS workflow hub: a user on Step 2 of the Evidence Pipeline workflow does not need to see Steps 4 and 5 in full. They need to see Step 2 fully and have confident peripheral awareness that a structured path continues beyond. The left-side step list provides this peripheral awareness without forcing the user to read and process all five step descriptions before starting. This is precisely the progressive disclosure pattern — the detail is available but not imposed.

Nielsen's ten heuristics remain the most widely used evaluation framework in professional UX practice, precisely because they are concrete enough to apply to specific interface elements while remaining general enough to cover virtually any system. The heuristics most relevant to ATLAS are: (1) Visibility of system status — the user should always know where they are, what is happening, and what just happened; (6) Recognition rather than recall — the interface should show options rather than requiring the user to remember them; (8) Aesthetic and minimalist design — every element that is not serving the user's current task is competing with it; and (9) Help users recognize, diagnose, and recover from errors — error messages should be in plain language, identify the problem precisely, and suggest a solution.

Nielsen's empirical finding that five users reveal approximately 85% of usability problems (Nielsen & Landauer, 1993) is the foundation of the Phase 3 walkthrough methodology in COGS 160. You are, in effect, the five users. The structured walkthrough you conduct in Week 8 is a formal application of Nielsen's discount usability testing method.

Nielsen's 10 Heuristics — apply to every page you audit

  • H1 Visibility of system status — user knows where they are and what is happening
  • H2 Match between system and real world — speak the user's language; follow real-world conventions
  • H3 User control and freedom — provide clearly marked "emergency exits" for unwanted states
  • H4 Consistency and standards — follow platform conventions; don't make users guess whether different words mean the same thing
  • H5 Error prevention — design to prevent errors before they occur, not just to handle them gracefully
  • H6 Recognition rather than recall — make options visible; minimize reliance on user memory
  • H7 Flexibility and efficiency of use — accelerators for expert users that are invisible to novices
  • H8 Aesthetic and minimalist design — every irrelevant element reduces the relative visibility of relevant elements
  • H9 Help recognize, diagnose, and recover from errors — plain language, precise problem statement, constructive suggestion
  • H10 Help and documentation — even well-designed systems sometimes need documentation; it should be searchable and task-focused
👤
Alan Cooper — About Face: The Essentials of Interaction Design (4th ed., 2014)
Goal-Directed Design · Persona methodology · Cooper UX

Cooper's argument begins with an observation that most software is designed to satisfy the requirements document rather than to satisfy a specific person with a specific goal. The result is software that addresses every possible use case but serves no particular user well. His solution — Goal-Directed Design — begins not with features but with personas: detailed, named descriptions of specific archetypes representing the actual human beings who will use the system.

A persona is not a demographic average. It is a named individual — Cooper's canonical example is "Kara, the experienced administrator" or "Chuck, the midlevel manager" — with specific goals, specific frustrations, and specific knowledge levels. The critical design insight is that different personas have primary goals that differ enough from each other to require genuinely different interfaces, not the same interface with different labels. A student exploring ATLAS for the first time and a researcher testing a hypothesis are using the same system but with fundamentally incompatible primary goals; giving them the same entry point is a failure of Goal-Directed Design.

Cooper's analysis of the elastic user is particularly relevant to ATLAS's mode-switch system. When designers talk about "the user," they implicitly invent a fictitious person who conveniently agrees with whatever the designer wants to do. The result is an interface that satisfies no one precisely because it was designed for everyone abstractly. ATLAS's six user types (student explorer, contributor, researcher, practitioner, instructor, theory-mechanism explorer) are an attempt to resist the elastic user — to commit to specific personas and design entry points that reflect their real primary goals.

Cooper also distinguishes between primary goals (what the user is fundamentally trying to accomplish — "I want to understand what the evidence says about ceiling height"), secondary goals (what the user needs to do to accomplish the primary goal — "I need to find the right search terms"), and false goals (goals the user doesn't actually have but that the designer assumes they have — "I want to understand the system's data model"). False goals produce features no one uses. The audit exercise for ATLAS should include an assessment of which interface elements serve false goals.

Cooper's design principles for your ATLAS audit

  • Identify the primary persona for each ATLAS page — who is this page primarily designed for?
  • Assess whether the page design serves that persona's primary goal or their false goals
  • Check whether the page "design for one" — i.e., serves one use case excellently — or "design for all" — i.e., serves many use cases mediocrely
  • Evaluate whether the user types accessible via the mode-switch actually receive different enough experiences to match their different primary goals
📊
Ben Shneiderman — Designing the User Interface (6th ed., 2016, with Plaisant)
Human-Computer Interaction Lab, University of Maryland · 8 Golden Rules

Shneiderman's 8 Golden Rules are the most frequently cited design heuristics in formal HCI curricula, in part because they are stated at a level of specificity that makes them directly applicable to individual interface elements. They are not philosophical positions — they are engineering constraints.

Rule 1 — Strive for consistency is violated whenever the same action requires different inputs in different parts of the same system, or whenever the same input produces different outcomes in different contexts. In ATLAS, inconsistent link styling (some links orange, some blue, some underlined, some not) is a Rule 1 violation: the visual language is not settled enough to be reliable.

Rule 4 — Design dialogues to yield closure is the rule most directly relevant to the workflow hub design. Shneiderman argues that sequences of actions should have a beginning, middle, and end; the end should be clearly marked and should give the user a sense of completion before moving to the next task. The completion screen on the workflow hub — the green "Workflow Complete" state with a summary of what was accomplished — is a direct implementation of Rule 4. Without closure, users cannot tell whether they finished or merely stopped.

Rule 6 — Permit easy reversal of actions is violated in several ATLAS flows where submitted data cannot be retrieved or edited. Shneiderman's observation is that reversibility relieves anxiety: users who know they can undo are more willing to explore, which is exactly the exploratory behavior ATLAS wants from its student contributors. An article submitted in error should be retractable; a registration submitted with the wrong track preference should be correctable.

Rule 7 — Support internal locus of control holds that users should feel they are in control of the system, not the reverse. ATLAS currently violates this in its demonstration mode: the mode-switch behavior changes the page in ways that are not fully explained, leaving users uncertain about what they changed and how to change it back. Rule 7 violation produces learned helplessness — users who stop exploring because they no longer trust their ability to predict outcomes.

Rule 8 — Reduce short-term memory load is the rationale for keeping workflow steps separated rather than presenting them all at once. The cognitive science behind this rule is Baddeley and Hitch's (1974) working memory model, which places the capacity of human working memory at approximately 7 ± 2 items (Miller, 1956). A workflow step description that requires the user to hold seven pieces of information in mind simultaneously while also navigating to an external page will fail: some of those items will drop.

Shneiderman's 8 Golden Rules — audit checklist

  • R1 Consistency — same action, same result; same element, same appearance
  • R2 Shortcuts for expert users — keyboard shortcuts, abbreviations, macros available
  • R3 Informative feedback — every action produces a response visible to the user
  • R4 Closure — sequences have clear end states; users know when they are done
  • R5 Error prevention and simple error handling — errors are rare and recovery is easy
  • R6 Easy reversal of actions — undo is available; submissions are retractable
  • R7 Internal locus of control — user feels in charge; system responds to actions, not the reverse
  • R8 Reduce short-term memory load — don't require users to remember information from one part of the screen to use it in another
💛
Aarron Walter — Designing for Emotion (2011)
InVision · MailChimp Head of UX · A Book Apart series

Walter's central argument is that usability is necessary but not sufficient. A system can be fully functional, reliable, and usable and still produce no loyalty, no delight, and no return visits. The hierarchy he proposes — functional → reliable → usable → pleasurable — is a taxonomy of failure modes: a system that is pleasurable but not functional is a toy; a system that is usable but not pleasurable is a tool that users will abandon the moment a more engaging alternative appears.

This hierarchy has direct implications for ATLAS's audit priority. Before assessing whether ATLAS is delightful, the audit should establish whether it is functional (does it do what it says it does?) and reliable (does it do so consistently?). A beautiful interface that submits articles to a database that never processes them is pleasurable on the surface and broken underneath. The Phase 3 walkthrough should work through the hierarchy in order: identify functional failures first, then reliability failures, then usability failures, then assess pleasurability last.

Walter's concept of personality in interface design is particularly applicable to the article collector. His argument, drawing on Maslow's hierarchy of needs and cognitive psychology, is that interfaces that express personality — that feel like they were designed by people who cared about the user's experience — create emotional resonance that functional equivalents without personality cannot. The pulse animation when an article is added to the basket, the "Added to basket ✓" flash, the click sound (if implemented) — these micro-interactions are in Walter's pleasurable tier. They are not decorations; they are signals that the system noticed what the user did and cares about acknowledging it. This acknowledgment builds trust in the system's reliability at the functional level.

Walter also argues for designing for the user's emotions at the moment of use, not for the user's emotions in the abstract. A student who is frustrated because they cannot find the article they just read is in a different emotional state from a student who just submitted their fifteenth article and is feeling accomplished. The interface should respond differently to these states. ATLAS's current design treats all users as emotionally neutral — a common failure mode that Walter calls designing for a "theoretical user" rather than a real one.

Walter's principles for your ATLAS audit

  • Check functional completeness first: does every promised feature actually work end-to-end?
  • Identify any pleasurable elements (animations, confirmations, personality) and assess whether they are built on a functional/reliable foundation
  • Note any moment in your walkthrough where the system felt indifferent to your actions — these are personality gaps
  • Assess whether error states are emotionally calibrated: an error message that is technically correct but tonally dismissive is a Walter failure
📱
Luke Wroblewski — Web Form Design (2008) & Mobile First (2011)
Google Product Director · Rosenfeld Media · A Book Apart series

Wroblewski's most influential design principle is deceptively simple: one primary action per screen. This principle emerges from his extensive research on form completion and conversion rates, and it generalizes well beyond forms to any page where the user must do something. When a page presents two equally prominent actions — say, "Submit Article" and "Save as Draft" — the user's attention is divided and cognitive load increases. When one action is primary (visually dominant, clearly afforded) and the other is secondary (visually subordinate, clearly available but not demanding attention), completion rates increase significantly.

The sticky CTA at the bottom of the workflow hub pages is a direct implementation of this principle: at every moment in the workflow, there is exactly one primary action (Next Step, or Complete Workflow on the last step) and one secondary action (Previous Step). The forward action is colored in the workflow's theme color; the backward action is neutral. The user's next move is never ambiguous.

Wroblewski's work on progressive disclosure in form design is directly applicable to the ATLAS article submission form. His finding — that breaking complex forms into smaller, sequenced sections (a "one question at a time" approach) significantly increases completion rates — is the basis for arguing that ATLAS's article submission should be a multi-step wizard rather than a single long form. The cognitive cost of seeing all fields at once is high; the cognitive cost of seeing only the fields relevant to the current step is low. More fields visible simultaneously also increases the perceived effort required, which is a conversion barrier independent of the actual effort required.

Wroblewski's Mobile First methodology argues that designing for the most constrained context first — the small screen, the slow connection, the divided attention of a mobile user — produces better designs for all contexts. The discipline of fitting everything essential into a small viewport forces prioritization: if something is not essential enough to appear on a 375px-wide screen, is it essential at all? This "constraint as design tool" approach is a useful heuristic for the ATLAS audit: for each page element, ask what a mobile version would include. Elements that do not survive this exercise are candidates for removal or demotion.

Wroblewski's principles for your ATLAS audit

  • Identify the primary action on each ATLAS page — is it clear, visually dominant, and unambiguous?
  • Count the actions competing for primary attention — if more than one, flag it
  • Assess the article submission form: is it a single long form or a sequenced multi-step wizard?
  • Check each page at 375px viewport width — what survives, what breaks, what disappears?
🌉
Jared Spool — Experience Gap & UIE Research
Center Centre · User Interface Engineering · Kathy Sierra collaboration

Spool's most important contribution to UX practice is the concept of the experience gap: the distance between what a designer knows about the system and what a new user knows. Every expert who has worked on a system for months has internalized its mental model so thoroughly that they can no longer see it as a novice sees it. The system that seems obvious to the designer is opaque to the user — not because the user is deficient but because the designer has forgotten what it is like to not know. The experience gap is the fundamental obstacle to good UX, and the solution is relentless exposure to actual users who have not internalized the system.

This is precisely the methodological rationale for the Phase 3 cross-audit structure in COGS 160: you are auditing a section of the site that you did not build. Your naivety is a methodological asset, not a limitation. Spool would say that the most valuable thing you bring to the audit is the experience gap — you see things that the team who built the section cannot see because they have closed the gap themselves.

Spool's work with Kathy Sierra introduced the principle that good design makes the user feel badass — competent, skilled, and capable — not just satisfied. The distinction matters: a user who successfully completes a task on a well-designed system feels capable; a user who successfully completes the same task on a poorly designed system may feel relieved. The former generates engagement; the latter generates avoidance of future similar tasks. For ATLAS, this means that a student who successfully identifies a gap in the evidence base and articulates it clearly should feel like a real researcher, not like someone who completed a homework assignment. The interface design should support this elevated sense of competence.

Spool's experience gap also applies within the ATLAS system itself. ATLAS was designed by people who understand environmental psychology, Bayesian epistemology, and warrant theory. Its users include students who may be encountering these concepts for the first time. The experience gap between designer and user is maximal in this system — which is both its greatest design challenge and the most important audit finding to document.

Spool's principles for your ATLAS audit

  • Note every moment in your walkthrough where you had to figure something out that "should" have been obvious — these are experience-gap failures
  • Ask, for each page: does this make the user feel competent and capable, or confused and relieved?
  • Identify any terminology or concepts on the page that require prior knowledge of environmental psychology or epistemology — these are experience gaps for naive users
  • Explicitly exploit your naivety as a methodological advantage — you are seeing the gap that the builders cannot see
📲
Josh Clark — Tapworthy (2010) & Designing for Touch (2015)
Big Medium · Modern UX practice · Design systems

Clark's foundational argument is that interfaces should be designed task-first: the first thing the user sees should be what they need to do right now, not an explanation of what the site is or who built it. This sounds obvious but is systematically violated by institutional and academic systems, which tend to foreground context and identity — who built the system, what it is called, what it is for — before the user's actual task. The homepage of a research tool should not tell the user what research is; it should help them do it.

Applied to ATLAS: the workflow hub pages were designed with Clark's task-first principle as a constraint. The step panel opens directly on the task ("Choose a Topic") with the action button ("Open Topics ↗") visually dominant, before the description of the task. This inversion of the typical academic document structure — context before action — is deliberate. A student who arrives at a workflow step has already committed to doing the task; they do not need to be convinced. They need to be enabled. The page should therefore begin with the enabling action, not the justification for it.

Clark's concept of context as the primary UX variable holds that the same interface element can be appropriate or inappropriate depending on the user's context at the moment of use. A long description of a data visualization tool is appropriate when the user is learning; it is an obstacle when the user is under time pressure and already knows what they are doing. ATLAS's student-explorer workflows and researcher workflows show the same pages but at different speeds and with different emphasis — the context differs even though the destination is the same.

Clark's principles for your ATLAS audit

  • Check each page: does it lead with the task (what to do) or with context (what this is)?
  • Identify any page where the user must scroll before reaching the first actionable element — this is a task-first violation
  • Ask: what is the user's most likely context when they arrive at this page? Does the page design serve that context?
🎨
Erik Kennedy — Learn UI Design (2018–present)
Practitioner educator · Visual hierarchy in web UI

Kennedy's contribution is the systematic application of visual design principles — drawn from graphic design, typography, and cognitive psychology — to screen-based interface design. His most teachable insight is that spacing, sizing, and color are the three levers of visual hierarchy. Every decision about how prominent an element appears relative to its neighbors is a function of these three variables. Understanding this lets designers fix visual hierarchy problems systematically rather than through aesthetic intuition alone.

On spacing: Kennedy observes that most interfaces fail because elements that are logically related are not visually grouped together, and elements that are logically unrelated are not separated. The Gestalt principle of proximity holds that elements that are close together are perceived as belonging together. An ATLAS page that places the evidence quality badge five elements away from the study citation it describes is a Kennedy spacing failure: the proximity relationship is not reflected in the visual layout.

On color: Kennedy argues that color should carry semantic weight, not decorative weight. In a well-designed system, color encodes meaning (green = success, red = error, blue = interactive) so reliably that a colorblind user (and a WCAG accessibility audit) can still parse the interface correctly. In a poorly designed system, color encodes personality ("our brand is orange") and information density is reduced. ATLAS's use of the track color as a theme variable — running the workflow's color through the entire page via a CSS custom property — is a Kennedy-aligned approach to semantic color: every colored element on the workflow hub pages carries the same semantic meaning (this is your current context, this is what you are doing).

On sizing: Kennedy's typography hierarchy argument holds that a page with five different font sizes, three different font weights, and two different font families has a hierarchy problem: too many levels of hierarchy make hierarchy meaningless. ATLAS's visual audit should count the number of distinct typographic levels in use and ask whether each level is semantically necessary or whether it reflects inconsistency in the CSS.

Kennedy's principles for your ATLAS audit

  • Count distinct font sizes, weights, and colors in use on each page — flag anything above 3-4 of each as a hierarchy problem
  • Check spacing: are logically related elements visually grouped? Are unrelated elements visually separated?
  • Check color semantics: does each color carry a consistent meaning, or is the same color used for unrelated purposes?
  • Check color accessibility: does the page remain navigable without color (grayscale test)?
