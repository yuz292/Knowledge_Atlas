Which RAG System Actually Finds the Evidence?

The Complication: Same Question, Different Answers

You search five different platforms for evidence about whether biophilic design reduces stress. Consensus returns 87 relevant papers with a 72% consensus finding. Scite returns 34 papers, but with more nuanced citation context: some papers support the claim, others challenge it. Semantic Scholar returns 143 papers sorted by citation count, not relevance. OpenAlex returns everything published with those keywords, regardless of rigor.

The evidence did not change between platforms. The platforms changed what counts as evidence. This week you discover exactly where those differences emerge—not in the databases themselves, but in how they index, rank, and present research. By the end of this week, you will have tested all five platforms systematically and produced a comparative matrix that makes the invisible differences visible.

Your Usability Evaluation Report (D2, Part 1) documents this discovery. It is not a consumer review. It is a technical investigation of how platform design choices—indexing strategy, ranking algorithm, interface simplicity, result filtering—shape which research questions get answered.

What You Will Know By Friday

Which platform returns results fastest. Which one shows abstracts (and which makes you click to see them). Which one lets you export your results. What happens when a platform has zero results. How long a first-time user should expect to spend learning the interface. Most importantly: which platform would you trust to tell you that no evidence exists for a claim?

Learning Outcomes

By completing this week's work, you will:

  • Evaluate interface design empirically: Instead of describing what you see, you will measure it—time to first result, number of clicks to access an abstract, whether filtering works as expected.
  • Distinguish between coverage and quality: A platform with 300 results is not necessarily better than one with 30. You will learn when volume matters and when precision matters more.
  • Recognize algorithmic bias in ranking: All five platforms rank results differently. You will understand why: citation count, recency, semantic relevance, user engagement. Each choice encodes an assumption about what counts as "best."
  • Apply think-aloud protocol rigorously: Rather than guessing what confuses users, you will watch users search and record exactly where they hesitate, click wrong buttons, or abandon the platform.
  • Design evaluation criteria before testing: You will write the evaluation matrix this week (before testing), not after. This prevents "seeing what you want to see" in the results.

Week 4 Task Sequence (4 Days)

Monday: Evaluation Design + Account Setup

Task 1a: Design the Evaluation Matrix

Before you test anything, write the evaluation criteria. This week's deliverable (Part 1 of D2) requires a comparative platform matrix. You define the columns NOW.

Create a spreadsheet (Google Sheets shared with your team) with these columns:

  • Query: The test question (we provide 5 below)
  • Platform: Name (Consensus, Elicit, Scite, Semantic Scholar, OpenAlex)
  • Results Count: Total number of papers returned
  • Time to First Result: How many seconds until any result appears on screen? (measured with a stopwatch)
  • Top 3 Titles: The first three paper titles shown
  • Abstract Visible?: Yes/No — do you see abstracts in the main results list, or do you have to click?
  • PDF Link Accessible?: Yes/Partial/No — can you access the full text without institutional login?
  • Filtering/Export Options: What can you do with results? (e.g., export to BibTeX, filter by year, sort by relevance)
  • Confusing Aspects (User 1-5): Direct quotes from users: "I didn't know where to click," "I expected the results to be sorted by year," etc.
  • Positive Aspects (User 1-5): What worked well? Direct quotes
  • Interface Notes: Your observations (colors, layout, button labels, documentation)

Save this as a template spreadsheet. You will fill it in on Wednesday–Thursday.

Task 1b: Create Accounts & Test Logins

Sign up for free accounts on all five platforms (if required). Test that logins work. Note any barriers:

  • Does the platform require institutional email? (If so, note this—it affects who can use it.)
  • Does it require phone verification?
  • Does it have rate limits (e.g., max 50 free searches/day)?
  • Is there a user guide or tutorial?

Record setup time for each platform (in minutes).

Tuesday: Finalize Test Queries

Task 2: Select & Validate 5 Test Queries

These five queries are provided. Your team will use them on all five platforms. They should be:

  • Specific enough that a researcher would ask them
  • Broad enough that each platform will return results (not zero-hit queries)
  • Representative of the types of questions ATLAS needs to answer

Your Five Test Queries:

  1. "Does biophilic design reduce psychological stress in indoor workspaces?"
  2. "What is the evidence that natural light exposure improves cognitive task performance?"
  3. "Does exposure to ceiling height affect creative thinking ability?"
  4. "How does visual complexity in architectural design influence attention and focus?"
  5. "Does the presence of plants in indoor spaces improve air quality and occupant health?"

Before testing, each pair should predict: which platform will return the most results? Which will be most relevant? Write your predictions in the spreadsheet as a baseline.

Wednesday–Thursday: Systematic Testing

Task 3: Run All Queries on All Platforms

Divide work among your three pairs:

  • Pair 1: Consensus + Elicit (Queries 1–3)
  • Pair 2: Scite + Semantic Scholar (Queries 1–3)
  • Pair 3: OpenAlex + review/validation (Queries 1–3)

On Thursday, all pairs run Queries 4–5 on all platforms to validate the comparative pattern.

For each query-platform combination, record:

  • Results count (exact number)
  • Time to first result (measure with stopwatch or browser developer tools Network tab)
  • Top 3 titles (copy exactly)
  • Whether abstracts are visible without clicking
  • Whether PDFs are accessible (check at least one paper)

What "counts" as a result: A paper with enough information to judge relevance (title + abstract minimum). Some platforms show relevance scores; record those. Some show only titles; note this.

Task 4: Think-Aloud Protocol (5 Minutes Per Platform)

For each platform, have one team member (who hasn't used it before) run ONE of your test queries while thinking aloud. The other pair member takes notes. This is not a full user test—it is a 5-minute structured observation to catch confusing UI elements.

What to record:

  • Where does the user click first? Is it obvious?
  • Do they find the search box immediately?
  • What confuses them? (e.g., "I don't know if this is filtering or searching," "The buttons look like links," "I can't tell if my query was submitted")
  • What works well? (e.g., "The autocomplete suggestions helped," "The results loaded instantly")
  • Direct quotes: "Oh, I see, I had to click here," "This is intuitive," "I'm lost"

Ericsson & Simon (1993) call this "concurrent verbalization." The goal is NOT to judge the user; it is to reveal where the interface fails to communicate its purpose.

Friday: Synthesis

Task 5: Comparative Matrix Analysis

Aggregate all data into your spreadsheet. For each query-platform cell, calculate:

  • Average results count across queries
  • Average time to first result
  • Frequency of abstract visibility (3 out of 5 queries: yes)
  • Frequency of PDF access (1 out of 5 queries: yes, others: no/partial)

Synthesize: Write 2–3 paragraphs summarizing the patterns. Example:

  • Consensus returns the fewest results but highest confidence scores.
  • Semantic Scholar returns the most results but requires clicking to see abstracts.
  • Scite is slowest (average 8 seconds) but provides citation context that other platforms miss.
  • OpenAlex has the best structured API but no user-friendly web interface.

Frequently Asked Questions

Do all five of us have to test all five platforms?
No. Divide the work: Pair 1 handles Consensus + Elicit for queries 1–3, Pair 2 handles Scite + Semantic Scholar for queries 1–3, Pair 3 handles OpenAlex + cross-validation. On Thursday, all pairs run queries 4–5 on all platforms to validate patterns. This prevents duplication and exposes each pair to all platforms before we move to Week 5.
What if a platform returns zero results for a query?
Record "0" in the Results Count column. This is data. Then try a simpler query: just "biophilic design" without the full question. If you get results with the simplified query, note the difference. This reveals the platform's query sensitivity and is important for Week 5 (D3), when we assess platform recall on a curated benchmark.
How precisely should I measure "time to first result"?
Use your browser's Network tab (Developer Tools → Network → time to first response) or a stopwatch from the moment you hit Enter until the first paper title appears on screen. If the platform loads in sections (first 10 results, then more), record the time to see the first 10. Consistency matters more than precision; use the same method for all platforms.
One platform requires institutional login. Can we skip it?
No. Use a library card number or institutional email if you have one. If you cannot access it, note this as a major usability barrier in your "Confusing Aspects" column. The fact that a platform is inaccessible to the public is important data—it affects ATLAS's choice.
What is "think-aloud protocol" exactly? Do we need an IRB approval?
Think-aloud is a simple observational method: you watch someone use a tool while they narrate their thoughts ("I'm looking for the search box... there it is... now I'm clicking..."). You record what they say and do. For classroom evaluation like this, you do not need IRB approval—you're testing tools, not recruiting human subjects. But do ask your observer's permission to take notes and use direct quotes.

The Epistemic Question That Makes This Week Important

The question you are answering this week is not "Which platform is best?"—it is "Which platform should ATLAS trust for a given research question?"

Different platforms encode different assumptions about what counts as evidence. Consensus assumes that consensus-by-volume is a good proxy for truth. Semantic Scholar assumes that citation frequency indicates quality. Scite asks: is this paper cited as support or as a warning?

When ATLAS integrates one of these platforms, ATLAS is not just adopting a search tool. It is adopting an epistemology—a philosophy of what makes a claim credible. Your comparative matrix documents those hidden epistemological choices. In Week 5, you use this documentation to decide which platform ATLAS should trust to answer what kinds of questions.
