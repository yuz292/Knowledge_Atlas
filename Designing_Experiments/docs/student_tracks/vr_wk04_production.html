VR Track – Week 4: Production Sprint 1

Building Scenes 1, 3, and 5 of Your Stimulus Set

Context & Input/Output

Input: Your D1 Design Plan from Week 3 (stimulus dimension definition, level specifications, confound list, asset requirements)
Output: Three working Unity scenes (Scene 1, Scene 3, Scene 5) with standardized documentation and photo evidence. No formal deliverable submission this week, but TAs will check progress during scheduled check-ins.

Week 4 is about translating your design specification into executable scenes. You will operate Unity, follow your design constraints precisely, and document your work systematically. This week builds the stimulus artifact that will be validated in Week 5-6.

Learning Outcomes

  • Operate a 3D graphics engine (Unity) within design specifications
  • Import and organize assets from standardized libraries
  • Build reproducible scenes with consistent lighting, camera positioning, and environmental properties
  • Execute preliminary confound audit—identify and document sources of unintended variation
  • Document visual evidence with standardized photography protocol
  • Work methodically from specification to implementation without scope creep

Step-by-Step Procedures

1 Environment Setup in Unity

Create a new 3D project in Unity (2022 LTS or later recommended). Configure the basic environment:

  • Project Settings: Confirm target platform (PC/Mac), graphics quality tier (high-fidelity for research), and build settings
  • Lighting Profile: Import the standardized lighting profile from your Week 3 specification doc. If not yet created, define it now: three-point lighting with specific intensity values (e.g., key light 1.2, fill 0.5, back 0.8) and color temperature (5600K D65 neutral daylight)
  • Camera Setup: Create a main camera. Set position to (0, 1.7, 0) meters—eye height for an average standing adult. This becomes your standard viewpoint for all three scenes
  • File Organization: Create folders: Assets/Scenes, Assets/Models, Assets/Textures, Assets/Lighting

2 Importing Assets from the COGS160: AI & VR Asset Library

Access the shared asset library at: /Users/davidusa/REPOS/COGS160_Assets/

  • Library Structure: Assets are organized by category (furniture, outdoor, lighting, textures, characters, etc.). Locate the assets specified in your Week 3 design plan
  • Import Process: Copy models (.fbx, .obj) and textures into your project's Assets folder. Use Unity's import settings to confirm scale is correct (typically 1 unit = 1 meter in real world)
  • Missing Assets: If an asset you need is not in the library, check the library README for the request form. Submit requests to cogs160.ta@ucsd.edu with: (a) asset description, (b) reference image or 3D model file, (c) desired scale, (d) deadline. Typical turnaround: 3–5 business days
  • Naming Convention: Rename imported models descriptively, e.g., Chair_OakWood_A or Desk_Standard_1.8m

3 Building Scene 1 (Lowest Stimulus Level)

Create Scenes/Scene_01_Level_Low.unity. This scene embodies the lowest level of your target stimulus dimension.

  • Reference Your Design: Open your Week 3 D1 plan. Identify all properties that vary across levels (e.g., clutter, lighting brightness, color saturation, spatial density)
  • Environment Setup: Create a base ground plane (simple mesh or imported landscape). Apply the lighting profile. Position the main camera at (0, 1.7, 0)
  • Asset Placement: Import and position the assets appropriate to Level 1. For example, if your dimension is "visual clutter," place minimal furniture and objects. If it's "lighting," set lights to lowest brightness specification
  • Consistency Checks: Confirm all objects have consistent material properties, no floating objects, and proper collision geometry (even though this is a VR passive view, good geometry prevents rendering glitches)
  • Save: Save the scene file with version control in mind. Use meaningful naming: Scene_01_Level_Low_v01.unity

4 Building Scene 3 (Middle Stimulus Level)

Create Scenes/Scene_03_Level_Mid.unity. This represents the middle level of your stimulus dimension.

  • Proportional Adjustment: If Scene 1 has a clutter score of 2/10, Scene 3 should have ~5/10. Scale all dimension-relevant properties proportionally from your specification
  • Reuse Components: You may reuse some assets from Scene 1 (e.g., base ground, lighting rig), but adjust their properties or add new elements for the middle level
  • Version Control: Save as Scene_03_Level_Mid_v01.unity

5 Building Scene 5 (Highest Stimulus Level)

Create Scenes/Scene_05_Level_High.unity. This represents the highest level of your stimulus dimension.

  • Full Specification: Implement all elements at their maximum specified values. If Scene 1 = clutter 2/10, this = clutter ~10/10 (or as specified in your design)
  • Consistency with Precedents: Ensure the progression from Scene 1 → Scene 3 → Scene 5 follows your specification logically and consistently
  • Asset Count: Document how many unique/instanced objects you've added to each scene. Use a spreadsheet or scene description file to track this
  • Save: Save as Scene_05_Level_High_v01.unity

6 Standardized Screenshot Protocol

Document each scene with three standardized photos. These become your visual record and are used in the confound audit.

  • Camera Position: All three views use camera height 1.7m (eye level)
  • Three Fixed Positions:
    • Front View: Camera at (0, 1.7, -5), looking north (0, 0, 1)
    • Right Side View: Camera at (5, 1.7, 0), looking west (-1, 0, 0)
    • Elevated Corner View: Camera at (3, 2.5, -3), looking toward scene center
  • Lighting Consistency: Ensure all three screenshots use the same lighting setup (no time-of-day changes, no shadow adjustments between shots)
  • File Naming: Save screenshots as Scene01_FrontView.png, Scene01_RightView.png, Scene01_CornerView.png (repeat for Scenes 3 and 5)
  • Storage: Create a folder Documentation/Screenshots/ in your project. Keep originals at full resolution

7 Preliminary Confound Audit Checklist

Complete the confound audit checklist below for each scene. Document findings in a shared report.

Confound Audit Checklist

  • Intended Dimension: Does each scene clearly differ in the target dimension (e.g., clutter, brightness)?
  • Shadow Consistency: Are shadows stable across the three scenes, or do they vary in a way that could confound the stimulus?
  • Reflections & Gloss: Do material reflectivity or specularity vary unintentionally?
  • Color Saturation: Is color saturation consistent unless explicitly part of your stimulus design?
  • Object Positioning: Could the spatial layout introduce confounds (e.g., do high-clutter scenes always have objects on the right)?
  • Viewing Angle Dependencies: Does the stimulus effect vary with viewing angle in a way that confounds perception?
  • Animation or Motion: Any unintended motion in lights or objects?
  • Audio (if applicable): Is audio consistent or is it an unintended confound?
  • Asset Realism: Do assets have similar visual quality/realism across scenes, or could quality differences confound the dimension?
  • Ground Plane & Background: Are background and ground plane identical across scenes?

Using AI Assistance

💡 How AI Can Help (and What It Cannot)

AI cannot build your scenes for you—Unity requires direct interaction and fine-tuning that you must do. However, AI excels at systematic analysis:

  • Confound Analysis: Describe your scenes in text or paste screenshots into Claude/ChatGPT. Ask: "Here is my experimental design [paste spec]. Here are my three scenes [describe or paste images]. What confounds might I have introduced? Rate each by severity." AI can systematically check against your specification and identify unintended variables.
  • Asset Planning: Before starting, describe your vision. Ask AI to generate a checklist of every asset you'll need, organized by category, with suggested sources.
  • Documentation: After building, use AI to write confound audit reports, scene descriptions, and technical documentation. Provide screenshots and design specs; AI can structure findings clearly.
  • Specification Gaps: If your Week 3 spec was incomplete (e.g., you didn't quantify "clutter"), ask AI to help operationalize the dimension. Give examples; let AI suggest metrics.

Frequently Asked Questions

What if an asset I need isn't in the COGS160: AI & VR library?
Submit a request to cogs160.ta@ucsd.edu with a clear description, reference image, and deadline. For assets not available anywhere, you can use the Sketchfab or TurboSquid free libraries (with appropriate licensing) or use Unity's built-in primitives as placeholders. However, match the quality and style of library assets to maintain consistency.
How do I ensure my lighting is consistent across all three scenes?
Save your lighting setup as a Prefab in Unity. This is a reusable template that you can apply identically to each scene. Set lights to fixed positions and intensities based on your Week 3 spec. After placing lights, document their coordinates, intensity values, and color in a spreadsheet so you can recreate them exactly if needed.
What counts as a "confound" in this context?
A confound is any variable that changes across your three scenes OTHER than your intended stimulus dimension, and that could plausibly affect participant judgment. Examples: if clutter is your dimension but shadows are deeper in the high-clutter scene (more objects cast shadows), that's a confound—shadows aren't your stimulus, but they might influence perception. If lighting color shifts from warm (Scene 1) to cool (Scene 5), that's a confound. Confounds don't always ruin an experiment, but you must document them so reviewers can assess whether they're a threat.
Can I reuse the same models across scenes?
Yes! In fact, you should reuse assets (ground plane, base furniture, camera rig, lighting) to reduce unintended variation. What changes is the count and arrangement of objects that vary your stimulus dimension. For example, a simple office desk stays the same across scenes; you add or remove clutter around it to vary clutter level.
Do I need to optimize scenes for real-time rendering or VR performance yet?
Not this week. Week 4 is about fidelity to specification. Focus on accuracy of stimulus dimensions. Performance optimization happens in Week 5. However, avoid obviously inefficient practices (e.g., 1000 instances of the same model without instancing). Basic sense-checking is enough.
