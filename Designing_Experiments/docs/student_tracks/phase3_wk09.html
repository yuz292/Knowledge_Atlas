These two tasks reinforce each other: writing a success condition often reveals whether a function is actually working correctly.

Workstream 2: Writing Success Conditions

A success condition is a precise, binary statement: given a specific input or user action, what must the page or function produce for it to be considered working correctly? Success conditions serve two purposes: they document what was intended, and they provide a test protocol that anyone can run to verify the function works.

The success condition format

Add to docs/phase3_audit/section_X/success_conditions.md using the following template for each function:

## SC-A-03: image_tag_table.html — Tag browser loads and filters correctly
 
**Page**: image_tag_table.html
**Function**: Sortable tag browser with domain/from_2d/status filters and full-text search
**Precondition**: File opened in any modern browser; no server required
 
**PASS criteria**:
✓ Page loads without JS errors in browser console
✓ All 424 rows are visible in the default view (confirmed via row count in UI)
✓ Clicking "Domain" column header sorts tags alphabetically by domain
✓ Clicking "Domain" again reverses the sort order
✓ Selecting "from_2d = yes" in the filter dropdown reduces visible rows to ≤239
✓ Typing "luminance" in the search box shows only tags whose canonical_name contains "luminance"
 
**FAIL criteria**:
✗ Page loads but shows 0 rows (data array not embedded)
✗ Sort click has no effect (JS event not wired)
✗ Filter dropdown exists but does not change visible rows
 
**Status as of Week 9**: PASS / FAIL / PARTIAL (describe)

Which functions need success conditions?

Every page in your section should have at least one success condition. Prioritise:

  1. Interactive elements: any link, button, filter, sort control, or form
  2. Navigation: every nav link leads to the correct page
  3. Code examples: every terminal command in a code block produces the stated output when run
  4. External references: every referenced file, tool, or URL is accessible
  5. Content completeness: no section headed "TBD" or "coming soon" without an explicit NOTE that it is intentionally placeholder

Numbering convention

Use SC-[Section]-[NN] where Section is A, B, C, or D and NN is a two-digit sequence number (SC-A-01, SC-A-02, …). This makes it easy to reference specific conditions in the Week 10 report.
