1. Every Design Decision About the Built Environment Is Also a Claim About Perception

When an architect chooses to cover a wall in rough stone rather than smooth plaster, they are betting — usually implicitly — that the texture will register differently in the nervous systems of the people who inhabit the space. Sometimes that bet is supported by environmental psychology research. More often it is not, because the research exists but the connection between the lab finding and the design decision has never been made explicit. ATLAS exists to make those connections explicit, using a corpus of environmental images tagged with the psychological constructs that decades of research have shown to matter: biophilic richness, spatial openness, luminance uniformity, fractal complexity, and 420 others.

Your track builds the tagging layer. Specifically, you implement the 424 scientific tags as computational extractors — algorithms that take a photograph and return a number between 0 and 1 — and validate that those numbers correlate with what the tags actually claim to measure. This is harder than it sounds, and that difficulty is the intellectual content of the track. The registry lists 30 tags as phase_0 (extractable from a 2D photograph without depth data). This designation is a hypothesis. Your triage work in Week 3 tests that hypothesis. Your implementation work in Weeks 4–6 operationalizes the tags that pass. Your validation work in Week 7 assesses whether the operations actually measured what they claimed to measure.

The system needs three kinds of extraction. Deterministic extractors use OpenCV and NumPy to compute explicit mathematical functions of pixel values — fractal dimension, color entropy, edge density. VLM semantic extractors use vision-language models to recognize semantic patterns that no closed-form formula captures — whether a space feels biophilic, whether it reads as cluttered. Spatial bounding applies to both: every tag eventually returns per-region values across a 3×3 grid, because knowing that a photograph is 0.72 on fractal complexity in its upper-left quadrant is more useful for design than knowing that the whole image averages 0.58.

Learning Outcomes

  • Scientific operationalization: You will translate abstract psychological constructs into specific computational claims, and defend those translations with citations to the environmental psychology literature that originally defined the constructs.
  • Test-before-code: You will write test cases against known ground truth (a blank white image has near-zero fractal dimension; a forest photograph scores near 1.0) before you write any implementation code — and you will use failing tests to diagnose implementation errors rather than adjusting the tests to fit the code.
  • Recognition versus generation: You will distinguish between what a VLM can reliably recognize (binary presence of semantic features) and what it cannot reliably generate (calibrated continuous values correlated with psychological outcomes). This distinction is one of the most important lessons of the track.
  • Construct validity as an ongoing negotiation: You will learn that "does this tag measure what it claims to measure?" is not a question with a yes/no answer. It has an answer on a spectrum — supported, contested, provisional, deprecated — and your tag passports will document where each of your tags sits on that spectrum and why.

4. Detailed Task Breakdown

4a. Scientific Tagging: Not Just Eyeballing

The Core Challenge

You have 424 tags from the CNFA registry. Each tag represents an environmental psychology construct. The tag must have a scientific definition that you can code.

Not okay: "This room looks biophilic."

Okay: "Biophilic richness = (# of plants + # of wood surfaces + # of water features) × (% natural light / 100) × (visual complexity score, 0-1)"

Step 1: Study the Registry

Each tag in the CNFA registry has an entry with:

  • Definition in plain English
  • Expected range of values (0-1? 0-100? binary?)
  • Operationalization notes: how other researchers have measured this
  • Literature references supporting the construct
  • Example images (high and low values)

Step 2: Classify Each Tag

For every tag, decide:

  • Deterministic (Pair 1): Can be computed with math/OpenCV? Examples:
    • Window-to-wall ratio (count pixels, compute ratio)
    • Average hue or luminance
    • Edge density (how many sharp transitions?)
    • Symmetry score
  • VLM semantic (Pair 2): Requires visual understanding beyond pixels? Examples:
    • Biophilic richness (are there plants? natural materials?)
    • Color harmony (do the colors go well together?)
    • Visual clutter (is this space orderly or chaotic?)
  • Blocked: Cannot be reliably measured from still images? Examples:
    • Acoustic properties (need sound measurements)
    • Thermal comfort (need temperature sensors)
    • Smell (need olfactory sensors)

Step 3: Write Mathematical Definitions (Pair 1)

For each deterministic tag, write an operationalization document that specifies:

Tag: Spatial Openness Definition: "The degree to which a space feels expansive rather than confined." Operationalization: spatial_openness = (room_volume / person_height^3) × ceiling_height_factor × window_area_factor Where: - room_volume: estimated from vanishing points and depth cues - ceiling_height_factor: 0-1, based on pixels above eye level - window_area_factor: window pixels / wall pixels Evidence: - Kaplan & Kaplan (1989): perceived spaciousness correlates with cognitive restoration - Augustin et al. (2006): enclosed spaces increase stress hormone levels - [more citations] Implementation: - Use Canny edge detection to find room boundaries - Estimate depth using [method] - Compute factor values in OpenCV - Normalize to 0-1 scale Test cases: - cathedral (high): volume >> person scale, many windows → should yield 0.8+ - narrow hallway (low): volume ~ person scale, few windows → should yield 0.2-

Step 4: VLM Semantic Prompts (Pair 2)

For VLM-based tags, write a prompt that Claude (or GPT-4V) can use to evaluate the tag:

VLM Prompt Template

You are analyzing an interior photograph to rate "biophilic richness"—the degree to which the space contains natural or nature-inspired elements that support psychological restoration. Biophilic richness includes: - Living plants or green walls - Natural materials (wood, stone, natural fibers) - Water features (fountains, aquariums) - Natural colors (greens, earth tones, sky blue) - Natural patterns (wood grain, leaf patterns) - Views of nature or nature imagery Rate this image on a scale of 0-1: - 0: No biophilic elements; purely artificial/concrete - 0.3: One or two minor elements - 0.6: Several biophilic elements, well-integrated - 1.0: Heavily biophilic; extensive nature integration Provide your rating and a one-sentence justification.

Literature Justification: REQUIRED

For every tag you implement, you must write a justification document with this structure:

  • What the tag measures: Plain English definition of the construct
  • Why it matters for environmental psychology: What cognitive or affective outcomes does it predict?
    • Examples:
      • Biophilic richness → +0.3 SD restorativeness (Kaplan & Kaplan, 1989)
      • Spatial openness → −0.4 SD stress cortisol (Augustin et al., 2006)
  • How we operationalize it: Math formula or VLM prompt
  • Validation evidence: Has this operationalization been used before? Does it correlate with known outcomes?
  • Known limitations: When will this fail? (e.g., "outdoor photos; artistic renderings; black-and-white images")

This becomes a registry entry that future users can understand and trust.

4b. Spatial Bounding: Regional Tag Application

Why Spatial Bounding Matters

A room might have high biophilic richness in one corner (potted plants) but none in another (concrete wall). If you only store a single "average" score, you lose important information.

Better approach: Divide the image into regions and compute tag values per region.

Step 1: Spatial Regions

Divide each image into 9 regions (3×3 grid):

  • Top-left, top-center, top-right (ceiling area)
  • Middle-left, middle-center, middle-right (wall area)
  • Bottom-left, bottom-center, bottom-right (floor area)

Or use quadrants (2×2) if 3×3 is too granular. Store region boundaries as normalized coordinates (0-1 scale).

Step 2: Compute Tags Per Region

  • Deterministic tags: Crop the image to the region; compute the tag value on the cropped region
  • VLM tags: Include region info in the prompt: "Rate biophilic richness of the top-left 25% of this room"

Step 3: Spatial Indexing & Queries

Store results like this:

image_id: "IMG_00124" tags: { "biophilic_richness": { "global": 0.65, "by_region": { "top_left": 0.8, "top_center": 0.7, "top_right": 0.4, "middle_left": 0.75, ... } }, "spatial_openness": { "global": 0.55, "by_region": { ... } } }

This allows queries like: "Show me images where the ceiling is biophilic" (high value in top_* regions) or "Show me images where the left wall is very open" (high spatial_openness on *_left regions).

Step 4: Create a Tag Viewer (Pair 3)

Build a web-based viewer showing:

  • The image with regional overlay (9 colored boxes for 3×3 grid)
  • Each region shows its tag values (bar chart or heat map)
  • Ability to select a region and see all tags for that region
  • Ability to compare two images side-by-side
  • Search: "Find images where top-left > 0.7 biophilic richness"

Success Criteria for Spatial Bounding

  • Every tag has both a global score and regional scores
  • Regions are consistent across all images (same grid)
  • Viewer allows filtering and comparison by region
  • Data structure is documented and used consistently

4c. Test-Before-Code: The Known-Case Methodology

Why This Matters

Before you write the code to compute a tag, you should know what the code should output. This is the "test-before-code" principle (similar to TDD in software engineering).

The Process (for Pair 1, deterministic tags)

Week 4: Pre-implementation testing

  1. Find or create 3-5 "known" images for each tag:
    • One image where the tag value should be high (0.7-1.0)
    • One image where the tag value should be low (0.0-0.3)
    • One image where the tag value should be medium (0.4-0.6)
    • Optional: edge cases
  2. Document why each image is a known case:
    • "This cathedral has a very high spatial openness because the volume is enormous and ceilings are 100+ feet"
    • "This narrow hallway has low spatial openness: volume ~15×8×10 feet, no windows"
  3. Have the pair manually estimate the tag value for each image (without the code)

Week 5: Implementation & validation

  1. Implement the tag's mathematical formula in OpenCV/NumPy
  2. Run your code on the known-case images
  3. Does your output match your manual estimates?
    • ✓ High confidence: Yes, matches well
    • ⚠ Medium confidence: Close, but not exact (off by 0.1-0.2)
    • ✗ Low confidence: Way off; formula is wrong
  4. If it doesn't match, debug: Is the formula wrong? Did you implement it correctly? Is the manual estimate wrong?

Example Test Case Documentation

Tag: Window-to-Wall Ratio Mathematical definition: (pixel count of windows) / (pixel count of walls) Known-case images: 1. Modern office with full glass wall (expected: 0.85+) - Image: IMG_12034_glass_wall.jpg - Manual estimate: 0.88 (glass dominates; small solid sections) - Rationale: Professional photos show ~80-90% glass on one wall 2. Historic stone building with small windows (expected: 0.15-) - Image: IMG_45201_castle.jpg - Manual estimate: 0.12 (few small openings, lots of stone) - Rationale: Medieval castles have small windows for defense; windows << wall area 3. Office with standard window-per-wall (expected: 0.50±0.1) - Image: IMG_78902_office.jpg - Manual estimate: 0.48 (one large window, other walls solid) - Rationale: Typical office design has windows on perimeter, solid interior walls Results after implementing: - IMG_12034: computed 0.86 ✓ (expected 0.88, diff 0.02) - IMG_45201: computed 0.11 ✓ (expected 0.12, diff 0.01) - IMG_78902: computed 0.47 ✓ (expected 0.48, diff 0.01) Confidence: HIGH. All known cases match manual estimates within margin of error (±0.02).

Document Everything

Your D2 deliverable includes a test report with:

  • Known-case images (with manual estimates)
  • Code output (actual computed values)
  • Accuracy analysis: how close are they? What's the error margin?
  • Confidence rating per tag: HIGH / MEDIUM / LOW
  • Any tags where your formula is still uncertain

4e. Connecting Images to Experiments (The Science Bridge)

What You're Solving For

ATLAS contains hundreds of experimental studies. Each study used specific stimuli (rooms, images, VR scenes). Your job: bridge the gap between study findings and the images in your database.

The Bridge

Study → Stimulus Description → Tag Values → Image Search

Example

Study: "The Restorativeness of Biophilic Design in Office Workspaces" (Cohen et al., 2018)

  • Finding: "High-biophilic condition showed +0.4 SD improvement in attention restoration compared to low-biophilic condition"
  • Stimulus description (from paper): "The high-biophilic condition used a 3.5m × 5m room with 12 potted plants distributed throughout, natural wood flooring, live wood edge furniture, 3200K warm lighting, and a view of a green courtyard."
  • Operationalized as tag values:
    • biophilic_richness: 0.78 (many plants, natural materials, natural light)
    • luminance: 0.65 (3200K warm, not bright)
    • color_harmony: 0.82 (greens + wood tones)
    • clutter_level: 0.25 (minimalist, orderly)
  • Search query: "Find images matching these tag ranges: biophilic_richness 0.7-0.85, luminance 0.6-0.75, color_harmony 0.75-0.9, clutter 0.1-0.3"
  • Result: Database returns 15 images from your collection that match the described stimulus

Step 1: Read Through ATLAS Evidence (Pair 3)

Look at the paper bundles. For papers that describe experimental rooms or images, extract:

  • Design dimensions (size, colors, furniture, plants, light, etc.)
  • Key findings (what psychology outcomes resulted?)
  • Study populations and context

Step 2: Characterize Stimuli in Tag Terms (with Claude)

Feed the stimulus description to Claude:

Stimulus Characterization Prompt

I'm analyzing an experiment from a scientific paper. The stimulus (experimental room/image) was described as: [CITATION + STIMULUS DESCRIPTION] I need to estimate the tag values for this stimulus so I can find matching images in a database. Here are the tag definitions: - biophilic_richness (0-1): degree of natural and nature-inspired elements - spatial_openness (0-1): how expansive vs. confined the space feels - luminance (0-1): brightness and light quality - color_harmony (0-1): how well the colors work together - clutter_level (0-1): orderliness (0=very organized, 1=cluttered) - [more tags] Based on the description, estimate: 1. The likely value for each tag (0-1) 2. The confidence (HIGH/MEDIUM/LOW) 3. Any caveats (e.g., "description was vague on lighting") Format as table: | Tag | Estimate | Confidence | Notes |

Step 3: Build the Image-Experiment Bridge (Search System)

Create a web interface or API endpoint:

  • Input: Select an ATLAS study or paste a stimulus description
  • Processing: Extract or estimate tag values
  • Search: Query image database for matches (allow ±0.1 tolerance on tag values)
  • Output: Return matching images with scores showing how well they match the description

Example query in pseudo-code:

query = { "biophilic_richness": {"target": 0.75, "tolerance": 0.1}, "spatial_openness": {"target": 0.60, "tolerance": 0.1}, "luminance": {"target": 0.65, "tolerance": 0.1} } results = search_images(query) # returns images where values fall within tolerance ranges results = sorted(results, key=lambda x: distance_to_target(x, query)) # sort by match quality display(results[0:10]) # show top 10 matches

Include in your final report:

  • Examples of 5-10 studies successfully bridged to matching images
  • Stimulus characterization document (tag values for each study)
  • Working search system (or demo)
  • Notes on success/limitations: Which studies were easy to bridge? Which were hard?
Future Impact: This bridge enables a Fall VR student (or any researcher) to say "I want to recreate the restorative condition from Cohen et al. Show me images like it" and get instant visual reference material. This is a powerful tool for translating research into design practice.

4f. Validation & Confidence Ratings

What You're Documenting

Not all tags are equally reliable. Some (like window-to-wall ratio) are highly reliable. Others (like "visual clutter," which is somewhat subjective) are less reliable. Your final report must document this honestly.

Confidence Tiers

  • GOLD (High Confidence): Deterministic extractors with clean known-case validation
    • Examples: window-to-wall ratio, average hue, luminance
    • Criterion: Code output matches manual estimates within ±0.05
  • SILVER (Medium Confidence): VLM-based or semi-deterministic tags, validated on sample set
    • Examples: biophilic richness, color harmony
    • Criterion: VLM output validated on 20-30 images; moderate inter-rater agreement (if multiple people rate)
  • BRONZE (Lower Confidence): Preliminary implementations; more validation needed
    • Examples: abstract constructs with complex definitions
    • Criterion: Works on known cases, but haven't done comprehensive testing
  • BLOCKED: Not implementable from images alone
    • Examples: acoustic properties, thermal comfort, smell

Validation Report Structure (for D3)

  • Executive Summary: How many tags implemented? Breakdown by confidence tier?
  • Tier Breakdown Table: All 424 tags listed with tier assignments
  • GOLD Tier Analysis: Per-tag validation results, test images, error margins
  • SILVER Tier Analysis: VLM accuracy data, example outputs, confidence intervals
  • BRONZE Tier Analysis: What's missing? What would improve confidence?
  • BLOCKED Tags Analysis: Why are they blocked? Could they be enabled with additional data?
  • Cross-Tag Consistency: Do related tags (e.g., different measures of "biophilicity") agree?
  • Next Steps Roadmap: What should be done in summer or fall to improve tier assignments?

This report is crucial. It tells users of the image database how much to trust each tag. A researcher using GOLD-tier tags can be confident in their results. Someone using BRONZE tags should be cautious.

7. Frequently Asked Questions

❓ What if I can't implement all 424 tags? Is that a failure?
No. The goal is not to implement all 424 tags in 5 weeks. The goal is to implement a subset well, understand the challenges, and provide a roadmap for completing the rest. Implementing 40-80 tags robustly is excellent progress. Your D1 triage table will categorize all 424, so Prof. Kirsh and future students will know what's feasible vs. blocked.
❓ Should I use GPT-4V, Claude vision, or another VLM for semantic tags?
Start with Claude (you have API access through the course). In Week 4, Pair 2 will run a VLM lab comparing Claude, GPT-4V, and maybe open-source alternatives. Document costs, accuracy, and latency. Claude is likely to be good enough for most semantic tags, but the comparison will inform your choice.
❓ How do I find "known-case" images if I'm building the database from scratch?
Good question. Strategies: (1) Borrow or adapt from the sources listed in 4d. LAION, ADE20K, and Places365 all have example images you can download for testing. (2) Use AI image generation (DALL-E, Midjourney) to create synthetic test images with specific properties. E.g., "Generate a modern office with 80% glass walls and no plants." (3) Describe the ideal test image and ask Claude to find it in public Flickr/Wikipedia. (4) Take your own photos or recruit a friend to shoot images matching your test specifications. For Week 5 D2, you don't need a giant test set—5 images per tag is plenty to validate your formula.
❓ What if the VLM's rating disagrees with my manual estimate on a known-case image?
That's valuable data, not a failure. Document it: (1) Is the VLM rating reasonable? Maybe your manual estimate was wrong. (2) Is the VLM rating unreasonable? Maybe the prompt is unclear, or the VLM has a different interpretation. (3) Iterate the prompt to get better agreement. This is part of the validation process. In your D2/D3 reports, document these discrepancies and what you learned.
❓ Do I need to implement the tag viewer perfectly, or is a prototype OK for D3?
A working prototype is perfect for spring. Key features for D3: ✓ Display image with regional overlay (3×3 grid) ✓ Show tag values for each region (as numbers or bar chart) ✓ Allow region selection to see all tags for that region ✓ Can search by tag value range A polished production UI is nice, but not required. Your focus should be on validating that the regional tagging approach actually works.
❓ How do I handle images with multiple rooms or ambiguous boundaries?
Good observation. For now: (1) If the image clearly shows one room, tag the whole room (9 regions). (2) If the image shows multiple rooms or is ambiguous, you have options: a. Exclude it from the database (simplest). b. Tag only the primary visible room. c. Create multiple tag entries for the same image (one per room). Document your choices in the image metadata. This is something for a future version to handle more rigorously (maybe with instance segmentation).
❓ Can I use my phone's camera to create test images?
Absolutely. If you want to create known-case images (e.g., a photo of a room with exactly 5 windows for testing window-to-wall ratio), your phone works great. Just: (1) Document where/when you took the photo (2) Make sure the lighting and angle are representative (3) Manually estimate the tag values (ground truth) before running your code on it This is a valid and often useful approach.

Key Takeaways for Image Track Students

  • Scientific rigor is non-negotiable: Every tag must have a definition and literature support, not just intuition.
  • Test-before-code saves time: Knowing what you want before you write the code catches problems early.
  • Spatial bounding is powerful: Regional tag values unlock new research and design applications.
  • Validation is honest: Document which tags are GOLD, SILVER, and BRONZE. Users need to know what they can trust.
  • The bridge to experiments is the payoff: Connecting image databases to study results makes ATLAS useful for designers and researchers.
