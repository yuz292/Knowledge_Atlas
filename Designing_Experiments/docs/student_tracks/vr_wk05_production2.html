VR Track – Week 5: Production Sprint 2 + Preference Testing Setup

Completing Your Stimulus Set and Piloting Preference Testing

← Week 4 All Tracks Week 6 →

Context & Input/Output

Input: Three completed scenes (Scenes 1, 3, 5) from Week 4, plus your D1 Design Plan specifying all stimulus dimensions and confound controls.
Output: D2 Stimulus Set—a complete, validated set of ≥5 scenes with documented assets, pilot preference testing data, and participant instructions. This deliverable is worth 15% of your final grade.

Week 5 completes your stimulus artifact. You build the remaining scenes (Scenes 2 and 4, filling gaps in your design dimension), verify asset integrity across all scenes, set up the adaptive preference testing protocol, and conduct one pilot run with an internal participant. By Friday end-of-week, D2 is submitted and your data pipeline is validated.

Learning Outcomes

  • Complete a validated ≥5-scene stimulus set that passes the design checklist from D1
  • Verify asset integrity across all scenes (no missing textures, consistent lighting, reproducible settings)
  • Configure and deploy an adaptive paired-comparison preference testing protocol
  • Conduct a pilot run with internal participant and verify data recording works end-to-end
  • Document stimulus dimensions and asset inventory for reproducibility and replication
  • Interpret early pilot data and validate algorithmic updates during preference testing

Deliverable: D2 Stimulus Set (15% of Final Grade)

D2 is a complete, validated stimulus set submitted as a Git repository archive or shared drive folder. It must contain:

D2 Submission Requirements

  • All scene files linked or committed — Full Unity project or scene export, verifiable by TAs
  • Asset manifest — Spreadsheet listing every texture, model, prefab, and light setting with file paths and version control info
  • 15 standardized screenshots — 3 camera positions per scene × 5 scenes, PNG format, full resolution
  • Pilot preference data — One complete session recorded as CSV with all required columns (see Step 6 below)
  • Participant instructions — Plain English, ≤1 page, no jargon, covers informed consent summary + task description
  • Brief QA notes — 1-2 pages documenting confound audit results, data quality, and any issues encountered during pilot

Grading Rubric (D2: 15%)

Criterion Weight Description
Scene Completeness 30% All ≥5 scenes built, loadable, with 3 screenshots each. Proper versioning and naming.
Design Fidelity 30% Each scene matches your D1 specification: stimulus dimension operationalized correctly, confounds documented and minimal.
Asset Integrity 20% All texture files present, lighting consistent, no broken references. Asset manifest is complete and accurate.
Successful Pilot Run 20% Pilot data recorded correctly, algorithm updated between trials, instructions clear and followed by participant.

Step-by-Step Procedures

1 Build Remaining Scenes (Scenes 2 and 4)

Create the intermediate stimulus levels that fill gaps between your Week 4 scenes (1, 3, 5).

  • Scene 2 (Low-Mid): Position this scene between Scene 1 (low) and Scene 3 (mid). If your dimension is clutter on a 1–10 scale, Scene 1 = 2, Scene 3 = 5, then Scene 2 ≈ 3–4. Adjust all relevant properties proportionally.
  • Scene 4 (Mid-High): Similarly positioned between Scene 3 (mid) and Scene 5 (high). If Scene 3 = 5 and Scene 5 = 10, Scene 4 ≈ 7–8.
  • Reuse and Modify: Use the same base assets and lighting rig from earlier scenes. Only adjust the properties that vary your stimulus dimension. Maintain the same camera position (0, 1.7, 0) and lighting setup.
  • Version Control: Save as Scene_02_Level_LowMid_v01.unity and Scene_04_Level_MidHigh_v01.unity
  • Timing: ~1–2 hours per scene (faster than Week 4 because you reuse components)

2 Asset Verification Sweep

Verify that all texture files, models, and lighting assets are present and correctly referenced across all five scenes.

  • Manual Check: Open all 5 scenes in Unity. For each scene, check the Inspector for broken assets (red X icons or missing component warnings).
  • Automated Script: AI can help you write a Python or bash script to verify file existence. Example request: "I have a Unity project with scenes referencing textures in Assets/Textures/. Write a script to check if all referenced files exist and report missing ones."
  • Texture Audit: Create a folder Documentation/AssetManifest/ with a spreadsheet listing:
    • Asset filename (e.g., Desk_Oak_v02.fbx)
    • File path in project (e.g., Assets/Models/Furniture/)
    • Used in scenes (e.g., "Scenes 1, 2, 3, 5")
    • Status: OK / MISSING / DEPRECATED
  • Lighting Parameters: Document each scene's lighting: key light intensity, fill light intensity, back light intensity, light color (hex or temperature), shadow distance. Create a table for quick reference.
  • Consistency Check: Confirm lighting parameters are identical across all scenes (unless lighting variation is your stimulus dimension).

3 Cross-Scene Consistency Check

Verify that all five scenes maintain visual consistency and that stimulus variation is clear and intentional.

  • Side-by-Side Review: If possible, open all 5 scenes in a split-screen or use a multi-monitor setup to compare them simultaneously. If not, take all 15 screenshots and view them as a contact sheet.
  • Checklist:
    • Do all scenes have the same ground plane and background?
    • Does the target stimulus dimension increase/decrease smoothly and consistently across scenes?
    • Are there any unintended visual differences (e.g., color shifts, shadow differences, reflectivity changes)?
    • Are all objects properly positioned (no floating, no clipping through geometry)?
    • Is the level of detail and asset quality consistent?
  • Screenshot Comparison: Use the same three camera positions for all five scenes (Front, Right Side, Elevated Corner). Compare each position across all scenes.
  • Confound Audit (Part 2): Revisit the confound audit from Week 4. Update it now with all five scenes. Flag any confounds and document their severity (low/medium/high).

4 Write Participant Instructions

Create clear, plain-English instructions for your pilot participant. This becomes part of D2.

  • Length: ≤1 page (500–700 words)
  • Sections:
    • Study Overview: "In this study, you will view pairs of 3D virtual scenes side-by-side and indicate which one you prefer."
    • Informed Consent Summary: Acknowledge that participation is voluntary, data is anonymized, they can withdraw at any time. (Full consent form is separate; this is a summary.)
    • What They'll See: "You will see 10 pairs of scenes. Each pair appears on screen for as long as you need. You have no time limit."
    • How to Make Comparisons: "Indicate your preference by clicking a button or pressing a key (clearly specify which). You may think aloud or stay silent—either is fine."
    • No Jargon: Avoid words like "stimulus," "adaptive," "algorithm," "confound." Use everyday language.
    • Reassurance: "There are no right or wrong answers. We're interested in your honest preference."
  • Example Prompt for AI: "I'm studying preference for visual clutter in VR scenes. Write one-page participant instructions for a paired-comparison study. Include informed consent summary, what they'll see, how to respond, and reassurance. Use no jargon."

5 Configure the Preference Testing Protocol

Set up your adaptive paired-comparison testing software or script. Define the trial sequence and data recording structure.

  • Trial Sequence:
    • Familiarization (2 trials): Show two random pairs to acclimate the participant. Discard these responses from analysis.
    • Adaptive Main Trials (8–10 trials): Use an adaptive algorithm (e.g., Bradley-Terry, Thurstonian) to select pairs based on participant responses. If participant strongly prefers scenes A and B over others, the algorithm will probe boundary pairs (e.g., A vs. a medium scene).
    • Total Session Time: ~10–15 minutes
  • Algorithm Not Yet Available? If your lab's adaptive algorithm isn't ready, use a fixed sequence of all possible pairs (non-adaptive). Document this as a limitation in your QA notes. Example: "All 10 possible pairs (Scene 1 vs 2, 1 vs 3, ..., 4 vs 5) presented in random order."
  • Data Recording Structure: Configure output to write a CSV file with these columns:
    • participant_id — Unique identifier (e.g., "Pilot_001")
    • trial_num — Sequential trial number (1–10)
    • scene_A_id — Identifier of left/first scene (e.g., "Scene_01")
    • scene_B_id — Identifier of right/second scene
    • choice — Participant's choice (A or B)
    • response_time_ms — Milliseconds from stimulus onset to response
    • algorithm_state_json — JSON string with algorithm's internal state after this trial (e.g., Bradley-Terry preference scores, confidence bounds)
  • Testing Software Options: PsychoPy, OpenSesame, or a custom Python script using Pygame/Panda3D for scene rendering. Consult course resources or lab documentation for your lab's preferred tool.

6 Configure Data Recording

Set up automated logging to ensure data is recorded reliably during the pilot session.

  • CSV Output Path: Define where the data file will be saved, e.g., Data/Pilot_001.csv with a timestamp in the filename (e.g., Pilot_001_20260317_1430.csv)
  • Verify Columns: Before running the pilot, manually test the data recording. Run a practice trial and confirm the CSV file is created with all expected columns.
  • Algorithm State Logging: If you're using an adaptive algorithm, ensure its internal state (e.g., Bradley-Terry preference estimates, confidence intervals) is logged after each trial. This allows you to verify the algorithm updated correctly.
  • Backup Recording: Also save a timestamped text log with informal notes (e.g., "Trial 3: participant said 'Scene 2 felt too cluttered'"). This qualitative data enriches interpretation.
  • Test Run: Run a dry run without a real participant, stepping through all 10 trials automatically, and verify the CSV has all 10 rows with correct data.

7 Conduct Pilot Run with Internal Participant

Recruit a labmate or friend and run the full preference testing protocol. This validates the data pipeline.

  • Participant Recruitment: You (the experimenter) can be the participant, or recruit a labmate who hasn't seen your scenes before. Aim for someone unfamiliar with your hypothesis.
  • Setup: Run the protocol in a quiet space with stable lighting. Use a standard display or VR headset (depending on your stimuli format—real-time VR or rendered images).
  • Session Flow:
    • Greet participant; hand them the instructions; answer clarification questions
    • Run 2 familiarization trials (discard data)
    • Run 8–10 adaptive or fixed-sequence trials (record data)
    • Debrief: ask "Did the scenes feel different to you? How?" Jot down qualitative observations
  • Duration: ~15 minutes total
  • Data Verification: Immediately after the session, check:
    • Is the CSV file created and saved correctly?
    • Are all rows present (10 data trials + header)?
    • Do response times look reasonable (~1–5 seconds per trial)?
    • If using an adaptive algorithm, did its estimates update after each trial? (Print the algorithm_state_json column and inspect.)
  • Issues Encountered: Document any problems: software crashes, missing data columns, algorithm errors, participant confusion about instructions. Note all in your QA report.

8 Assemble D2 Submission Package

Organize all required files and commit to Git or shared drive.

  • Folder Structure:
    • D2_Stimulus_Set/ (root)
    • ├── Scenes/ (all five .unity scene files, or exported .fbx/.obj assets)
    • ├── Documentation/
    • │ ├── AssetManifest.xlsx (or .csv)
    • │ ├── Screenshots/ (15 PNG files, organized by scene)
    • │ ├── ParticipantInstructions.pdf (or .txt)
    • │ └── ConfoundAudit.md (checklist results)
    • ├── Data/
    • │ └── Pilot_001.csv (preference testing data)
    • └── QA_Report.md (brief summary of confound audit, data quality, issues)
  • Commit to Git: If your lab uses version control, commit all files with a message: git commit -m "D2 Submission: ≥5 stimulus set + pilot preference data" and push to the shared repository.
  • Alternative (Shared Drive): If not using Git, create a folder on Google Drive or institutional storage, share with your instructor, and note the link in your submission.
  • Verification Checklist:
    • All 5 scene files present and loadable
    • All 15 screenshots taken and stored
    • Asset manifest complete and accurate
    • Participant instructions ≤1 page, plain language
    • Pilot CSV data recorded with all required columns
    • Confound audit documented
    • QA notes written (1–2 pages)

Using AI Assistance

💡 How AI Can Help (and What It Cannot)

AI cannot build your scenes or run the pilot session for you. However, it excels at documentation, scripting, and systematic analysis:

  • Participant Instructions: Describe your study briefly (dimension, what they'll see, response mechanism). Ask AI: "Write clear, jargon-free participant instructions for this study..." AI generates a draft in minutes; you edit and finalize.
  • Asset Verification Script: Ask AI: "I have a Unity project with scene files in Assets/. Write a Python script to check if all texture files referenced in the scenes actually exist on disk. Report missing files." AI generates the script; you run it and interpret results.
  • QA Notes & Confound Audit Report: After your pilot session, paste your asset manifest, screenshots, and design spec into Claude. Ask: "Here's my stimulus set. Do you see any confounds I should document? Any data quality issues from my pilot CSV?" AI performs systematic analysis and generates a structured report.
  • Data Interpretation: Paste your pilot CSV and ask: "Here's my algorithm's Bradley-Terry estimates after 8 trials. Is this behavior what I'd expect? Are the preference scores converging?" AI can help validate algorithmic updates.
  • Spreadsheet/CSV Generation: Ask AI: "Generate an Excel template for an asset manifest with columns: [filename, path, scenes used, status]. Make it 5 rows of example data." AI generates a template you can fill in.

Frequently Asked Questions

We can't reach 5 scenes because building them takes too long. What do we do?
If you have only 3 scenes by mid-week, continue building Scenes 2 and 4 in parallel while also working on the preference testing setup. Scene 2 and 4 are faster than Scenes 1, 3, 5 because you reuse components. However, the minimum is 5 scenes for D2—this is a hard requirement. If you fall behind, flag this to your TA by Wednesday so they can advise on scope reduction or timeline adjustment. Do not submit with <3 scenes.
The adaptive algorithm isn't available yet. How do we set up preference testing?
Use a non-adaptive approach: generate all possible pairs of your 5 scenes (10 unique pairs if you treat AB the same as BA) and present them in random order. Document this limitation clearly in your QA notes. Your pilot will still validate the data pipeline (CSV recording, response timing, etc.). When the lab algorithm is ready, you can re-run testing with it in Week 6.
Our pilot participant showed strong preferences for one scene. Does that mean our spacing is wrong?
Not necessarily. One participant's preferences are noisy data. Strong preferences could mean: (a) your stimulus dimension is clear and effective, or (b) unintended confounds make one scene obviously better/worse. This is why you run pilot data to see the pattern, then investigate. In your QA notes, flag the pattern and hypothesize causes. In Week 6 validation, you'll collect data from multiple participants and see if the pattern holds.
What's an "asset manifest"? Do we need special software?
An asset manifest is simply a spreadsheet (Excel, Google Sheets, or CSV) listing every file your scenes reference: models, textures, lighting presets, etc. Columns: filename, file path, which scenes use it, file size (optional), status (OK / MISSING). No special software needed—Excel or a CSV file works fine. AI can help you generate a template if you ask.
Can scenes be rendered images rather than real-time VR?
Yes. If real-time VR isn't feasible, you can render high-quality stills from your scenes in Unity (File → Build & Run, or export screenshots) and display them as image pairs in your preference testing software. This is common in preference research. Document that you used rendered stills rather than real-time VR in your QA notes, and note any implications (e.g., participants can't look around; you're measuring static image preference rather than interactive VR preference).
