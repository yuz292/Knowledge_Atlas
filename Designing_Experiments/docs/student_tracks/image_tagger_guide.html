1. What the Image Tagger Is and Why It Exists

The central research question driving the Image Tagger is: can we reliably extract features from photographs of built environments that predict cognitive and affective responses? The system operationalises this question by running a battery of scientific analyzers over each image, producing a structured observation document that ATLAS can then query.

Within the COGS160 course, the Image Tagger track asks students to understand, validate, and extend this measurement infrastructure — not to build a viewer or a website, but to engage critically with the scientific pipeline that transforms raw photographs into evidence.

The Tag-UCSD/image-tagger is a full production application (version 3.4.75) that manages the complete lifecycle of image tagging: upload → automated feature extraction → VLM semantic annotation → human review → validated observation export. It is composed of a Python FastAPI backend, a PostgreSQL database, four React micro-frontend applications, and an Nginx reverse proxy, all coordinated via Docker Compose.

The application is not a toy prototype. It implements a governance layer that detects drift in critical modules, a tiered analyzer architecture that controls computational cost, and a formal output contract that specifies exactly what any consuming system (such as ATLAS) can expect from a tagged image. Students encounter it because it is the canonical source of image-level evidence for the ATLAS knowledge base.

Relationship to the CNFA Registry

The CNFA Tag Registry (Tagging_Contractor repo, registry_v0.2.8.json, 424 tags) defines what should be measured. The Image Tagger defines how to measure it — the code that actually runs analyzers and produces values. The two systems are linked by tag_id: a tag such as luminance_mean appears in the registry with its extractability and method_family fields, and in the image tagger's science pipeline as the key produced by a specific analyzer.

See the Tag Browser (424 tags) to explore the registry and understand which tags are phase_0 (measurable from 2D photographs) versus which require 3D capture or computational derivation.

2. How to Get the Repo: Fork, Clone, and Branch

The image-tagger repo is publicly hosted at github.com/Tag-UCSD/image-tagger. Students do not clone this repo directly for development — they fork it first. The distinction matters: a fork creates a copy under your own GitHub account that you control, so you can push commits freely and then submit a Pull Request (PR) to the upstream Tag-UCSD organisation when your work is ready for review.

Step-by-step: First-Time Setup

1 Fork the repository on GitHub

Go to github.com/Tag-UCSD/image-tagger in your browser. Click the Fork button in the upper-right corner. GitHub will ask which account to fork into — choose your personal account. After a few seconds you will have a copy at github.com/YOUR-USERNAME/image-tagger.

This fork is yours. Commits you push here do not affect the upstream repo until you open a PR.

2 Clone your fork to your local machine
Terminal
$ git clone https://github.com/YOUR-USERNAME/image-tagger.git
$ cd image-tagger

Replace YOUR-USERNAME with your actual GitHub username. This downloads the repository to a folder named image-tagger on your machine.

3 Add the upstream remote (important for syncing)
Terminal
$ git remote add upstream https://github.com/Tag-UCSD/image-tagger.git
$ git remote -v
# You should see:
#   origin    https://github.com/YOUR-USERNAME/image-tagger.git (fetch/push)
#   upstream  https://github.com/Tag-UCSD/image-tagger.git (fetch/push)

The upstream remote lets you pull in updates from the main repo whenever the instructors push changes:

$ git fetch upstream
$ git merge upstream/main
4 Create a feature branch for your deliverable
Terminal
$ git checkout -b image-track/D1-team-name
# Branch naming convention: image-track/[deliverable]-[team-name]
# Examples:
#   image-track/D1-mesa-group
#   image-track/D2-hillcrest-team

Always work on a feature branch, never directly on main. This keeps your changes isolated and makes it easy to open a clean PR when you're done.

5 Commit and push your work
Terminal
$ git add src/science/analyzers/my_analyzer.py
$ git commit -m "feat(D1): add luminance_variance analyzer"
$ git push origin image-track/D1-team-name
6 Open a Pull Request to the upstream repo

Go to your fork on GitHub. GitHub will show a yellow banner: "This branch is ahead of Tag-UCSD/image-tagger:main." Click Compare & pull request. In the PR form:

  • Base repository: Tag-UCSD/image-tagger | base: main
  • Head repository: YOUR-USERNAME/image-tagger | compare: your branch
  • Title: [COGS160 D1] Team name — brief description
  • Description: what you built, which tags it covers, test results

The teaching team reviews and merges. PRs that touch locked modules (see §9 Governance) will be automatically flagged.

Do not push to Tag-UCSD/image-tagger directly. Only the course staff have write access to the upstream repository. Your fork is your sandbox — all development happens there, and your work reaches the upstream only via approved PRs.

3. Two Student Tracks: Docker vs. Colab

The image-tagger supports two distinct modes of student engagement, defined in STUDENT_START_HERE.md at the root of the repository. The choice between them depends on your assignment deliverable and your local computing setup.

Track A — Full Docker Application

What it is: The complete production system running locally via Docker Compose — FastAPI backend, PostgreSQL, all four React micro-frontends, Nginx gateway.

Use when: Your deliverable requires interacting with the full application UI, querying the database, or modifying the backend science pipeline.

Requirements:

  • Docker Desktop installed (Mac/Windows/Linux)
  • ~4 GB RAM available for containers
  • ~8 GB disk space

Quick start:

$ docker compose up --build
# Access at http://localhost:3000

Persistent: Image data and annotations survive container restarts (PostgreSQL volume).

Track B — Google Colab VLM Health Lab

What it is: A self-contained Python notebook that imports the science pipeline modules and runs VLM analysis on sample images, without Docker or a local database.

Use when: Your deliverable focuses on the science pipeline, VLM integration, or analyzer validation — not the web UI.

Requirements:

  • Google account (for Colab)
  • VLM API key (Gemini or OpenAI; Anthropic optional)
  • No local installation required

Quick start: Find the Colab link in docs/ops/Student_Quickstart_v3.4.73.md → "VLM Health Lab Notebook".

Ephemeral: Results exist only in the notebook session unless you explicitly save them.

Which track is right for your COGS160 deliverable?

Deliverables D1 and D2 (registry triage and augmentation) do not require the running application — they work with the registry JSON files. Deliverables D3 and D4 (analyzer validation and implementation) are best done with Track B (Colab) or Track A with the backend running. Check your weekly assignment page for specific guidance.

All four apps communicate with a single FastAPI backend through an Nginx reverse proxy that routes requests based on URL path.

5. The Science Pipeline: Analyzers and Feature Keys

The science pipeline is the intellectual heart of the system. It is a tiered sequence of analyzer objects, each of which reads an AnalysisFrame (an input container holding the raw image and any previously computed features), runs a measurement, and writes new feature keys back into the frame. The pipeline terminates when all registered analyzers have run, producing a complete dictionary of feature values that becomes the basis for the observation JSON.

The AnalysisFrame

Every image enters the pipeline wrapped in an AnalysisFrame:

backend/science/contracts.py (simplified)
@dataclass
class AnalysisFrame:
    image_id: str            # UUID from PostgreSQL
    image_path: str          # local filesystem path to the image file
    image_array: np.ndarray  # decoded pixel array (H×W×C, uint8)
    features: dict           # accumulates computed values as pipeline runs
    metadata: dict           # exif, source URL, capture date, etc.

145 Feature Keys: Three Populations

The science pipeline currently defines 145 feature keys in total (docs/SCIENCE_TAG_MAP.md). These fall into three distinct populations, which differ in their implementation status:

Population Count Status Examples
Mathematical / Deterministic 61 Implemented luminance_mean, glcm_contrast, fractal_dimension_fd, edge_density
Stub / Planned 69 Not yet implemented — return NaN symmetry_bilateral, depth_cue_strength, material_roughness_est
VLM Semantic 14 Implemented via VLM prompt spatial_clarity, sensory_richness, symbolic_density, valence, arousal
VLM Cognitive 1 Implemented via VLM prompt attention_capture

Primitive and Composite Measurements

The 61 mathematical analyzers build up from primitive measurements to composite scores:

Measurement Family Feature Keys What It Captures
Color statistics luminance_mean, luminance_variance, saturation_mean, hue_diversity Pixel-level colour distribution in HSV/Lab space
GLCM texture glcm_contrast, glcm_dissimilarity, glcm_homogeneity, glcm_energy, glcm_correlation (×4 orientations) Grey-level co-occurrence matrix — captures surface texture regularity
Fractal dimension fractal_dimension_fd, box_count_slope, lacunarity Self-similarity across scales; related to Kaplan et al. biophilic complexity work
Spatial complexity edge_density, contour_count, region_count, clutter_index Canny edge detection and region segmentation; visual information load
Visual richness (composite) visual_richness_score Weighted combination of saturation, texture, and edge density
Organised complexity (composite) organised_complexity_score Fractal dimension × inverse clutter — high scores indicate structured rather than random complexity

Validation tables track whether each computed value falls within the expected range for the image corpus; out-of-range values trigger a warning in the Monitor app.

6. VLM Integration: Cognitive and Affective Dimensions

The VLM (Vision-Language Model) integration layer submits the image plus a structured prompt to a large multimodal model and parses the response into numerical scores. It is defined in docs/VLM_INTEGRATION.md and implemented in backend/science/vlm/.

Supported Providers

ProviderModelsNotes
Google Gemini gemini-1.5-flash, gemini-1.5-pro Default recommendation for Track B (cost-efficient, fast)
OpenAI gpt-4o-mini, gpt-4o, gpt-4.1 gpt-4o-mini recommended for cost; gpt-4o for highest accuracy
Anthropic claude-3.5-sonnet Optional; requires ANTHROPIC_API_KEY environment variable
Stub — Returns fixed values (0.5 for all dimensions); use for testing pipeline without API calls

The Ten VLM Dimensions

Each VLM call produces scores on ten dimensions (0.0–1.0 continuous scale), split into two groups:

Cognitive Dimensions (5)

KeyDefinition
attention_captureDegree to which the scene immediately draws and holds visual attention
spatial_clarityHow legibly the 3D layout of the space is communicated
sensory_richnessDensity and diversity of sensory information (visual texture, colour variety)
symbolic_densityPresence of signs, text, cultural artefacts, or other symbolic content
temporal_resonanceSense of historical depth or temporal layering conveyed by the scene

Affective Dimensions (5)

KeyDefinition
valenceHedonic quality: 1.0 = strongly positive, 0.0 = strongly negative
arousalActivation level: 1.0 = highly stimulating, 0.0 = calm/sedating
aweSense of vastness or transcendence that challenges existing schemas
tranquilityPeacefulness and restorative quality (related to Kaplan ART)
uneaseDiscomfort, threat, or anxiety evoked by the scene

Cost safety: Each VLM call costs real money. The system enforces rate limits and a daily budget cap configurable in config.py. For Track B Colab sessions, use the stub provider during development and switch to a real provider only for final validation runs.

What does "VLM reliability" mean here?

The VLM dimensions are not ground truth — they are one model's interpretation of the image. The image tagger's VLM Health Lab (Track B) is specifically designed to test inter-rater reliability across providers and across prompt variations. Deliverable D3 asks you to quantify this reliability, which is a genuine open research question: how consistently do different VLMs score the same image, and does inter-model agreement predict human rater agreement?

7. Key Contracts: The Analyzer Protocol and the Observation Schema

Two contracts govern everything students interact with in the science pipeline. Both are enforced at runtime: code that violates them will fail with a descriptive error rather than silently producing wrong output.

Contract 1: The Analyzer Protocol (contracts.py)

Every feature extractor in the system must be an instance of a class that satisfies the Analyzer protocol. The protocol is defined in contracts.py at the root of the application:

contracts.py — Analyzer Protocol (Python Protocol class)
class Analyzer(Protocol):
    """All science analyzers must implement this interface."""

    name: str
    # Unique identifier for this analyzer.
    # Must match the key used in AnalyzerRegistry.
    # Example: "glcm_texture_analyzer"

    tier: int
    # Computational tier: L0 (fastest, no API) → L5 (VLM call, expensive)
    # L0: pure math, milliseconds
    # L1: numpy/scipy, <1 second
    # L2: scikit-image, 1–5 seconds
    # L3: deep features (no VLM), 5–30 seconds
    # L4: local VLM inference
    # L5: remote VLM API call

    requires: list[str]
    # Feature keys that must already be present in frame.features
    # before this analyzer runs. Pipeline scheduling honours these
    # dependencies automatically.

    provides: list[str]
    # Feature keys this analyzer writes into frame.features.
    # Must be a subset of keys declared in the CNFA registry.

    def analyze(self, frame: AnalysisFrame) -> dict[str, Any]:
        """
        Run the analysis and return a dict mapping feature keys to values.
        Return value is merged into frame.features by the pipeline runner.
        Raise AnalyzerError (not bare Exception) on failure.
        """
        ...

Students implementing new analyzers (Deliverable D4) must satisfy all five fields. The provides list is especially important: the keys listed there must be registered in the CNFA registry, and the pipeline runner will reject any analyzer that claims to provide an unregistered key.

Contract 2: The Observation Output Schema

Every image that completes the pipeline produces an observation JSON document. The schema is defined in Tagging_Contractor/core/trs-core/v0.2.8/schemas/json_schema/observations/image_tagger_observation.schema.json. This is the document that ATLAS ingests when it asks "what features does this photograph have?"

image_tagger_observation.schema.json — Required structure
{
  "contract_version": "0.2.8",          // Must match registry version
  "image_id":        "uuid-string",      // PostgreSQL image primary key
  "run_id":          "uuid-string",      // Analysis job identifier
  "tags": [
    {
      "tag_id":    "luminance_mean",   // Must be registered in CNFA registry
      "value_type": "continuous",     // binary | ordinal | continuous | categorical | region_map
      "value":     0.734,
      "confidence": 0.98,             // 0.0–1.0; math analyzers use 1.0
      "status":    "computed",        // computed | manual | stub | error
      "evidence":  "GLCM L2-norm measurement on 256×256 patch"
    },
    ...
  ]
}

Notice the confidence field: mathematical analyzers report 1.0 (deterministic); VLM analyzers report a lower value reflecting prompt variability. The evidence field is a human-readable explanation of how the value was computed — required for all non-stub tags.

8. Database Models: What Gets Stored

The PostgreSQL database is the persistent layer of the image tagger. Its schema is defined via SQLAlchemy ORM models in backend/models/. Understanding these models matters for Deliverable D3 (validation analysis) and for any work that queries the database directly.

Model fileTable(s)Key fields
assets.py images id (UUID), source_url, filename, upload_date, metadata (JSONB), processing_status
attribute.py attributes image_id (FK), tag_id, value, value_type, confidence, status, evidence, run_id, computed_at
annotation.py annotations image_id (FK), annotator_id, tag_id, value, annotation_type (human/vlm), created_at
jobs.py analysis_jobs job_id, image_id, status (queued/running/complete/failed), started_at, finished_at, error_msg
users.py users id, username, role (admin/annotator/viewer), api_key_hash
usage.py api_usage user_id, model_provider, tokens_used, cost_usd, called_at
config.py system_config key, value, updated_at — runtime configuration (VLM provider, budget cap, rate limit)

The initial image corpus is seeded from backend/database/google_image_data.json, which defines image records with source URLs, filenames, and metadata. This gives students a working dataset from day one — no manual image upload is required.

The attributes table is the key output table

The attributes table is where computed feature values live after the pipeline runs. Each row is one (image, tag) pair. A full analysis run for one image with all 145 feature keys would ideally produce 145 rows in this table (in practice, stubs produce no row, and error tags produce an error-status row). When ATLAS asks "what is the luminance_mean of image X?", it queries this table and reads the value field.

9. Governance: The Drift Shield and Locked Modules

The governance system exists because the image tagger is collaborative infrastructure: student contributions could inadvertently break the pipeline for the entire class. Two mechanisms prevent this.

governance.lock

The file governance.lock at the repo root contains a JSON map from file paths to SHA256 hashes. Every file listed in this map is a locked module — a component so central to the system's integrity that it must not change without explicit approval:

governance.lock (example structure)
{
  "contracts.py":                     "sha256:a3f4...",
  "backend/science/contracts.py":     "sha256:b7c2...",
  "backend/database/core.py":         "sha256:d1e9...",
  "backend/database/mixins.py":       "sha256:f2a8..."
}

guardian.py

The file guardian.py is imported at FastAPI startup. It computes the SHA256 hash of every locked module and compares it against governance.lock. If any hash does not match — meaning the file has been modified — the application refuses to start and prints which file has drifted:

Terminal output when drift detected
GOVERNANCE VIOLATION: contracts.py has drifted from locked state.
Expected: sha256:a3f4...  Got: sha256:9b12...
The application cannot start. Revert the locked module or obtain admin approval
to update governance.lock.

Do not modify locked modules. If your analyzer implementation requires a change to contracts.py (for example, to add a new tier or a new field to AnalysisFrame), open a PR with a clear rationale and wait for staff approval. Modifying a locked module and committing it will cause the application to refuse to start for everyone running your branch.

10. Five-Phase Student Onboarding Sequence

The file docs/STUDENT_ONBOARDING.md defines a structured five-phase sequence for engaging with the system. The phases are cumulative: each one builds the knowledge and trust required to proceed safely to the next.

0 Read Read the "constitution": STUDENT_START_HERE.md, docs/, contracts.py. No code changes.
1 Orient Run the system (Track A or B). Browse the Explorer. Identify 5 images and record their attribute values.
2 Verify Run the full test suite. Run health checks. Compare stub vs. computed attributes for 3 images.
3 Experiment Modify only files in backend/science/analyzers/. Test your changes against the seed corpus.
4 Contribute Implement a stub analyzer. Write tests. Pass governance checks. Open a PR.

Phase 0 is not optional

The documents in docs/ are written by the system architects specifically for students. They contain design rationale, known limitations, and gotchas that are not obvious from the code alone. Reading them before touching the code is the fastest path to productive work — not the slowest.

11. Safe Contribution: The Anatomy of a New Analyzer

For students working on Deliverable D4 (implementing a stub feature key), the following template shows the correct structure for a new analyzer. This example implements luminance_variance, one of the phase_0 mathematical keys that is currently a stub.

backend/science/analyzers/luminance_variance_analyzer.py (template)
from __future__ import annotations
import numpy as np
from contracts import AnalysisFrame, AnalyzerError


class LuminanceVarianceAnalyzer:
    """
    Computes luminance_variance: the standard deviation of per-pixel
    luminance values in the Y channel of YCbCr colour space.

    Tier: L1 (numpy only, deterministic, <0.1 seconds per image)
    Provides: luminance_variance
    Requires: nothing (operates on raw image_array)
    """

    name     = "luminance_variance_analyzer"
    tier     = 1
    requires: list[str] = []
    provides: list[str] = ["luminance_variance"]

    def analyze(self, frame: AnalysisFrame) -> dict:
        try:
            img = frame.image_array  # H×W×3, uint8, RGB
            # Convert to float YCbCr; Y channel is luminance
            r, g, b = img[..., 0], img[..., 1], img[..., 2]
            y = (0.299 * r + 0.587 * g + 0.114 * b) / 255.0
            variance = float(np.var(y))
            return {
                "luminance_variance": {
                    "value":      variance,
                    "confidence": 1.0,
                    "status":     "computed",
                    "evidence":   "np.var on Y channel of YCbCr",
                }
            }
        except Exception as e:
            raise AnalyzerError(f"luminance_variance failed: {e}") from e

Then register the analyzer in the application's startup file so the pipeline runner discovers it:

backend/main.py (where analyzers are registered)
from backend.science.analyzers.luminance_variance_analyzer import LuminanceVarianceAnalyzer
# ... after other imports ...
registry.register(LuminanceVarianceAnalyzer())

Required Tests for a New Analyzer

Every new analyzer must ship with at least three tests:

  1. Registration test: the analyzer is discoverable by name in the AnalyzerRegistry
  2. Output schema test: analyze() returns a dict with all keys in provides, each with value, confidence, status, and evidence sub-fields
  3. Range test: the returned value falls within the expected range for the tag (as specified in the CNFA registry's semantics.range field)
backend/tests/test_luminance_variance_analyzer.py (template)
import numpy as np
import pytest
from backend.science.analyzers.luminance_variance_analyzer import LuminanceVarianceAnalyzer
from contracts import AnalysisFrame

def make_frame(h=64, w=64):
    img = np.random.randint(0, 255, (h, w, 3), dtype=np.uint8)
    return AnalysisFrame(image_id="test", image_path="", image_array=img, features={}, metadata={})

def test_provides_correct_keys():
    result = LuminanceVarianceAnalyzer().analyze(make_frame())
    assert "luminance_variance" in result

def test_output_schema():
    entry = LuminanceVarianceAnalyzer().analyze(make_frame())["luminance_variance"]
    for field in ("value", "confidence", "status", "evidence"):
        assert field in entry, f"Missing field: {field}"

def test_value_in_range():
    result = LuminanceVarianceAnalyzer().analyze(make_frame())
    v = result["luminance_variance"]["value"]
    assert 0.0 <= v <= 0.25, f"Variance {v} out of expected range [0, 0.25]"

Before opening your PR: run python3 -m pytest backend/tests/ -x -q and confirm all tests pass. Confirm python3 guardian.py --check reports no governance violations. Include the test output in your PR description.
