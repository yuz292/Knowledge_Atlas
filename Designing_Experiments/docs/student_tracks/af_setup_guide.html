Step 0 — Prerequisites: What You Need First

0a
A GitHub account

Create one at github.com if you don't have one. Use your UCSD email address so the course organization can find you. Once you have an account, email your GitHub username to the course instructor so you can be added to the ucsd-cogs160 organization with write access to the repositories.

Do not start the clone steps until you have received an invitation email from GitHub and accepted it.

0b
Git installed on your computer

Check whether Git is installed:

$ git --version git version 2.39.2 ← any version ≥ 2.30 is fine

If Git is not installed: on macOS, running git in Terminal will prompt you to install the Xcode Command Line Tools — click Install. On Windows, download Git from git-scm.com. On Linux (Ubuntu/Debian): sudo apt install git.

0c
Python 3.10 or higher

The Article_Eater codebase uses Python 3.10+ features (match statements, structural pattern matching). Check your version:

$ python3 --version Python 3.11.4 ← 3.10 or higher required

If you have Python 3.9 or lower: on macOS, install via Homebrew (brew install python@3.11). On Windows, download from python.org. On Linux: sudo apt install python3.11 python3.11-venv.

macOS note: The system Python on macOS is often 3.9. The Homebrew Python is at /opt/homebrew/bin/python3. You may need to use that path explicitly if python3 resolves to the system version.

Step 1 — Clone the Repositories

1
Clone Article_Eater (AF Track and Image Tagger Track)

Choose a location on your computer for course repos — for example, ~/Documents/COGS160/. Then:

# Create a workspace folder $ mkdir -p ~/Documents/COGS160 && cd ~/Documents/COGS160 # Clone Article_Eater (the main AF pipeline) $ git clone https://github.com/ucsd-cogs160/Article_Eater_PostQuinean_v1_recovery.git Cloning into 'Article_Eater_PostQuinean_v1_recovery'... # Clone Designing_Experiments (course materials, Streamlit app, registry) $ git clone https://github.com/ucsd-cogs160/Designing_Experiments.git Cloning into 'Designing_Experiments'...
Wrong branch? Article_Eater has multiple branches. The active branch is codex/recovery-cc-migration-artifacts. After cloning, confirm you are on it:
$ cd Article_Eater_PostQuinean_v1_recovery $ git branch --show-current codex/recovery-cc-migration-artifacts
If you see main or any other branch, switch: git checkout codex/recovery-cc-migration-artifacts

Step 2 — Set Up the Python Environment

2a
Create a virtual environment

Always use a virtual environment. It isolates the packages you install for this project from your system Python, preventing version conflicts with other software.

# From inside Article_Eater_PostQuinean_v1_recovery/ $ python3 -m venv .venv # Activate (macOS / Linux) $ source .venv/bin/activate (.venv) $ ← the prefix confirms activation # Activate (Windows PowerShell) PS> .venv\Scripts\Activate.ps1

You must activate the virtual environment every time you open a new terminal window before running any project code. Add a note somewhere visible — many students forget this step after the first week.

2b
Install dependencies
# Upgrade pip first (.venv) $ pip install --upgrade pip # Install all project dependencies (.venv) $ pip install -r requirements.txt Successfully installed ... (this takes 1–3 minutes) # If you see errors about specific packages, try: (.venv) $ pip install -r requirements.txt --ignore-requires-python
Optional packages. Two dependencies are optional and may fail to install on some systems without C compiler tools: sentence-transformers (semantic similarity) and scikit-learn (some analysis scripts). They are not required for D1–D3. If installation fails, skip them with: pip install -r requirements.txt --ignore-requires-python 2>&1 | grep -v "sentence-transformers\|scikit"
2c
Verify the environment
# Run a quick import check (.venv) $ python3 -c "from src.services.voi_search import VOISearchCoordinator; print('OK')" OK # Run the test suite (expect some pre-existing failures) (.venv) $ python3 -m pytest tests/ -x -q \ --ignore=tests/test_figure_suggestion_service.py \ -k "not test_follow_ups_enabled" ...................................... (some dots, then summary) X passed, Y warnings ← any result with fewer than 5 failures is fine
Success looks like: More than 80 tests pass and fewer than 5 fail. The two pre-existing broken test files are test_figure_suggestion_service.py and any test with test_follow_ups_enabled in its name — skip those with the flags above.

Step 3 — GitHub Workflow for Submitting Work

All team deliverables live in the repositories. You do not submit through Canvas — you submit by opening a pull request. Here is the complete workflow from starting a new assignment to getting it reviewed.

3a
Create a feature branch for each deliverable

Never commit directly to main or the active recovery branch. For each deliverable, create a named branch:

# Convention: [track]-[deliverable]-[team-name] $ git checkout -b af-d1-team-gecko Switched to a new branch 'af-d1-team-gecko' # Or for Image Tagger: $ git checkout -b image-d1a-team-gecko
3b
Commit regularly

Commit whenever you reach a natural stopping point — after writing a section, after a test passes, after fixing a bug. Small commits with clear messages are better than large commits that bundle unrelated changes.

# Stage specific files (never use git add . for the first commit) $ git add reports/D1_Pipeline_Orientation.md scripts/gap_trace.py # Commit with a message that says what changed and why $ git commit -m "AF D1: add end-to-end query trace for natural light question Traces query through QueryParser → QueryEngine → GapDetector. Shows how a VALIDATION gap is generated when credence < 0.45. Table of top-5 gaps from DB inspection added to report."
3c
Push to GitHub
# Push your branch to GitHub (first time) $ git push -u origin af-d1-team-gecko Branch 'af-d1-team-gecko' set up to track 'origin/af-d1-team-gecko'. # Subsequent pushes (shorter) $ git push
3d
Open a pull request

When your deliverable is ready for review, go to the repository page on GitHub and click Compare & pull request (the yellow banner that appears after you push). Set:

  • Base branch: codex/recovery-cc-migration-artifacts (not main)
  • Compare branch: your feature branch
  • Title: [AF] D1 Pipeline Orientation Report — Team Gecko
  • Description: One paragraph saying what the deliverable covers and any known limitations or questions for the reviewer

Add the course instructor as a reviewer. The pull request URL is what you submit to the course gradebook.

3e
Tag each deliverable after it is graded

After your PR is merged and graded, tag the commit so you can find it later:

$ git tag -a v0.1 -m "D1: Pipeline Orientation Report — graded" $ git push origin v0.1

Convention: v0.1 = D1, v0.2 = D2, v0.3 = D3, v0.4 = D4, v0.5 = D5, v1.0 = final report.

Registry Access — Image Tagger Track

The Image Tagger track works with the CNFA tag registry — the list of 424 environmental tags that ATLAS uses to describe photographs. The registry file lives in the Tagging_Contractor repository, which is a separate clone.

R1
Clone Tagging_Contractor
$ cd ~/Documents/COGS160 $ git clone https://github.com/ucsd-cogs160/Tagging_Contractor.git
R2
Locate the registry file

The canonical registry is at:

Tagging_Contractor/core/trs-core/v0.2.8/registry/registry_v0.2.8.json

This is a JSON dictionary of 424 tags. Each tag key is a dotted identifier like spatial.openness or luminous.daylight_availability. The Registry Tag Browser shows all 424 tags with sortable columns — use it to find tags quickly without opening the JSON directly.

R3
Open the registry in Python
# Load the registry (.venv) $ python3 >>> import json >>> with open('Tagging_Contractor/core/trs-core/v0.2.8/registry/registry_v0.2.8.json') as f: ... reg = json.load(f) >>> tags = reg['tags'] >>> print(len(tags)) 424 >>> print(list(tags.keys())[:5]) ['affect.cozy', 'affect.jarring', 'affect.scary', 'affect.sublime', 'affect.uncanny']
R4
Submit registry edits via pull request

In D1b (registry augmentation), you add the bilingual block to 5–7 tags. Do this on a feature branch in the Tagging_Contractor repo, following the same workflow described in Step 3 above. The branch naming convention for the Image Tagger track is image-[deliverable]-[team-name].

Do not edit registry_v0.2.8.json directly on the main branch. Changes must go through pull request review so the course instructors can validate the bilingual field format before they are merged. Merging malformed JSON will break the tag browser for all students.

Common Errors and Fixes

E1
ModuleNotFoundError: No module named 'src'

You are running Python from the wrong directory. Always run commands from the root of the cloned repository (the folder that contains src/, tests/, and requirements.txt). Check your current directory with pwd and navigate there with cd.

E2
fatal: repository not found on git clone

You have not accepted the GitHub organization invitation, or your account is not yet in the organization. Check your email for the invitation from github.com. If you accepted more than 24 hours ago and still get this error, email the course instructor with your GitHub username.

E3
sqlite3.OperationalError: no such table: beliefs

The Web of Belief database has not been seeded yet. Run the seeding script: python3 scripts/seed_web.py --sample. If that script doesn't exist in your version, ask on the course Slack — the database seed files are shared there.

E4
Tests fail with ImportError: cannot import name 'X' from 'src.services.voi_search'

You may be on the wrong branch. Run git branch --show-current. If you are not on codex/recovery-cc-migration-artifacts, switch to it: git checkout codex/recovery-cc-migration-artifacts. Then reinstall dependencies: pip install -r requirements.txt.

E5
On macOS: pip: command not found

Use pip3 instead of pip. On macOS with Homebrew Python, the correct commands are python3 and pip3. If you have activated a virtual environment (Step 2a), pip and python (without the 3) should work inside the environment.
