The Complication: No Single Platform Covers the Literature

Your D3 measurements showed it clearly: Semantic Scholar finds 79% of papers on "natural light → attention" but only 43% on "computational modeling of restoration." Consensus returns high-precision results but misses older foundational work. Scite shows citations in context but doesn't index design literature. No platform excels at everything.

The question is no longer "Which platform should ATLAS use?" The answer is "All of them, strategically." This week moves from measurement to design. You will architect a VOI-to-Query Bridge: a system that takes a belief gap from ATLAS ("insufficient evidence for attention recovery from natural light in older adults") and translates it into optimized, platform-specific queries, executes them across multiple platforms intelligently, and returns a deduplicated, ranked candidate list that ATLAS integrates into its evidence corpus.

This bridge is not a finished product. It is a technical specification—a blueprint for Week 8's implementation. A good spec makes implementation straightforward. A vague spec means Week 8 bogs down in ambiguity. Your job: make your choice concrete and defensible.

What You Will Deliver

A 10–12 page technical proposal that a developer (Week 8 student or external engineer) can read and implement. It includes: input schema, query generation logic per gap type, platform selection rules based on D3 data, deduplication strategy, relevance filtering, output schema, three architectural trade-offs with explicit choices, and a one-page implementation roadmap for D5.

Learning Outcomes

By completing this week's work, you will:

  • Reverse-engineer query generation: You will examine the QueryGenerator class in the codebase and understand how it maps gap types (VALIDATION, MECHANISM, DIRECTION, BOUNDARY) to search strategies. Then decide whether to extend it or replace it.
  • Design for heterogeneous systems: Each platform has different API contracts, ranking algorithms, and indexing strategies. Your bridge must abstract over those differences—mapping a single logical query to platform-specific syntax without losing meaning.
  • Make architectural trade-offs concrete: Every design choice involves a trade-off (precision vs. recall, latency vs. completeness, cost vs. quality). You will identify three key trade-offs, enumerate options, and explain your choice with data.
  • Connect evaluation to implementation: Your D3 data should drive decisions. Example: "Semantic Scholar is platform choice for neuroscience VOIs because D3 shows 85% recall on this domain, 15% higher than alternatives."
  • Specify system contracts: You will define input and output schemas precisely—what a gap looks like, what a candidate paper looks like—so another team can build against these contracts without ambiguity.

The VOI-to-Query Bridge: Conceptual Architecture

What Is a Bridge?

A bridge takes a gap description from ATLAS and produces a ranked list of candidate papers from multiple platforms. It is a translation layer:

  • Input: A gap object from GapDetector (class: GapType, belief statement, domain, confidence estimate)
  • Processing: (1) Translate gap to platform-agnostic query intent, (2) Generate platform-specific queries, (3) Select which platforms to query, (4) Execute queries in parallel, (5) Deduplicate results, (6) Rank by relevance, (7) Filter by quality threshold
  • Output: A ranked list of candidate papers with metadata and provenance (which platform found it, confidence score)

Why Not Just Query One Platform?

D3 showed you why: no single platform is comprehensive. But the bridge isn't just running queries on all five platforms. It is:

  • Platform-smart: Don't query Consensus for pure methodology VOIs (Consensus synthesizes findings, not methods). Do query it for clinical questions.
  • Cost-aware: Scite and Elicit have metered APIs. Budget them for high-impact VOIs, use free platforms for exploratory gaps.
  • Latency-conscious: Run fast platforms (Semantic Scholar, OpenAlex) in parallel; run slower platforms (Elicit) only if needed.
  • Deduplication-rigorous: A paper found by two platforms is still one paper. Merge results, aggregate platform confidence scores.

Gap Types: Different Gaps, Different Queries

The QueryGenerator class (which you will examine) recognizes that different gap types need different query strategies:

Gap Type Query Focus Best Platform(s) Example Query
VALIDATION Does the claimed effect exist? Primary evidence Consensus, Semantic Scholar "natural light attention cognition effects"
MECHANISM Why does it work? Causal pathways Semantic Scholar, OpenAlex (broad indexing) "attention recovery circadian melatonin light exposure mechanism"
DIRECTION Are effects consistent? Contradictions/moderators Scite (shows supporting vs. disputing citations) "natural light attention contradictions boundary conditions"
BOUNDARY When does it fail? Limits/exceptions Semantic Scholar, OpenAlex (design + implementation literature) "natural light attention failures limitations context-dependent"

Thursday: Identify & Compare Architectural Trade-Offs

Task 6: Three Design Alternatives with Explicit Trade-Offs

Every system design involves hard choices. Identify three key architectural decisions. For each, enumerate at least two alternatives and recommend one. Examples:

Trade-Off 1: Query Execution Strategy

  • Alternative A — Sequential (Platform 1, then Platform 2, etc.): Simplifies code, reduces API contention, but slow (total time = sum of individual platform latencies). Good for low-frequency searches; bad for interactive use.
  • Alternative B — Parallel (all platforms simultaneously): Fast (total time = slowest platform latency). But requires async code, error handling if one platform fails, and careful API rate-limit management.
  • Alternative C — Smart Sequential (fast platforms in parallel, slow platforms conditional): Run Semantic Scholar + OpenAlex in parallel (free, fast). If recall is <60%, query Consensus/Scite next. Balances speed and quality.
  • Recommendation: Alternative C. Justification: D3 shows Semantic Scholar + OpenAlex achieve 70–80% average recall; only 30% of searches need additional platforms. This saves API costs and keeps latency <3 seconds for 85% of queries.

Trade-Off 2: Query Language Strategy

  • Alternative A — Universal Query Language: Translate all gap-to-query logic into a single format (e.g., "concept:X AND concept:Y AND (outcome:Z OR outcome:W)"). Each platform adapter converts to its native syntax. Scales well as you add platforms.
  • Alternative B — Platform-Specific Generation: Each gap type has platform-specific query code. Semantic Scholar gets one query, Scite gets a different query tuned to its ranking. Harder to maintain but potentially higher quality results.
  • Alternative C — Hybrid: Use universal format for the query intent, let each platform adapter optionally enhance with platform-specific optimizations (e.g., "if Semantic Scholar, add fielded search; if Consensus, add synthesis filter").
  • Recommendation: Alternative C. Justification: Universal format keeps the architecture clean, but platform adapters can exploit platform strengths. This allows you to improve results incrementally without refactoring the entire pipeline.

Trade-Off 3: Deduplication & Merging

  • Alternative A — Exact Matching Only (DOI or exact title): Safe but misses papers with missing/incorrect metadata. May show the same paper twice if one platform uses slightly different title.
  • Alternative B — Fuzzy Matching (title similarity >80%): Catches more duplicates, but risk of false merges (e.g., two different papers on the same topic with similar titles).
  • Alternative C — Smart Merging (exact match + fuzzy match on author+year): If DOI matches, merge. If title 80%+ similar AND authors 60%+ similar AND year within 1 year, merge. Otherwise, keep separate.
  • Recommendation: Alternative C. Justification: Reduces false positives while catching genuine duplicates. Cost of implementing fuzzy match is low; benefit of showing users one paper instead of three is high.

Your proposal must include three such trade-offs with explicit recommendations and brief justifications. They should be meaningful (not trivial) and grounded in data from D3 or system requirements from earlier weeks.

Frequently Asked Questions

Should my bridge be a replacement for IntegratedQueryService, or an enhancement?
Good question. If IntegratedQueryService already does most of what you need, enhance it (clarify what you're adding, why it needs enhancement). If it doesn't fit your architecture, replace it and explain why the new design is better. Either way, your proposal should make the choice explicit and justify it.
Can I propose platforms beyond the five we tested in D2/D3?
Yes, but you lose the D3 data that justifies platform choice. If you propose adding Google Scholar API or PubMed, explain why it's necessary (e.g., "Biomedical literature is indexed better in PubMed than in Semantic Scholar; D3 showed 40% recall for biomedical VOIs across all five platforms") and acknowledge the implementation cost (learning a new API).
How detailed should my query generation logic be?
Detailed enough that a developer can implement it. You don't need to write Python code, but pseudocode or English-language algorithms should be precise. Example: "For VALIDATION gaps, expand {main_term} with synonyms using WordNet, then combine with {population} using AND, then filter results to year >= [current_year - 10]." Not: "generate smart queries."
Do I need to propose cost estimates for platform APIs?
Yes. Get current pricing (Scite, Elicit, Consensus all have metered APIs). If your system will run 20 queries per session, estimate monthly cost. This informs your platform selection trade-offs (maybe Scite is great, but it's $0.50/query—budget limits its use).
The epsilon-greedy question feels hard. What's a good answer?
Engage thoughtfully. Epsilon-greedy works well in environments where you have many samples and can learn over time. In your case, you have ~200 total queries over the semester with budget constraints—fewer samples. Consider: "Epsilon-greedy is good for learning; but for a bounded student project, a simpler heuristic (platform selection based on D3 data) is more efficient. We recommend epsilon-greedy for future years when ATLAS runs continuously, but use deterministic platform choice for D5."
