Week 6 — Deliverable D4

Personalization & AI Help System Proposal

Shifting from topic-based to context-sensitive, adaptive help

Due: Friday, Week 6 (11:59 PM) Weight: 11% of GUI track Team: Pair 2 Deliverable Type: Proposal + Prototype

Intellectual Challenge

Most help systems are designed as if all users have identical questions at identical moments. Real users don't. An enrolled biology student asking "How do I compare species using this tool?" needs different help than a guest scientist asking "What datasets are available?" at the same interface location. The intellectual core of this deliverable is recognizing that help must be context-sensitive — determined not just by what button the user pressed, but by who they are, what task they are performing, and where they stand in their task progression.

The Design Failure Behind the Question Mark. The help button on most software is a last resort — a button users press after they have already given up trying to figure something out themselves. This is a design failure, not a user failure. Help that arrives after the user has given up serves only as confirmation that the task was theoretically possible. Help that arrives at the moment of decision — before the user commits to the wrong action, at the precise checkpoint where confusion is most likely — changes outcomes. Your task is to design a help system that knows when decision points arrive.

Core Design Question

You have two weeks of empirical data from D1 (user type profiles and checkpoint moments of confusion) and D3 (expert panel assessment of AI capability). The question now becomes: How can we deploy adaptive help that intercepts confusion before it becomes failure?

This requires three architectural decisions: (1) what triggers help to appear (the detection layer), (2) who gets which help message (the personalization layer), and (3) what the message says (the content layer). Most help systems skip (1) and (2) entirely — they assume a generic user at a generic moment — and focus only on (3).

What You Must Deliver

1
Usability Issues Matrix (from D1 & D3)

Create a ranked table of the top 5–8 usability problems you documented in D1, augmented with the expert assessments from D3. For each issue, specify: (a) which user type(s) encounter it, (b) at what checkpoint moment, (c) estimated severity (frequency × impact), and (d) whether AI-assisted help could mitigate it. Format as a table with explicit severity scoring (e.g., High: affects 50%+ of a user type at a critical decision point).

2
Context-Sensitive Help Architecture

Design and document the triggering logic for your help system. Specify: (a) what observable user behaviors or interface states should trigger help (e.g., "cursor hovers on VOI query input for >3 seconds without typing" or "user clicks 'submit' with invalid parameters"), (b) how your system will identify the user type (from session metadata, prior interactions, or explicit self-identification), and (c) how message content will be selected based on user type × checkpoint combination. Diagram this as a decision tree or flowchart.

3
Personalization Rules (Content Mapping)

For each of your top 5–8 issues, write three versions of the help message: (a) generic (one-size-fits-all, topic-based), (b) user-type-sensitive (tailored to one specific user type's mental model), and (c) context-sensitive (tailored to the specific checkpoint moment AND the user type). Document these in a structured table showing before/after comparisons. For example, help for the "VOI query" checkpoint should differ for an enrolled biology student (task-focused: "to compare two species, enter their names in...") vs. a guest scientist (system-focused: "VOI queries compare value-of-information scores; to run a query, use this syntax...").

4
Interactive Prototype (HTML/JS)

Build a clickable mockup showing your help system in action at one specific checkpoint. Choose the highest-severity usability issue you identified in step 1. Create an HTML page showing: (a) the interface state at that checkpoint (you may use wireframe graphics or screenshots from D1), (b) the triggering condition that causes help to appear (e.g., "cursor dwell on input"), (c) the personalized help message appearing as an overlay or sidebar panel, and (d) documentation of what would happen if the user were a different user type (show alternative message in comments or a toggle). Include CSS animations or subtle interactions to make the help feel integrated, not intrusive.

5

This becomes the input for Pair 3's UI evaluation in Week 7. Include at least one complete HTML or UI component that could be integrated into the live system. Document the implementation in brief comments so Pair 3 understands what they are evaluating.

6
Handoff Documentation for Pair 3

Write a 1–2 page summary document explaining: (a) which help improvements were implemented, (b) which checkpoint moments and user types they address, and (c) what specific UI evaluation questions Pair 3 should answer (e.g., "Did the proactive help reduce the frequency of invalid VOI queries?" or "Did users notice the help message, and did they find it relevant?").

Design Principles from Research

Your help system should reflect principles from human-computer interaction research. Norman (2013) distinguished between two barriers to usability: the gulf of execution (difficulty expressing intentions) and the gulf of evaluation (difficulty interpreting feedback). Context-sensitive help targets the gulf of execution by making the correct action obvious at the moment of decision. Card, Moran, and Newell's (1983) GOMS model emphasizes that expert users and novices have fundamentally different task decompositions for the same interface action. Your help system should recognize this: a routine task for one user type is a novel problem for another.

Fischer (2001) noted that user modeling in human-computer interaction succeeds only when the system makes assumptions about the user visible and revisable. If your system decides "this user is a student," the interface should make that assumption transparent so the user can correct it if the system is wrong. Horvitz (1999) developed design principles for mixed-initiative UI — systems that combine human and computer agency — and emphasized that proactive help works only when its cost (annoyance, interruption) is lower than its benefit (clarification, error prevention). Help that appears too often becomes invisible; help that appears at the exact moment of decision is noticed and remembered.

Worked Example: The VOI Query Checkpoint

WORKED EXAMPLE Context-Sensitive Help at the VOI Query Input

The Checkpoint Moment

From your D1 research, you identified that the VOI query panel is a moment of confusion for several user types. New students don't understand what "VOI" means. Guest scientists understand VOI but don't know how it's implemented in this system. Let's design help for these two moments.

Generic (Topic-Based) Help

Help: Value of Information (VOI)
Value of information (VOI) is a decision-theoretic measure of the expected utility of acquiring information before making a decision. To run a VOI query, enter two species names or two environmental conditions in the input fields. Click "Submit" to calculate their information values.

Problem: This explanation assumes the user already knows what "utility" means and why they would want "information value." It treats the user as a passive recipient of a definition rather than someone trying to accomplish a goal.

User-Type-Sensitive Help (Student)

Comparing which species would teach you more?
This tool helps you pick between two species to study by calculating which one would answer more of your open questions. Enter the two species you're deciding between, and this tool will score them on how much new information comparing them would give you. Higher score = more learning opportunity.

Improvement: Grounded in the student's actual task (deciding what to study), uses their vocabulary ("learn," "discover"), and explains the outcome they care about (making a better decision about what to research).

Context-Sensitive Help (Student + Data Entry Trigger)

🎯 Tip for this step:
You're about to compare two species to see which study would teach you more. This tool calculates how much new information you'd gain from each comparison.

How to use it: Type any two species names (e.g., "wolf, moose"), and we'll score which comparison would answer more of your research questions. Try it with organisms you're actually comparing for your project.

Improvement: Appears at the exact moment the user is about to enter data (after hovering/focusing on the input field). Uses first-person language ("you're about to"). Connects the action to the user's real project. Reduces cognitive load by telling them exactly what to type.

Triggering Logic

if (user_type == "student"
    && interface_location == "voi_query_input"
    && user_action == "focus") {
  show_help_message("context_sensitive_student_voi");
  set_display_duration(10_seconds_or_until_dismissed);
}

Evaluation Criteria

Your deliverable will be assessed on: (1) the quality of your usability diagnosis (how well you identified the real problems from D1), (2) the coherence of your help architecture (do the triggering rules, personalization rules, and message content all align?), (3) the realism of your prototype (does it show genuine understanding of how the interface works?), and (4) the sophistication of your personalization (have you genuinely differentiated help for different user types, or just written different versions of the same message?).

A weak deliverable treats help as content variation: writing different words for different users. A strong deliverable treats help as architecture: designing when help should trigger, who should receive it, and why those decisions matter for outcomes.
