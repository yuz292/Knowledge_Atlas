Section 1: Track Overview

What Is the AF Track?

The Article Finder (AF) Track is a 6-week research engineering project that tracks down and retrieves relevant scientific articles to expand the ATLAS research corpus. Your team (6 students, 3 pairs) will build VOI-driven search queries, retrieve articles via RAG platforms and open databases, and create the infrastructure for researchers to contribute their own PDFs. At the end of the 6 weeks, you'll have built a functional component of the ATLAS/Knowledge Accelerator system that enables researchers to discover and contribute evidence-backed findings.

How AF Contributes to ATLAS/KA

The broader ATLAS project aims to build a coherence-based knowledge system for computational epistemology. The Article Finder track addresses a critical bottleneck: how do we systematically discover evidence to fill gaps in the system's knowledge? Your work produces:

  • Smart RAG Evaluation Report: Comparison of Consensus, Elicit, Scite, Semantic Scholar, and OpenAlex for evidence discovery
  • VOI-to-Query Bridge: A translation layer that converts research questions into optimized search queries for multiple backends
  • Public Contribution System: Interface for researchers to suggest topics, upload PDFs, and contribute findings
  • AI Subscription Architecture: Blueprint for running AI-powered discovery on sustainable subscriptions (not per-call costs)
  • Governance & Self-Repair: Automated testing and monitoring so the system catches failures and notifies overseers

Your 6-Week Deliverable Chain

Each week's deliverable builds on the previous one. By Week 8, you'll have a working, tested component that plugs into the larger ATLAS system.

3b. VOI-to-Query Bridge (Core Innovation — Pair 1 Primary, All Support)

This is the heart of the AF subsystem. The VOI (Voice of Information) contract produces research QUESTIONS, not Boolean queries. Your job is to design and implement an intelligent translation layer.

The Problem

A VOI question might be: "Does moderate alcohol consumption increase or decrease cardiovascular risk?"
This is not a Boolean query. Each RAG platform expects different input:

  • Consensus: Natural language question, prefers plain English
  • Elicit: Structured query with variables (e.g., "effect of alcohol on cardiovascular outcomes")
  • Scite: Keywords + citation context filters
  • Semantic Scholar: Title/abstract keywords + optional filters
  • OpenAlex: Title field, author, publication year, DOI if available

The Solution: Intelligent Query Adaptation

Your bridge takes ONE research question and expands it into multiple semantically-equivalent but platform-optimized queries:

Input: "Does moderate alcohol consumption affect cardiovascular risk?" Consensus Query: "Does moderate alcohol consumption increase or decrease cardiovascular risk?" Elicit Query: "Effect of alcohol consumption on cardiovascular risk" Variables: [alcohol_level: moderate, outcome: cardiovascular_risk] Scite Query: Keywords: ["alcohol" OR "ethanol", "cardiovascular", "risk"] Citation context: "dispute" (find papers that challenge consensus) Semantic Scholar Query: q: "alcohol cardiovascular risk" filters: [venue_type: "journal", year_min: 2000] OpenAlex Query: title: ["alcohol", "cardiovascular"] publication_date: "2000-01-01:2024-12-31"

Your code will:

  • Parse the input question (extract key concepts, relationships)
  • Generate platform-specific queries (using Claude API to optimize)
  • Execute queries in parallel across all backends
  • Deduplicate results (same paper found in multiple sources)
  • Rank by relevance (combine confidence scores from each platform)
  • Return unified result set to the user

Success Criteria for the Bridge

  • Given a research question, produces ≥3 usable queries per platform (no syntax errors)
  • Retrieves relevant papers from all five platforms for the same question
  • Deduplicates results (identifies same paper across sources)
  • Gracefully handles API failures (if one platform is down, others still work)
  • Ranks results sensibly (not just arbitrary order)
  • Response time <10 seconds for typical queries

3c. AI Subscription Model for Article Finding (Pair 2 Primary)

All AI calls must run on subscriptions (Claude, OpenAI, etc.), NOT per-call APIs. This avoids runaway costs.

The Challenge

If you call Consensus API 100 times per day, OpenAlex 100 times, Semantic Scholar 100 times, your bill explodes. Solution: use subscriptions for your AI logic, then call platforms economically.

Proposed Architecture

Option A: Claude-in-Terminal (Recommended)

  • Run Claude as a backend service in a dedicated terminal/Docker container
  • Your Flask app sends queries to Claude via the Anthropic SDK (subscription-based, not per-call)
  • Claude handles: question parsing, query generation, result ranking, ambiguity resolution
  • RAG platforms are called only once (or a few times) per question, not per intermediate step

Option B: Browser Agent for Consensus & Scholar AI

  • Consensus and Scholar AI are available as ChatGPT plugins (in browser)
  • Can you automate browser interactions to run queries through ChatGPT (on subscription)?
  • Pair 2 should prototype: browser automation agent that talks to ChatGPT + plugins
  • If viable, this gives free (subscription-funded) access to Consensus + OpenAI's RAG

Pair 2 Responsibilities:

  • Investigate and prototype both options (Terminal + Browser Agent)
  • Cost analysis: compare per-call API costs vs. subscription-based architecture
  • Document chosen architecture in D5 (Week 8) implementation guide
  • Handle authentication, error recovery, rate limits

3d. Public-Facing Contribution Pages (Pair 3 Primary)

Build three web pages for researchers to contribute findings to ATLAS. These should have a distinct "Contribute" branding, slightly different color/style from admin pages.

Page 1: Research Topic Suggestion

  • Purpose: Researchers or the public suggest a topic; system interprets it and finds it in the VOI gap list or adds it
  • UX: Simple form: "What research question would you like ATLAS to explore?" + optional contact email
  • Backend: Parse question using Claude API, check against existing VOI questions, rank by similarity, show top matches. If not found, queue for VOI team review
  • Visual Distinction: Use teal accent (#1a7a5e) but softer than admin; "Suggest a Topic" heading instead of admin-style terminology

Page 2: PDF Upload (ANFA Contributors)

  • Purpose: ANFA members and other researchers can upload PDFs they think should be in ATLAS
  • UX: Drag-and-drop upload, optional metadata (title, authors, source), auto-queue for processing
  • Backend: Accept PDFs, extract metadata (using pdfplumber or similar), auto-run title/abstract through Semantic Scholar to find DOI, queue for ingest pipeline
  • Feedback: User gets confirmation: "PDF received! We'll process it and notify you when it's available."

Page 3: Registered Contributor Dashboard

  • Purpose: Researchers register → get assigned a VOI question → upload results to their assigned page
  • Workflow:
    1. Researcher fills form: name, institution, expertise areas
    2. System matches them to an open VOI question (e.g., "Effects of blue light on circadian rhythm")
    3. They get a personalized page showing the VOI question + upload interface for their findings
    4. After upload, findings are vetted by ATLAS team before integration
  • Visual Branding: Friendly, encouraging tone; "Your Contribution" instead of "Data Submission"

Pair 3 Responsibilities

  • D2 (Weeks 4–5): UX/UI design for contribution pages. Prototypes, user feedback
  • D3 (Week 6): Usability testing with target users (researchers, ANFA members)
  • D4 (Week 7): Technical architecture proposal (frontend framework, database schema, backend API)
  • D5 (Week 8): Working implementation + deployment guide

3e. APA/Abstract/PDF Collection (Supporting Role — Pair 2 Contributes)

For each paper found by the AF subsystem, retrieve and store: APA citation, abstract, and PDF (if open access).

Data Pipeline

  1. DOI → Metadata Lookup: Use OpenAlex API to resolve DOI → full citation, abstract, publication year
  2. Citation Graph: Semantic Scholar API for citing papers, cited-by relationships
  3. PDF Discovery: Check for open-access PDFs (unpaywall API, arxiv, author websites)
  4. Storage: Cache metadata in database (PostgreSQL); store PDFs in object storage (S3 or local filesystem)

Responsibilities

  • Pair 2 researches and prototypes OpenAlex + Semantic Scholar integration
  • Handles DOI → PDF lookup pipeline
  • Integrates into D5 code (Pair 1 calls, Pair 2 supplies metadata)

3f. Success Conditions, Tests, Self-Repair (All Pairs Implement)

Critical: Every component must have automated testing, self-repair, and notification logic. This is NOT optional.

Required Governance Architecture (For Each Component)

1. Success Condition Definition

  • Write in JSON: what does "working" mean for this component?
  • Example: "VOI-to-Query bridge: processes query in <10 sec, returns results from ≥3 platforms, zero API errors"

2. Automated Test Suite

  • pytest or Jest tests for each component
  • Example tests: "test_query_generation_produces_valid_syntax", "test_bridge_handles_api_timeout", "test_deduplication_finds_duplicate_papers"
  • Target: ≥80% code coverage

3. Self-Repair Routine

  • If a test fails or success condition is violated, the system attempts recovery
  • Example: "If Consensus API returns 500 error, retry with exponential backoff; if still failing after 3 retries, use Semantic Scholar instead"
  • Log all recovery attempts

4. Notification to Overseer

  • If self-repair fails, send alert email to TA/faculty
  • Include: component name, error description, timestamp, suggested fix
  • Implement: Python script with email + logging

This governance layer ensures the AF subsystem is production-ready and maintainable by future teams.

Section 5: Git Branch Setup & Workflow

Each pair works on a feature branch. This prevents conflicts and keeps the main codebase clean.

Branch Naming Convention

feature/af-{component}-{pair_initials}

Examples:
  feature/af-voi-query-bridge-jm         (Pair 1: Jenna & Mike)
  feature/af-ai-subscription-sk          (Pair 2: Sophie & Kevin)
  feature/af-contrib-pages-ar            (Pair 3: Aisha & Raj)

Workflow Steps

1. Create Your Feature Branch (Week 3, Day 1)

git checkout -b feature/af-{component}-{initials}

Example:

git checkout -b feature/af-voi-query-bridge-jm

2. Develop & Commit Frequently

Make small, meaningful commits as you work:

git add .
git commit -m "Add query translation for Consensus platform"

Commit Message Style: "Add [feature]" or "Fix [bug]" or "Refactor [component]". Keep messages ≤80 characters.

3. Push Daily

git push origin feature/af-{component}-{initials}

Keeps work backed up and visible to teammates.

4. Pull Request Checklist (When Ready for Review)

  • All tests pass locally: pytest tests/ or npm test
  • Code follows style guide (PEP 8 for Python, ESLint for JavaScript)
  • README updated with new functionality
  • No merge conflicts with main branch
  • Assign reviewers (both other pairs + TA)

5. Code Review & Merge

At least one other pair must review before merging. Reviewers check:

  • Code quality (clear, documented, no obvious bugs)
  • API compatibility (does this work with other components?)
  • Test coverage (new code has tests)

After approval, merge to main:

git checkout main && git pull
git merge --no-ff feature/af-{component}-{initials}

6. Tag Releases (End of Each Week)

After Week 3 deliverable (D1) is ready, tag it:

git tag -a v0.1 -m "D1: Pipeline Orientation Report"
git push origin --tags

Continue: v0.2 (D2), v0.3 (D3), etc.

Section 6: Frequently Asked Questions

How do we coordinate between pairs if we're working in parallel?
Weekly sync meetings (30 min, Tuesdays at 2pm recommended). Each pair shares: (1) progress on D#, (2) blockers, (3) what you need from other pairs. Use a shared Google Doc or Slack channel. Pair 1 leads the meeting (since bridge is the core component). Also: define API contracts early (what data flows between components?). Example: "Pair 1's bridge returns {question, results[], timestamps}. Pair 3's contribution page consumes that JSON. Here's the schema." Get alignment on schema before coding.
What if one pair finishes early? Can they help other pairs?
Yes, absolutely. If Pair 1 completes the VOI-to-Query bridge by mid-Week 7, they can help Pair 2 with integration testing or Pair 3 with backend API design. Tag these contributions clearly in commits (e.g., "co-authored-by: jenna@ucsd.edu, mike@ucsd.edu"). Document cross-pair contributions in the final report under "Team Contributions."
What's the minimum viable scope for D5 (implementation)?
Working, tested code for your assigned component. Not production-grade, but functional. Example: Pair 1's bridge successfully translates 10 test questions into valid queries for all 5 platforms, deduplicates results, and ranks them. No UI needed; command-line interface is fine. Pair 3's pages need a basic HTML/CSS frontend + Flask backend. Pair 2's AI subscription setup needs to successfully route queries to Claude and receive results. All must have tests and a README.
Can we use existing libraries or must we code from scratch?
Use libraries! LangChain, pydantic, requests, flask, react, etc. This is real engineering, not a coding bootcamp. You're evaluated on system design, API integration, and testing—not on reinventing the wheel. Just cite libraries in your report and documentation.
How much should we rely on AI assistance (Claude/ChatGPT)?
Heavily! Use AI for: code scaffolding, debugging, testing strategy, documentation writing, API integration examples. Your job is to understand what the code does, test it, and integrate it into the larger system—not to write every line from scratch. Just don't use AI to avoid thinking. For major design decisions, discuss with your team and TA before implementing.
What if a Smart RAG platform changes its API or pricing mid-project?
Document the change and adapt. This is realistic engineering. Your bridge is designed to handle multiple backends precisely so a single failure doesn't break everything. If Consensus changes pricing, you can still fall back to Semantic Scholar. Update your D6 report to discuss this risk and how your architecture mitigated it.
Who owns the final code? Can we use it after the class?
It belongs to UC San Diego / Professor Kirsh's research group. Your team may use it for job interviews (as a portfolio project). If you want to publish the work, discuss with the faculty supervisor. This is normal academic/research practice.

Final Notes

The AF Track is ambitious—you're building research infrastructure used by real scientists. Your work validates Smart RAG platforms, designs a novel VOI-to-Query bridge, and creates pathways for researchers to contribute evidence. By Week 8, you'll have a working system that could be deployed. By Week 10, you'll have a publication-quality report documenting the entire effort.

This is research engineering, not coursework. Mistakes are learning opportunities. Unexpected challenges (APIs go down, scope changes) are realistic. Document everything, test rigorously, and integrate constantly.

Questions? Ask the TA (office hours) or post in the Slack channel. We'll help you succeed.
