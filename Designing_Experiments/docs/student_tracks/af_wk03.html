ATLAS is a question-answering system for neuroarchitecture — it answers questions like "Does biophilic design reduce cortisol levels?" by synthesizing evidence from dozens of scientific papers. But ATLAS can only answer questions about evidence it has already extracted. When a researcher asks about a topic that the corpus doesn't cover, ATLAS doesn't say "I don't know" — it gives a low-confidence answer based on whatever loosely related beliefs it holds. That answer can mislead.

The Article Finder track exists because the solution to that problem is not better AI reasoning — it is more evidence. Your team's job is to build the pipeline that finds new articles, evaluates their relevance, and routes them into the ATLAS extraction workflow. Before you can build it, you need to understand exactly what the pipeline already does, where its knowledge gaps are, and what it expects from incoming articles.

D1 is that orientation. By the end of this week, you will have read the six key source files that define the AF pipeline, traced one query end-to-end through the system, and written a report that your team will use as a shared reference for the next seven weeks.

The AF Module: Function, Structure, and Key Contracts

The Article Finder module sits at the intake end of the ATLAS pipeline. Its job is to answer one question: given a belief that ATLAS holds with insufficient evidence, what papers should we read to improve that evidence? The module translates that question into database queries, executes those queries against academic search APIs, scores the results for relevance, and returns a ranked list of candidate papers for human or automated review.

The module is not a web scraper, not a PDF downloader, and not a citation manager. It is specifically a gap-driven retrieval system: it starts from a belief the system already holds (or wants to hold) and works backward to find the evidence that would change its credence. This architecture distinguishes it from a general-purpose literature search tool, and it is the first thing students must understand before evaluating any of the search platforms in D2 and D3.

Key Data Contracts

Every file in the pipeline communicates through Python dataclasses. These dataclasses are the contracts: changing one field requires changing every consumer that reads it. The three contracts students must understand for D1 are EpistemicGap, ArticleReference, and AutomatedTargetRun.

Dataclass Source file Key fields What it represents
EpistemicGap voi_search.py belief_id, gap_type, voi_score, belief_content, credence_value A belief the system holds whose evidence is insufficient. The voi_score is the expected gain in epistemic state from resolving this gap — not classical value of information but Expected Epistemic Gain. The gap enters the search queue as a retrieval target.
ArticleReference src/queue/models.py doi, title, authors, year, relevance_score, addresses_gap, gap_closure_estimate A candidate article returned by a search. The relevance_score is computed from abstract presence (0.55 weight) and citation count (0.45 weight). The gap_closure_estimate is how much this article would reduce uncertainty in the target gap — currently set equal to relevance_score, which is a known simplification.
AutomatedTargetRun automated_searcher.py target_id, status, n_candidates, top_titles, message The result of one automated search run on one gap target. Status values are FOUND_RELEVANT, NOT_FOUND, ALREADY_COVERED, and NEEDS_REVIEW. A NOT_FOUND result with is_research_opportunity=True means this gap may be a genuine research frontier — no indexed papers exist yet.
SearchResult src/queue/models.py result_type, queries_used, databases_searched, confidence, articles_found, is_research_opportunity The report the searcher files when it completes a target run. The confidence value is 0.6 for found results and 0.35 for not-found — hardcoded in the current implementation. The is_research_opportunity flag is one of the system's most important outputs: it means the gap is real and unstudied.

The Four Gap Types

The GapType enum in voi_search.py defines four categories of epistemic gap. Understanding these categories is essential because each gap type requires a different search strategy — which is exactly what the StrategySelector class encodes, and which your D4 proposal will need to evaluate:

GapType taxonomy (src/services/voi_search.py, lines 64–84)

TypeMeaningTypical query strategyExample in neuroarchitecture
VALIDATION A claim exists but its credence is low — it needs replication evidence Systematic keyword search across multiple databases; look for meta-analyses "High ceilings improve creative thinking" — has one study, needs five
MECHANISM An effect is known but its causal mechanism is unexplored Process-focused queries: "neural mechanism," "mediating pathway" "Biophilic design reduces cortisol" — known, but why? HPA axis? Attention restoration?
DIRECTION Contradictory evidence exists and the direction of effect is unclear Comparative search: find studies on both sides; look for moderators "Open-plan offices improve collaboration" — conflicts with "open plans harm focus"
BOUNDARY An effect exists in some populations/contexts but its scope is unknown Moderator search: "age," "culture," "individual differences," "effect size heterogeneity" "Fractal patterns preferred at D≈1.3" — does this hold cross-culturally?

Where the Current Implementation Falls Short

Understanding the gaps in the current implementation is part of what D1 asks you to report on. Three limitations stand out and directly motivate D2–D5:

  • Semantic Scholar only. The AutomatedQueueSearcher currently calls only the Semantic Scholar API (see automated_searcher.py, line 29: from app.services.semantic_scholar import search as semantic_scholar_search). The five platforms you will evaluate in D2 are not yet integrated.
  • Relevance scoring ignores content. The _row_to_reference() method computes relevance from abstract presence and citation count — it does not read the abstract. A highly-cited paper that is irrelevant to the gap gets the same score as a highly-cited paper that directly addresses it.
  • No persistence for warrant assignment. The gap_closure_estimate field exists in ArticleReference but is never used to update the belief's credence in the Web of Belief. The feedback loop between finding a paper and improving the system's confidence is currently broken.

Frequently Asked Questions

The code imports from modules that don't seem to exist. Is the setup broken?

Several imports in the AF source files use optional try/except blocks — if a module isn't installed, the system logs a warning and continues with reduced functionality. The Setup Guide lists the mandatory vs. optional dependencies. Focus on the six core files; ignore optional integrations (sentence-transformers, discovery funnel) for D1.

The database file (web_persistence_v2.db) is missing or empty. How do we get the top-5 gaps?

If the database isn't populated, you have two options. First, run the seeding scripts described in the Setup Guide — they load a sample corpus. Second, read the GapDetector.detect_gaps() method (voi_search.py, ~line 935) and trace through what it would return for hypothetical beliefs from the neuroarchitecture literature. For D1, a hand-traced example is acceptable; for D3, you need real data.

What is "Pearl's ladder of causation" and why does the QueryParser care about it?

Pearl (2009) distinguishes three levels of causal reasoning: (1) association ("what is correlated with X?"), (2) intervention ("what would happen if we changed X?"), and (3) counterfactual ("what would have happened if we had changed X?"). The QueryParser classifies queries to these levels because different question types require different kinds of evidence — an associational question can be answered from observational data, but an interventional question requires an experiment. The classification affects which beliefs the QueryEngine searches and how confidently it answers.

The relevance scoring seems too simple — abstract presence and citation count only. Can we improve it for D5?

Yes, and it is one of the most important improvements D5 can make. The current scoring is in _row_to_reference() in automated_searcher.py. Your D4 proposal should specify a better relevance model (e.g., semantic similarity between abstract and gap description using sentence-transformers). The existing code has a comment noting this as a known simplification.

The StrategySelector uses epsilon-greedy. What does that mean for us?

Epsilon-greedy means the system usually picks the best-known query strategy for a given gap type, but with probability epsilon it tries a random strategy instead. The epsilon decays over time (see StrategySelector._update_epsilon(), voi_search.py ~line 1239). This is a reinforcement-learning exploration/exploitation trade-off: without occasional random exploration, the system might never discover that a strategy it hasn't tried yet works better. You will evaluate whether this approach is appropriate for a student-operated system in D4.
